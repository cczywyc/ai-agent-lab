## 二、关键认知变化

### 认知 1：工具描述是影响 Agent 质量的最大杠杆

周一读文档时看到 Anthropic 说"description 对性能的影响远超其他因素"，当时理解浮于表面。周三亲手写代码后才真正体会到——同一个工具，description 写得好和写得差，模型的选择准确率和参数填写质量差别巨大。

**实例**：`web_search` 的 description 里加上 "Does NOT fetch full content — use fetch_webpage for that" 后，模型明显减少了对搜索工具的误用，需要深入内容时会主动选择 fetch_webpage。

### 认知 2：模型不执行工具，只决定调用什么

这个设计原则是整个安全控制体系的基石。周三写第一个 tool calling 示例时，我清楚地看到了这个分界线：模型返回 `tool_calls` 后，**所有的执行权在我的代码手中**。这意味着参数校验、权限检查、危险操作确认都可以在执行前拦截。

### 认知 3：最危险的失败是"看起来像成功"的失败

周二读到这个概念时印象深刻，但周四实测时更直观地感受到了——Case 1（诺贝尔奖）模型直接回答了一个"看起来正确"的答案，完全跳过了搜索工具。如果模型的训练数据里有错误信息，这个回答就会是自信的错误，而用户完全无法分辨。

### 认知 4：Agent 的控制循环设计比想象中复杂

周三写的 Agent Loop 看起来就是一个简单的 while 循环，但周四实测暴露了大量边界问题：退出条件太简单、最大轮次不够灵活、错误恢复依赖模型自行决策、上下文无限增长。一个"能跑通"的 Loop 和一个"生产可用"的 Loop 之间差距巨大。

### 认知 5：工具设计和 system prompt 需要协同优化

单独优化工具或单独优化 prompt 都不够。实测中发现，即使工具的 description 写得很清楚，如果 system prompt 没有明确约束"事实性问题必须先搜索"，模型仍然会跳过工具直接回答。反过来，如果 system prompt 约束了但工具描述模糊，模型也会选错工具。两者必须一起设计、一起迭代。

## 四、Anthropic vs OpenAI/千问 格式对比

### 两者的本质是相同的

Tool calling 的核心流程在任何 LLM 上都一样：**定义工具 → 模型决策 → 执行工具 → 返回结果 → 模型综合**。区别只在于 API 的字段名和消息结构。

### 关键差异点

| 维度 | Anthropic (Claude) | OpenAI / 千问 |
|------|-------------------|---------------|
| 工具参数字段 | `input_schema` | `parameters`（嵌套在 function 对象内） |
| 工具定义结构 | 扁平：直接 name/description/input_schema | 包装：`{"type": "function", "function": {...}}` |
| 模型要调用工具的信号 | `stop_reason == "tool_use"` | `finish_reason == "tool_calls"` |
| 工具调用在哪里 | response.content 中的 tool_use block | message.tool_calls 列表 |
| 工具参数格式 | `input` 是 dict | `arguments` 是 JSON 字符串，需要 `json.loads()` |
| 工具结果的角色 | `role: "user"` + `type: "tool_result"` | `role: "tool"`（专用角色） |
| 示例机制 | 原生 `input_examples` 字段 | 无原生字段，需在 description 中手写 |
| Strict 模式 | 无 | `strict: true` 保证参数严格匹配 schema |

### 通用的、与 LLM 无关的概念

这些概念换任何模型都适用，是 Agent 工程师的核心素养：
- 工具三分类：Data / Action / Orchestration
- Description 六要素模型
- 返回值的"高信号低噪音"原则
- 错误返回的标准格式：error_type + message + recoverable + suggestion
- Agent Loop 的 while 循环模式 + 最大轮次保护
- 工具注册表模式（TOOL_REGISTRY）

## 五、Agent Loop 核心模式

Agent Loop 是所有 Agent 系统的心脏。通过本周的实践，我理解的 Agent Loop 本质是：

```
初始化 messages = [system_prompt, user_message]

for turn in range(max_turns):
    response = LLM.call(messages, tools)

    if response.wants_to_answer:        # finish_reason == "stop"
        return response.text            # 出口 1：正常结束

    if response.wants_tools:            # finish_reason == "tool_calls"
        messages.append(response)       # 保留模型的决策记录

        for tool_call in response.tool_calls:
            result = execute(tool_call)  # 开发者执行
            messages.append(result)      # 追加工具结果

        continue                        # 回到循环顶部

return "达到最大轮次"                    # 出口 2：安全终止
```

### 当前实现的不足（本周发现）

1. **退出条件太简单**：只看 finish_reason，没有判断"模型是否应该先搜索再回答"
2. **无 fallback 机制**：工具连续失败时没有主动引导模型降级
3. **上下文无管理**：messages 无限增长，长对话可能溢出
4. **错误元数据未利用**：工具返回了 recoverable/suggestion，Agent Loop 没有读取和利用

这些都是第三周需要改进的方向。

## 六、评测结果分析

### 总体数据

| 指标 | 数值 |
|------|------|
| 总测试用例 | 7 |
| 通过（工具选择正确 + 有回答） | 4/7（57%） |
| 工具使用准确率 | 4/7（case 1/5/7 应该用工具但没用） |
| 有回答率 | 6/7（case 3 耗尽轮次无输出） |
| 平均轮次（有工具调用） | 3.0 |
| 错误率 | 1/7（case 3 遇到 403） |

### 逐案分析

**Case 1 — 诺贝尔奖（❌ 未调用工具）**
模型直接从训练数据回答，跳过了搜索。回答看起来正确，但无法验证时效性。暴露了 system prompt 对"事实性问题必须搜索"的约束不足。

**Case 2 — MCP 协议（✅ 正常）**
搜索 → 直接从 snippet 总结，2 轮完成。模型正确判断了这是需要搜索的技术概念。

**Case 3 — LangGraph vs CrewAI（❌ 耗尽轮次）**
搜索成功但 fetch_webpage 连续遇到 403（Medium、DataCamp 等反爬站点），5 轮全部用于重试抓取，最终无输出。核心问题：无 URL 黑名单 + 无 fallback 到 snippet 的机制。

**Case 4 — Qwen3 features（✅ 正常）**
英文搜索 → 英文回答，2 轮完成。模型正确使用了英文关键词。

**Case 5 — RAG 解释（❌ 未调用工具）**
模型认为 RAG 是已知概念，直接回答。回答质量不错但没有引用来源。反映了"常识 vs 需搜索"的边界模糊。

**Case 6 — 写诗（✅ 正常）**
正确判断不需要工具，直接创作。

**Case 7 — 不存在的东西（❌ 未调用工具）**
模型直接说不知道，没有先尝试搜索。理想行为应该是先搜一下确认确实不存在，再告知用户。

### 核心发现

1. **模型倾向于"能回答就回答"**，即使 system prompt 要求先搜索。这是 LLM 的固有行为模式，需要更强的 prompt 约束或 Agent Loop 层面的强制机制。
2. **工具失败的连锁效应很严重**。一次 403 不可怕，但连续 403 + 无降级机制 = 整个任务失败。
3. **工具使用率（43%）远低于预期**。说明当前的 system prompt 和工具描述在"何时该用工具"的引导上不够有效。

## 七、对后续 Roadmap 的思考

### 第三周（Prompt + Structured Output + 控制循环）

我的搜索 Agent 最需要改进的就是这三个方面：
- **System prompt 优化**：更精确地区分"常识问答"和"需搜索验证的问答"
- **Structured output**：让模型输出固定 JSON 格式（如 `{"action": "search", "query": "..."}` 或 `{"action": "answer", "content": "..."}`），而不是自由决策
- **控制循环改进**：加入 fallback 策略、重试上限、上下文裁剪

### 第四周（RAG）

搜索 Agent 的 `web_search` + `fetch_webpage` 模式和 RAG 很相似——都是"检索 → 获取内容 → 基于内容回答"。区别在于 RAG 用向量检索代替关键词搜索，用预处理的文档代替实时网页。搜索 Agent 的 tool calling 模式和 Agent Loop 可以直接复用。

### 第五周（记忆系统）

当前 Agent 没有任何记忆——每次对话都是独立的。如果用户先问了"MCP 是什么"，再问"它和 REST 有什么区别"，Agent 不知道"它"指的是 MCP。需要加入对话历史管理和上下文压缩。

### 第六周（LangGraph）

目前的 Agent Loop 是手写的 while 循环，状态管理完全靠 messages 列表。LangGraph 会把这个循环变成可视化的状态图，每个节点（搜索/抓取/回答）可以独立配置和观测。我现在手写的 Loop 正好能帮我理解 LangGraph "为什么要这么设计"。

## 八、本周最大收获

如果只能说一句话，我这周最大的收获是：

**Agent 不是一个高深的架构概念，它的核心就是 "LLM + 工具 + 循环"。真正的难度不在于让它跑起来，而在于让它稳定地、可控地、可观测地运行。** 工具设计的质量、prompt 的精确度、循环的控制策略、错误的恢复机制——这些"工程细节"才是区分 demo 和生产系统的关键。
