## 二、本周做的事，一句话

把第八周 v6.0 里 researcher 内层那套**手写工具注册表 + `UnknownTool` 兜底**，升格成"**经 MCP client 发现并调用的工具契约**"：真实 FastMCP server **包一层**暴露 `local_search`（包 `retrieve_documents`）/ `http_fetch`（包 `fetch_webpage`），client 经 stdio `tools/list` 发现、`tools/call` 调用，researcher 内层 `tools` 这一环从"调 `execute_tool`"换成"调 MCP client"。**核心认识：本周是给工具层插一道进程边界，不重写引擎、不动 supervisor 拓扑**——server 直接 `import` v6.0 函数（没复制），引擎/闸门/supervisor 全复用，真正"新"的只有三块：MCP server、client 站、工具表从硬编码换成发现来的。

**"包一层不重写"有多硬，看证据**：① server 直接 import v6.0 的 `retrieve_documents`/`fetch_webpage`（不是 fork）；② 周四 `verify_thursday` 的 C 部分——researcher loop 换接 MCP client 后，`turn_count=4/empty_retries=0/synthesis_forced=True` 与周三桩**逐位一致**（决策 G 无回归）；③ 周五真实跑 synthesis-reserve 闸门在真实模型 over-retrieve 时照样触发。新增的少、复用的多——和第八周 token 账同一个"blast radius 最小"的量化逻辑。

## 三、给周一两根刺打分（本周独有的反思）

周一《MCP 概念》带着两根刺去读（第八周《多 Agent 概念》§七留的），逐条兑现，**且关键一条由真实 SDK 给了反转实证**：

| 周一带进来的刺 | 本周结果 |
|---|---|
| ① MCP 标准化接口层，那"调用漂移 / 幻觉名 / 结果不接地要 retry / 语境锚怎么不丢"这些**控制流**，它管不管？ | ✅ **不管，且实测比预想更不管**：真实 FastMCP **连"失败长什么样"都没标准化**——未知工具名 / 参数违 schema / 业务失败**全塌成一层 `isError:true`**，我桩里假设的"协议 error（异常）vs isError（字段）两层"被**证伪**（周四）。于是"两层错误分流"这件控制流得 **host 自己在派发前后重建**。周一"框架给原语、控制流自己兜"那条主轴被顶到更硬的实证：**框架的失败词汇表比想象中更弱，控制流更彻底归我**。 |
| ② typed 契约能把"幻觉名运行时兜住"提前到发现期拒吗？ | ◑ **部分，且答案比预想更微妙**：错参数理论上能 schema 层拒——但真实 FastMCP 在 server 侧 pydantic 校验后**回成 isError、不给 client 一个协议 error**，所以 host 得拿发现来的 `inputSchema` 在 **client 侧自己再校一遍**（决策 E 从"可选优化"升成"承重墙"）。"幻觉名"挪不动到 schema 层（模型在 MCP 上游），只能 host 派发前守卫拦——**周五真实跑活体坐实**：活模型管线里幻觉名 `edit` → `pre_dispatch_reject`、根本不触达 server。 |

两根刺都有了答案，而且**①是被真实 SDK 反转着回答的**——这正是周一"别一上来就觉得接了 MCP 控制流就干净了"那份警惕的兑现：不光没干净，连"失败分几类"这种看似框架该给的，都得自己兜。

## 四、关键决策

### 七个设计决策（草稿 v0.1 锁定，A–G）

| # | 决策 | 非显然之处 | 实证 |
|---|---|---|---|
| A | 接入形态 = 包一层（不重写）；Python SDK（FastMCP）兜协议 | 把"学 MCP"和"重构工具"解耦、缺陷可归因 | E6 桩 trace 全等 + 周四 server 直接 import v6.0 函数 |
| B | 工具花名册：`local_search`+`http_fetch`，各 owns 明确 | description 写"何时调/不干什么"当软锚 | E1 + 周四 schema 由类型注解自动生成 |
| B′ | RAG = tool（model-controlled），不是 resource | resource 是固定只读数据的甜区，本周不做 | （概念决策）+ 真实跑 T2 模型主动按需调 |
| C | 工具 I/O 契约四字段 + 空结果 ≠ 失败 | 空 chunks 是业务结果非错误 | E3/E5 + 周四真检索 chunks=5 |
| D | 传输 = stdio（本机）；Streamable HTTP 留企业部署 | 老 HTTP+SSE 已废弃、不碰 | 周四 stdio 链路真跑通 |
| E | 守卫迁移：错参数→schema、幻觉名→派发前守卫、重试→留 loop | 唯一真前移的是错参数；幻觉名只换比对对象 | **v0.3 翻案**：真实 SDK 回 isError、host 须 client 侧自校 → 守卫升承重墙；真实跑 T3′ 活体坐实 |
| F | 两层错误映射：协议→上报 skip、isError/空→内层 retry | 两类失败走不同恢复路径（跨两周悬念结案） | **v0.3 发现型**：真实 SDK 塌成一层、判据挪到 host 派发前重建；真实跑 healthy 路径不退化 |

### 实现/真实跑期才浮现的两处（草稿没写、被实测逼出来）

| 决策/发现 | 起因 | 落地 |
|---|---|---|
| **决策 E/F 判据从"SDK 形态"挪到"host 派发前"**（v0.3） | 周四真实 FastMCP 把未知名/错参数/业务失败全回成 isError、不给协议异常 | `MCPToolsStation` 加 client 侧 `_validate_args`（jsonschema），名字守卫 + schema 校验在 `call_tool` 前重建二分；`verify_thursday` 11/11 |

## 六、踩坑与本周发现

### 最该记的一条：桩测的"必要非充分"，这次深到了"被验对象本身"

第七周教训是"离线全绿、真实跑连摔三跤"；第八周认识到"桩验断路器、真实跑验主力路径"。**第九周又深一层**：

> 桩测里我把"MCP 两层错误（协议 error vs isError）"当成**已知事实**建模、还写了三态干净二分的对照组、全绿。**周四真实 SDK 一问就翻——它只给一层 isError。** 桩能验"给定契约模型下接线对不对"，**锁不住"契约的真实形态是不是我假设的那样"**。所以这次救命的是**桩里留的 `[周四确认]` seam**：我没把桩里的假设当结论，标了"这部分桩答不出、留真实 SDK 问"，一问就翻案、就地折回 v0.3。**离线全绿是必要非充分——这次"非充分"非到了连被验对象本身都是假设。**

### 两个发现型 + 一处 scope 分歧

1. **真实 SDK 把两层错误塌成一层 isError（v0.3，周四）**：未知名/错参数/业务失败全回成 isError、client 不抛协议异常。→ host 派发前重建二分（名字守卫 + client schema 校验），决策 E 守卫从可选优化升成承重墙。**这是本周的中心发现**——也写进了《MCP 适合接哪些企业系统》§三当压舱石。
2. **引用格式漂移使接地软闸门字面失配（v0.4，周五）**：T3 generic 题模型把 `[doc#section]` 引用串重排（整段标题塞进 doc 位），精确字符串匹配大面积失配（0.167，底层来源多为真实本地文档）。坐实接地闸门 load-bearing（0.167<0.5 在完整图会判 retry），但本周 minimal loop 没接 critic 闸门、没拦下。处置两条记 v0.4 待复跑。
3. **scope 分歧（诚实记一笔）**：评审把"完整图 async-in-LangGraph 整合（Task 1）"列为地基。**我判断成独立增量、没硬塞进周五**——理由：设计草稿 §六/决策 G 既定 scope 就是"researcher 内层 + server"；完整 supervisor 栈 × 真实模型烧 token 多、qwen3.7-plus 也是 free tier 有炸穿风险（max 就是这么死的）；内层 ESCALATE 已 real 验、它喂的 supervisor skip 机器是第八周 topic D 已坐实的件。**这是个真实分歧——我用 researcher-loop 答了全部核心问题，但完整图整合确实没做**，留独立增量。

### 方法论沉淀（这周新长出来的）

- **真实跑前先 smoke test 外部依赖，别假设上周能用这周还能用。** 周五第一件事是拿 1 个 cheap call 探 qwen3.7-max——果然还是 403。先探明、沿用 plus fallback，再规划真实跑，没在大跑里撞额度。
- **给框架的"我以为它会给"的东西，留一个 seam 去真实验，别当结论。** 桩里 `[周四确认]` 标记这次直接救场。框架的失败语义/错误形态，尤其值得留 seam——它最容易"看文档以为两层、真跑只有一层"。

## 八、跨模型 / 通用设计原则（本周新增，每条挂证据）

- **框架的失败词汇表可能比你假设的更弱——接之前别假设它替你分好类。** 真实 FastMCP 只给一层 isError（未知名/错参数/业务失败全塌一起），我桩里假设的两层被证伪。接任何框架，"失败长什么样、分几类"要留 seam 去真实验，不是看文档拍脑袋。
- **桩测能验"给定契约下接线对不对"，锁不住"契约的真实形态是不是你假设的那样"**：桩里 `[周四确认]` seam 救场——别把桩里的建模假设当结论。这是"离线验接线、真实验策略"深到"被验对象本身"的一层。
- **框架不给干净协议信道时，host 侧守卫/白名单是承重墙、不是可选优化**（决策 E 从优化升承重墙，真实跑 T3′ 活体坐实）：未知名/错参数没有协议信道、都塌进 isError，host 派发前的名字+schema 守卫是唯一干净拦截点。
- **包一层不重写、blast radius 最小**（server 直接 import v6.0 函数、verify 11/11 无回归、loop trace 逐位全等）：第八周用 token 账量化、本周用"无回归 + 直接 import"量化，同一个原则的又一次成立。
- **真实跑前先 smoke test 外部依赖**（qwen3.7-max 403、先探明再规划）：别假设上周能用的额度/服务这周还在。
- **scope 诚实要敢于和"评审/计划"分歧**（Task 1 完整图判成独立增量，写清理由）：不是不做，是按既定 scope + 额度风险排期，并诚实标出"确实没做的那块"。

## 九、本周最大收获

**本周最大收获不是"接通了 MCP"，而是接的过程里撞出一个比设计假设更硬的真相：我以为 MCP 至少把失败分成两类（协议 / 执行）报清楚，真实 SDK 只给一层 isError。** 这把周一"框架给原语、控制流自己兜"的主轴，顶到了一个更不舒服也更真的位置——**连"失败分几类"这种看似框架该给的，都得我自己在 host 侧重建**；而这套重建在真实 qwen + 真实 stdio 上 healthy 路径不退化、守卫兜得住。MCP 买到的是接口可移植，买不到失败语义、更买不到控制流。

但比"接通"更深的两课：

1. **桩测的"必要非充分"，这次非到了被验对象本身。** 桩里我把"两层错误"当已知事实建模、全绿，周四真实 SDK 证伪。救命的是桩里留的 `[周四确认]` seam——我没把假设当结论。**经验沉淀成一条习惯：凡是"我以为框架会这样给"的东西，桩里就标个 seam、留到真实层去问，别在桩的全绿里把假设固化成结论。**

2. **scope 诚实包括敢于和计划分歧、并把"没做的"写清楚。** 评审让先搭完整图（Task 1 地基），我按既定 scope + 额度风险判成独立增量、用 researcher-loop 答了核心问题——但完整图整合确实没做，端到端的 supervisor skip、模型自然失败 recover 都还挂着。**把"我做了什么、为什么这么排、什么确实没做"三件事分开写清，比"按计划全做了"的假象诚实得多**——这也是这套学习方法（设计→桩测→实现→真实跑→复盘，每关留发现型和 scope 诚实）一直在练的。
