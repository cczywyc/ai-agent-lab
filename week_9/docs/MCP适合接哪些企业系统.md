# MCP 适合接哪些企业系统

> 定位：第九周·周六复盘日产出（roadmap week 9 的《MCP 适合接哪些企业系统》总结）。不转述别人的观点——拿本周自己的 demo（`search_agent_mcp/`：真实 FastMCP server 包两个工具 + client 经 stdio 调 + 真实模型跑）当压舱石，每条判据挂本周实测证据。
> 设计前提：周一《MCP 概念》§七（M×N→M+N 价值、判据初稿）+ 设计草稿 v0.4（决策 B′ tool/resource 分界、决策 D 传输、§四/§五 守卫与两层错误的真实跑实证）。

---

## 0. 一句话判据

**接口能被描述成稳定 typed 工具契约的系统，适合接 MCP——买到的是"接口可移植"（M×N→M+N）。但 MCP 只标准化接口层，不标准化失败语义、更不碰控制流。所以"价值/难点出在控制流"（失败怎么恢复、语境锚怎么不丢、哪些工具能调）的系统，光接 MCP 不解决——接了，控制流该兜的一样得兜，而且因为 MCP 的失败词汇表比想象中更弱，兜得更多。**

这条主轴是本周一路撞出来的，不是周一就知道的——周一以为"MCP 不管控制流"，周四真实 SDK 把判断顶得更硬（见 §三）。

---

## 一、MCP 买到的：M×N → M+N（接口可移植）

价值一句话：把"每个模型 × 每个 API 都写一遍适配器"（M×N）换成"一个工具暴露一次、所有 host 发现调用"（M+N）。

**本周 demo 实证**：`local_search` / `http_fetch` 在 server 侧用 `@mcp.tool()` 暴露**一次**，client 经 `tools/list` 发现、`tools/call` 调用，传输走 JSON-RPC over stdio。工具定义在本地/远程可移植，换底层模型（本周 qwen3.7-max→plus 的 fallback）不用重写工具适配。这就是企业最想要的：内部系统接一次，换 AI 应用/换模型不重接。

---

## 二、甜区判据：接口越稳定、越能被 typed 契约描述，越适合

**本周两个工具就是两类企业接口的原型**：

| 本周工具 | 企业对应 | 为什么是甜区 |
|---|---|---|
| `local_search`（本地文件搜索） | 内部文档库 / 知识库 / 文件系统 / 内部数据库（读） | 接口稳定、`{query, top_k}→{chunks}` 能钉死 typed 契约；读语义清楚 |
| `http_fetch`（HTTP API 调用） | 企业内部 API / SaaS（工单/CRM/文档库）的 REST 端点 | `{url, max_chars}→{content,status}` 能描述；但**失败多**（403/超时/限流）——甜区，但失败处理归 host（见 §三） |

**判据**：有稳定接口、读写语义清楚、能被描述成 `inputSchema`/`outputSchema` 的系统——内部数据库、文件系统、企业内部 API、SaaS 工单/CRM/文档库——是 MCP 的甜区。**本周实测佐证**：真实模型对"本项目版本演进"题（T2）**只选 local_search、8/8 调用 success、引用 1.0 接地**，一个描述清楚的 typed 工具，模型就能稳定调对、不漂。

---

## 三、本周最硬的一条实证：MCP 连"失败长什么样"都没完全标准化

这是本周接 MCP 接出来的、最该写进企业判据的一条——也是最容易被"接了 MCP 就有完整错误处理"这种期待坑到的地方。

**实测（周四 `verify_thursday`，真实 FastMCP）**：未知工具名、参数违 schema、业务失败（抓取 403）——**三者全部回成 `isError:true` 的同一种结果字段**，client `call_tool` **不为前两者抛 JSON-RPC 协议异常**。我设计阶段假设的"协议层 error（异常）vs 执行层 isError（字段）两层分开报"被真实 SDK **证伪**：协议层与执行层在 SDK 层**塌成一层**。

**对企业接入的含义，两条**：

1. **接 MCP 买到的是接口可移植，买不到失败语义的标准化。** "这次失败是集成坏了（该换/该报警）还是业务失败（该 retry/换源）？该 skip 这个子任务还是整个崩？"——MCP 不替你分。真实 SDK 甚至只给一层 `isError`，**两层错误分流得 host 自己在派发前后重建**（本周用"名字守卫 + client 侧 schema 校验"把"集成问题"那一支从 isError 里重新拣出来）。**企业选型时别把"接了 MCP"等同于"有了健壮的错误处理"**——错误处理的策略层始终在你的应用里。
2. **这反过来强化了一条主轴**：框架（MCP/FastMCP）给的是接口原语，失败恢复、retry 判据、降级、语境锚——控制流——全归 host。MCP 的失败词汇表比通用框架还窄（一层 isError），所以"价值在控制流"的系统接 MCP，控制流不但不会自动变干净，反而要兜得更细。

---

## 四、所以 host 侧守卫 / 工具白名单在企业规模下更重要，不是更轻

承 §三。既然未知工具名、错参数都没有干净的协议信道（都塌进 isError），**host 在派发前的守卫——拿发现来的 `tools/list` 比对名字 + 拿 `inputSchema` 在 client 侧校参数——成了唯一干净的拦截点**。

**本周实测（周五真实跑 T3′ 活体探针）**：活模型管线里，幻觉的工具名 `edit` → host 派发前 `pre_dispatch_reject`（本地拒、根本不触达 server）；错参数 → `invalid_args`（client 侧 schema 校验拦）；真业务失败 → 走内层 retry。守卫一关，幻觉名就被 server 当 isError 业务失败、**会被误当"可 retry 的业务失败"反复重试**。

**对企业的判据**：系统接的工具越多、agent 越多、模型越自由，host 侧的**工具白名单 + 参数校验守卫**越是承重墙，不是可选优化。企业部署 MCP，"哪些工具允许这个 agent 调、参数合不合法"必须在 host 侧有一道显式守卫——这道守卫是 MCP 不替你做的（接第十一周工具白名单）。

## 五、接口设计判据：tool 还是 resource？（决策 B′）

MCP 三原语里 tools / resources 的分界，直接是企业接入的一条设计判据——**区别在"谁控制"**：

| | tool（model-controlled，模型主动调） | resource（app/host-controlled，被动读进上下文） |
|---|---|---|
| 适合 | 需要**按请求动态决定查什么、查几次**的接口 | **固定要进上下文**的只读数据 |
| 企业例 | 按工单号查 CRM、按 query 检索文档库、调一个内部 API | 公司政策手册、固定的数据字典 / schema、当前用户的权限上下文 |

**本周决策**：RAG 检索做成 **tool**（researcher 按子任务动态决定查什么）。**留的甜区**：企业里"每次都该在上下文里、不依赖具体请求"的固定只读数据（政策、字典、长期事实），才是 resource 的甜区。选错（把该 tool 的做成 resource 会撑爆上下文、把该 resource 的做成 tool 会让模型反复去调）——这条分界本身是 MCP 接入的一个设计判据。

## 六、传输：本机 stdio，企业部署上 Streamable HTTP + OAuth（决策 D）

- **本周 demo 用 stdio**：本机、同机进程、零网络变量，调试最省。
- **企业部署该上 Streamable HTTP**：远程、多客户端、HTTPS、可横向扩、能接 OAuth 鉴权。老的 HTTP+SSE 独立传输已被取代、不碰。

企业接 MCP 的鉴权/多租户/审计，都落在 Streamable HTTP 这一侧——但这是部署形态，不改"接口契约"本身（同一套 JSON-RPC 工具定义在 stdio 和 HTTP 间可移植）。

## 七、判据汇总表

| | 适合接 MCP | 光接 MCP 不解决 |
|---|---|---|
| 接口形态 | 稳定、读写语义清楚、能钉死 `inputSchema`/`outputSchema` | 强依赖隐式上下文、语义模糊、接口不稳定 |
| 价值所在 | 价值在"接口可移植"（M×N→M+N、换模型不重接） | 价值/难点在**控制流**（失败恢复、语境锚、流程编排） |
| 企业例 | 内部数据库、文件系统、企业内部 API、SaaS（工单/CRM/文档库） | 需要复杂业务流程编排、跨步骤状态、人审介入的系统 |
| 本周证据 | T2：稳定 typed 工具，模型 8/8 调对、grounding 1.0 | §三 isError 塌一层、§四 守卫承重墙——失败语义/守卫仍归 host |
| 接了之后还得自己兜 | 失败分流、retry/skip 判据、工具白名单、语境锚、tool/resource 选型 | 同左，且因 MCP 失败词汇表更弱（一层 isError）兜得更多 |

---

## 八、收口

**接口标准化 ≠ 控制流标准化，甚至 ≠ 失败语义标准化。** MCP 实打实给的是 typed 接口契约的可移植性（暴露/发现/调用 + 一层 isError 失败信号）；失败该怎么恢复、语境锚怎么不丢、哪些工具能被调——这些控制流，连同"把一层 isError 重新分成集成问题 vs 业务失败"的二分，全归 host。

所以企业选型一句话：**接口稳定、能被 typed 契约描述、价值在"接一次到处用"的系统，是 MCP 的甜区；价值/难点在控制流的系统，接 MCP 解决的是接口可移植那一层，控制流该兜的一样得自己兜——而且本周实测证明，因为 MCP 的失败词汇表比想象中更弱，这部分往往要兜得更细，不是更省。**

这正是本周主轴在企业语境的落地——和周一《MCP 概念》§七判据同源，只是现在有了自己 demo 的实测当压舱石，不是转述。
