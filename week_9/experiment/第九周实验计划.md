# 第九周实验计划：MCP 接线机制隔离验证（E1–E6）

> 周三实验/桩测日。延续第七/八周 E1–En 的打法：**桩隔离验框架接线**——不接真实 LLM、不接真实 MCP SDK、不碰真实 RAG/HTTP，只忠实建模 MCP 2025-11-25 的接口契约（`tools/list` 发现、`inputSchema` 校验、两层错误模型），验**派发前守卫迁移（§四）、两层错误分流（§五）、包一层无回归（§六）**这些**我自己掌控的接线逻辑**是否成立。零 API、确定、可复现，每条配**对照组**（验"对的能跑"也验"拆掉它怎么错"）。
> 对应：《第九周设计草稿 v0.1》的"待验证清单"6 项 / 决策 A–G。结果跑完并入草稿 v0.2。
> 落地：`experiment/mcp_stub.py`（桩仪器）+ `test_e1..e6.py`（pytest，16 个断言函数）+ `run_all.py`（零依赖总跑，打印实测探针值 + 总计）。

---

## 总原则（承第七/八周，每条都不许破）

- **桩不接 LLM、不接真实 SDK**：桩 server 返回固定 payload，桩 client 建模 `discover/call` 契约，路由判别（决策 F）是纯 Python——把"框架接线行为"和"模型质量 / SDK 确切形态"分开。本周验前者，后者留周四（换真实 FastMCP）/周五（接真实模型）。
- **离线验接线、真实验策略 + 离线全绿是必要非充分**（第七/八周最硬的教训）：桩能锁"幻觉名被派发前拒、两层错误在 loop 里分得开、换 station 后闸门不变"，**锁不住**"真实 SDK 把协议 error / isError 暴露成什么形态、stdio 链路真不真通"。**故 E3/E4 凡涉及'真实 SDK 形态'的部分一律标 [周四确认]，桩只坐实我这侧的路由逻辑。**
- **每条配对照组**：不光验机制成立，还要验"拆掉它会怎么错"——没有对照组的绿是没说服力的绿（E2 关守卫撞协议 error、E6 扰动 station 被检出，都是为此）。
- **零 API 纪律的坑，务必先躲开（第八周 E1 的翻版）**：MCP 的 tools 是 model-controlled，但桩里驱动调用的"agent"**绝不能接真实 LLM**。第八周原想拿 prebuilt supervisor 当对拍基线，实测它是 LLM 调 handoff 工具、非零 API，只好改"写歪路由"探针。今天同理——E2 的"幻觉工具名"用**确定性探针**（脚本化 `call_tool('edit', ...)` 喂清单外的名字），而非真等一个模型去幻觉。整个 harness 喂确定的"工具名 + 参数"，才跑得出可复现的桩测。
- **桩里也要能造失败**：`mode` 旋钮注入三种返回——协议层 error（未知名/违 schema）、`isError` 业务失败、空结果——是验两层分流（E4/E5）的前提。

## 实验清单

### E1 — 工具发现（`tools/list`）

**验什么**：client 经 `discover()`（建模 `tools/list`）拿到 `local_search`/`http_fetch` 两工具的 **name + `inputSchema`**，且**发现来的清单 = 有效工具集的权威来源**。

- **桩设置**：`build_server()` 注册两工具，`client.discover()` 拉清单，断言 `discovered_names` 与各 `inputSchema.required` 就位。
- **判据**：两工具可发现；`required` 正确（`local_search→query`、`http_fetch→url`）。
- **对照组**：`add_tool('summarize')` 变更工具表 → `list_changed_pending` 置脏；**重发现前** client 缓存仍旧（看不到 summarize），**重发现后**才反映——坐实"权威来源住在发现期、`listChanged` 才同步"。
- **预判**：巩固型。

### E2 — 派发前守卫拒幻觉名（决策 E）

**验什么**：模型吐一个清单外的名字（`edit`）→ **派发前守卫**本地拒、**不发 `tools/call`**；且守卫比对的是**发现来的清单**、不是又写死一份。

- **桩设置**：`guard=True`，确定性地 `call_tool('edit', ...)`，断言抛 `METHOD_NOT_FOUND` 且 `server.call_count==0`（从未触达 server）。
- **判据**：本地拒；`call_count==0`（派发前拦截，不上线）。
- **对照组**：`guard=False` → `edit` 触达 server、撞**运行期协议层 error**（`call_count==1`）。验"两条路径都在、默认走派发前"。
- **预判**：巩固型。**要确认**：守卫读的是 `client._discovered`（发现来的），非硬编码副本。

### E3 — `inputSchema` 拒错参数（决策 C/E）  [发现型候选]

**验什么**：给 `local_search` 传**缺 `query`/类型错**的参数 → 被 schema 层拒（`INVALID_PARAMS`）；合规参数正常通过。

- **桩设置**：`call_tool('local_search', {'top_k':5})`（缺 query）、`{'query':123}`（类型错），断言均 `INVALID_PARAMS`；`{'query':'...'}` 合规通过、拿到非空 chunks。
- **判据**：两种错参数被拒；合规通过。
- **对照组**：合规参数正常返回（chunks=1）。
- **发现型 seam**：草稿 §四写的是"schema 校验在 **client 或 server 侧**"——**今天拿桩问"这问题桩能不能结案"**。桩把校验默认建模在 server 侧（`client_validates=False`），并提供 client 侧 seam 证两侧等价。**[周四确认]**：真实 SDK 究竟在哪侧 fire、是抛异常还是回协议 error——这部分桩答不了，标记留周四。
- **预判**：发现型候选（不是机制不成立，而是"哪一侧/什么形态"是真实 SDK 问题、桩只能证等价）。

### E4 — 两层错误路由分叉（决策 F 核心；跨两周悬念结案）  [发现型候选·核心]

**验什么**：桩 server 分别注入**协议层 error** 和 **`isError` 业务失败** → 验证前者路由到"上报 supervisor 走 skip"（`ESCALATE`）、后者进"内层 `empty_retries`"（`INNER_RETRY`），两层**走不同路径**。

- **桩设置**：协议 error（关守卫让未知名从 server 冒出）→ `route_tool_outcome` 给 `ESCALATE/protocol_error`；`local_mode='error'` 的 `isError` → `INNER_RETRY/business_error`；断言两者 `path` 不等。
- **判据**：协议 error→ESCALATE；isError→INNER_RETRY；两路径不同。
- **对照组**：**空结果**（`isError=False`、chunks 空）走 `INNER_RETRY` 而非 `ESCALATE`——空结果绝不能被误当协议 error 上报。
- **发现型核心**：这是第三周 Loop Q3、第八周 Q3 那个"纠正/降级在哪层生效"在 typed 失败信道下**第一次有素材分开**。桩坐实"**给定契约模型（协议 error=异常 / isError=结果字段）下，harness 能在派发点干净二分**"。**[周四确认]**：真实 SDK 把两层暴露成什么形态（异常 vs 结果字段），决定 outcome 怎么被接住——若形态与设计假设不符，就是要折回 v0.2 的发现。
- **预判**：发现型候选·核心。

### E5 — 空结果 ≠ 失败（决策 C，E4 二分的地基）

**验什么**：`local_search` 桩回空 chunks（`isError:false`）→ **不被误判成 error**、由内层重试接管；空 / 业务失败 / 成功三态 reason 两两可分。

- **桩设置**：`local_mode='empty'` → `is_error=False`、`content==[]`、路由成 `empty_result/INNER_RETRY`。
- **判据**：空结果 `is_error` 仍 False、走 INNER_RETRY。
- **对照组**：遍历 `ok/error/empty` 三 mode，断言三个 reason `{ok, business_error, empty_result}` 两两可分——loop 才能各走各的恢复路径。
- **预判**：巩固型（但它是 E4 二分的地基，要一起看）。

### E6 — 包一层无回归（决策 A/G）

**验什么**：researcher 的 `tools` 站从"v6.0 硬编码 dict 派发"换成"MCP client 派发"后，内层 `turn_count`/`empty_retries`/synthesis-reserve/注入序列**行为不变**。

- **桩设置**：同一脚本调用序列，分别喂 `_old_station`（dict + UnknownTool 兜底）和 `_new_station`（MCP client），断言两条 `LoopTrace`（闸门计数 + injected）完全相等、且这条脚本触发 synthesis-reserve。
- **判据**：old/new trace 全等；`synthesis_forced==True`。
- **对照组**：故意扰动的 station（吞内容、多走 INNER_RETRY）→ trace 与 old 不一致、被检出——证明回归断言有分辨力、不是空转。
- **预判**：巩固型（blast radius 应够小）。**万一**接 client 扰动了 `turn_count`/inject 路径，就是发现型。
- **注**：此为隔离的最小内层模型；端到端版周四搬进真实 v6.0 图（`test_graph.py`）复跑——与第八周"桩测 → 搬进真实图复跑"两层结构一致。

## 判据汇总表

| E | 验的机制 | 主判据 | 对照组（错的怎么错） | 验哪个决策 | 预判 |
|---|---|---|---|---|---|
| E1 | 工具发现 | 两工具可发现、schema 就位 | 变更后不重发现 → 缓存仍旧 | B | 巩固型 |
| E2 | 派发前守卫拒幻觉名 | `edit` 本地拒、`call_count==0` | 关守卫 → 撞 server 协议 error | E | 巩固型 |
| E3 | `inputSchema` 拒错参数 | 缺/错参数 → INVALID_PARAMS | 合规参数通过 | C、E | 发现型候选 |
| E4 | 两层错误路由分叉 | 协议→escalate / isError→retry，路径不同 | 空结果→retry 而非 escalate | F | 发现型核心 |
| E5 | 空结果≠失败 | 空 `is_error=False`→retry | 三态 reason 两两可分 | C | 巩固型 |
| E6 | 包一层无回归 | old/new trace 全等 | 扰动 station 被检出 | A、G | 巩固型 |

## 与设计草稿的闭环

跑完 E1–E6 回头审草稿 v0.1（第七/八周教训：**实验不止验假设，还是给设计挑错的数据源**）。重点回审：

1. **E3/E4 若把"schema 在哪侧 / 两层 error 什么形态"问成了"桩答不了"** → 在 §四/§五 显式补一句"桩坐实路由逻辑，SDK 形态 [周四确认]"，别把"桩全绿"误读成"决策 E/F 已完全证实"。
2. **E4 若发现协议 error 的 escalate 还需穿到 supervisor 做 skip** → 桩里没有 supervisor，这段（信封回传 + `replan_count+1`）属第八周外层、本周桩不覆盖，scope 诚实记一笔、周四端到端复证。
3. **E6 若扰动了内层** → 按草稿 §0 补 state schema（参照第八周 E5"隔离=显式投影"那种发现型修正）；若无扰动，则**坐实 §0"不新增图级字段"**——两层错误类型是 loop 内瞬态路由信号、不持久化进 `AgentState`。

**无意外也要点明"巩固型 / 发现型 / scope 诚实"**（第七/八周纪律）：本周机制多是 v6.0 接线的升格，预期 E1/E2/E5/E6 为巩固型；真正可能出意外的是 **E3/E4——但意外不在"机制不成立"，而在"桩这种零 API 仪器到底能把决策 E/F 坐实到哪一步"**。带着这个预期跑，重点盯 E3/E4 的 scope 边界。
