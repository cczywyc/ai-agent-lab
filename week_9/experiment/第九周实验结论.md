# 第九周实验结论（E1–E6：MCP 接线机制隔离验证）

> 目的：用纯进程内忠实桩（不接 LLM / 不接真实 MCP SDK / 不碰真实 RAG/HTTP，只建模 MCP 2025-11-25 的 `tools/list` 发现 + `inputSchema` 校验 + 两层错误模型）验证 [设计草稿 v0.1](../docs/第九周设计草稿.md) 锁定的**接线机制**——工具发现 / 派发前守卫迁移（§四）/ 两层错误分流（§五）/ 包一层无回归（§六）——是否成立。零 API、确定、可复现，每条配对照组。
> 对应《第九周实验计划》E1–E6 与草稿 v0.1"待验证清单"6 项 / 决策 A–G。结果并入草稿 v0.2。
> 实测环境：Python 3.12.3（项目 `.venv`）/ pytest 9.1.0；桩仪器 `mcp_stub.py` 为**纯 stdlib**（不依赖 LangGraph / MCP SDK / 网络）。本文所有数据均为 **2026-06-17** 实跑输出：`run_all.py` **总计 6/6**、`pytest` **16 passed**（16 个断言函数）。复现见文末。

1. **巩固型（升格后照旧成立，符合预期）**：E1/E2/E5/E6 —— "发现期=有效工具集权威来源""幻觉名守卫从比对硬编码表升格成比对发现来的清单""空结果≠失败""换 tools station 后内层闸门逐位不变"，全部在桩上坐实。决策 A/G"包一层、blast radius 最小"是本周地基，桩证它没扰动内层。

3. **scope 诚实（桩锁不住的三处）**：① 决策 **D（stdio）完全没被桩验**——桩进程内直调、抽掉了传输，"stdio 链路通"留周四第一次真跑；② **E4 的 escalate 后半段没验**——桩里无 supervisor，只验到"内层把协议 error 分类成 escalate 并退出内层"，"压缩成集成失败信封 → supervisor skip-and-advance（`replan_count+1`）"属第八周外层、周四端到端复证；③ 反过来，**E6 坐实了草稿 §0"不新增 `AgentState` 字段"**——两层错误类型是 loop 内**瞬态路由信号**（算完即被 escalate/retry/inject 消费），不需要持久化进图 state。

## E1：工具发现（`tools/list`）

**验证（决策 B）**：client 经 `discover()`（建模 `tools/list`）拿到 `local_search`/`http_fetch` 的 name + `inputSchema`，且**发现来的清单 = 有效工具集的权威来源**。对照组：变更工具表（`add_tool('summarize')`）→ `listChanged`，重发现前缓存仍旧、重发现后才反映。

**实测输出**：

```
[E1 工具发现] PASS
      discover=['http_fetch', 'local_search']、schema.required 各就位；变更后未重发现仍旧=True；重发现后 summarize 入清单=True
```

| 判据 | 期望 | 实测 | 结果 |
|---|---|---|---|
| 两工具可发现 | `{local_search, http_fetch}` | 同左 | ✅ |
| `inputSchema.required` 就位 | `local_search→query`、`http_fetch→url` | 同左 | ✅ |
| 对照：变更后未重发现 → 缓存仍旧 | summarize 不在清单（`list_changed_pending=True`） | True | ✅ |
| 对照：重发现 → 反映变化 | summarize 入清单（`pending=False`） | True | ✅ |

**结论**：草稿 §一/§四的"权威来源从代码挪到发现期"坐实——client 缓存的发现清单是工具集的唯一权威，工具表变更必须经一次重发现（`listChanged` 同步）才生效，不像 v6.0 硬编码死。巩固型，无意外。

## E2：派发前守卫拒幻觉名（决策 E）

**验证（决策 E）**：确定性探针 `call_tool('edit', ...)`（清单外的名字）→ 守卫**本地拒、不发 `tools/call`**，且守卫比对的是**发现来的清单**（`client._discovered`）。对照组：关守卫 → `edit` 触达 server、撞**运行期协议层 error**。

**实测输出**：

```
[E2 派发前守卫拒幻觉名] PASS
      守卫开 'edit' 本地拒 call_count=0（派发前、从未上线）；守卫关触达 server call_count=1（运行期协议 error）——两路径都在，默认前者
```

| 判据 | 期望 | 实测 | 结果 |
|---|---|---|---|
| 守卫开：`edit` 本地拒 | `METHOD_NOT_FOUND` 且 `call_count==0` | 同左 | ✅ |
| 守卫读发现来的清单（非硬编码副本） | 比对 `client._discovered` | 是 | ✅ |
| 对照：守卫关 → 触达 server 协议 error | `METHOD_NOT_FOUND` 且 `call_count==1` | 同左 | ✅ |

**结论**：巩固型，且确认守卫读的是发现来的权威清单。

**精确化（落点 → §四）——主/对照差在拦截点、不在"拒不拒"**：主路径与对照组**都拒**（都是 `METHOD_NOT_FOUND`），区别全在**拦截发生在哪**：主在**派发前**（`call_count==0`，从未触达 server）、对照在**运行期 server**（`call_count==1`）。这正是草稿 §四"幻觉名守卫**挪不动到 schema 层**、只能在 host 派发前拦或撞 server 协议 error"的落地——守卫从"比对硬编码表"升格成"比对发现来的权威清单"，**但仍是 host 侧控制流**（不是 MCP 替我做的）。两条路径都在、默认走派发前，是为了"幻觉名根本不上线、省一次 RPC"。

## E3：`inputSchema` 拒错参数（决策 C/E）★发现型

**验证（决策 C、E）**：给 `local_search` 传缺 `query` / 类型错的参数 → schema 层拒（`INVALID_PARAMS`）；合规参数正常通过。seam：把校验切到 client 侧（`client_validates=True`）应等价拒绝。

**实测输出**：

```
[E3 inputSchema 拒错参数] PASS
      缺 query→INVALID_PARAMS(-32602)；类型错→INVALID_PARAMS(-32602)；合规通过 chunks=1；校验切 client 侧等价拒=True（桩默认 server 侧校验，client_validates=False）
```

| 判据 | 期望 | 实测 | 结果 |
|---|---|---|---|
| 缺必填 `query` | `INVALID_PARAMS` | -32602 | ✅ |
| 类型错（`query=123`） | `INVALID_PARAMS` | -32602 | ✅ |
| 合规参数通过 | `is_error=False`、chunks=1 | 同左 | ✅ |
| seam：校验切 client 侧等价拒 | `INVALID_PARAMS` | 同左 | ✅ |

**结论**：这是草稿 §四里 **MCP 唯一实打实把守卫前移到 schema 层**的地方——"给真工具传错参数"能在执行前被 `inputSchema` 拒掉（区别于幻觉名只能换比对对象）。桩上坐实：错参数在 schema 层被拒、且 client/server 两侧建模**等价**。

**发现型（落点 → §四 / 实验计划 E3）——计划措辞被修正**：实验计划与草稿 §四原话是"schema 校验在 **client 或 server 侧**……**今天就要拿桩问出来到底在哪一侧 fire**"。本次落地把这条**修正**了：**纯进程内零 API 桩问不出"真实 SDK 在哪侧 fire、抛异常还是回协议 error"——那是真实 SDK 的外部行为，不是桩能结案的**。桩默认建模 server 侧（`client_validates=False`）、并证 client 侧 seam 等价拒；真正的"哪侧"答案是 **[周四确认]**（换真实 FastMCP 时观测，把 `client_validates` 设成与其一致即可）。→ §四该把"拿桩问出 schema 在哪侧"收紧为"桩坐实'无论哪侧、路由都把它当协议层 `INVALID_PARAMS`，且两侧等价'；真实 SDK 哪侧 fire 留周四"。

## E4：两层错误路由分叉（决策 F 核心）★发现型·核心，跨两周悬念结案

**验证（决策 F）**：桩 server 分别注入协议层 error 和 `isError` 业务失败 → 验证前者路由 `ESCALATE`（上报 supervisor 走 skip）、后者 `INNER_RETRY`（内层 `empty_retries`），两层走**不同**路径。对照组：空结果（`isError=False`、chunks 空）走 `INNER_RETRY` 而非 `ESCALATE`。

**实测输出**：

```
[E4 两层错误路由分叉] PASS
      协议error→escalate_to_supervisor；isError→inner_retry；空结果→inner_retry≠escalate_to_supervisor（协议层上报 skip / 执行层进内层 retry，两层真分得开）
```

| 判据 | 期望 | 实测 | 结果 |
|---|---|---|---|
| 协议层 error | `ESCALATE` / `protocol_error` | 同左 | ✅ |
| `isError` 业务失败 | `INNER_RETRY` / `business_error` | 同左 | ✅ |
| 两层路径不同 | `ESCALATE != INNER_RETRY` | 同左 | ✅ |
| 对照：空结果 | `INNER_RETRY` / `empty_result`（≠ `ESCALATE`） | 同左 | ✅ |

**结论（跨两周悬念结案）**：第三周《Agent Loop 设计笔记》Q3、第八周 Q3 那个"纠正/降级在哪层生效"——在单 agent / 同质失败信号下分不开、悬了两周——在 MCP 的 typed 失败信道下**第一次有素材分开**。桩坐实：`route_tool_outcome` 用 `isinstance` 在派发点把三态（协议 error / 业务失败 / 空结果）**干净二分到三条恢复路径**，互不混淆。决策 F 的两层映射在 harness 层成立。

**发现型·核心 + scope 诚实（落点 → §五）——干净二分建立在一个建模假设上**：这个干净二分**依赖**"协议 error 以 **Python 异常**（`MCPProtocolError`）形态、`isError` 以**结果字段**（`ToolResult.is_error`）形态出现"——而草稿 §五正是**这么假设**的。所以桩证的是**设计的内部自洽**（决策 F 的路由逻辑没写歪），**不是真实 SDK 的外部行为**。两处 [周四确认]/scope：

1. **SDK 形态依赖 [周四确认]**：真实 FastMCP/SDK 把两层暴露成什么形态，决定 outcome 怎么被 `isinstance` 接住。若形态与假设不符（如两层都走异常、或都塞进结果字段），`route_tool_outcome` 的二分依据要改——**那就是要折回 v0.2 的发现**。这是周四最该盯的一刀。
2. **escalate 后半段桩没覆盖（scope）**：桩只验到"**内层**把协议 error 分类成 `ESCALATE` 并退出内层 loop（`break`）"。"压缩成集成失败信封 → **supervisor** 走 skip-and-advance（`replan_count+1`）、不穿透成全局崩溃"这后半段——桩里**没有 supervisor**、没验，它属第八周外层决策 H 的事，周四搬进真实 v6.0 图端到端复证。

## E5：空结果 ≠ 失败（决策 C，E4 二分的地基）

**验证（决策 C）**：`local_search` 桩回空 chunks（`isError:false`）→ 不被误判成 error、由内层重试接管。对照组：遍历 ok/error/empty 三 mode，三态 reason 两两可分。

**实测输出**：

```
[E5 空结果≠失败] PASS
      空 is_error=False/content=[]→empty_result/inner_retry；三态 reason=['business_error', 'empty_result', 'ok'] 两两可分（loop 才能各走各的恢复路径）
```

| 判据 | 期望 | 实测 | 结果 |
|---|---|---|---|
| 空结果 `is_error` | False | False | ✅ |
| 空 `content` | `[]` | `[]` | ✅ |
| 路由 | `empty_result` / `INNER_RETRY` | 同左 | ✅ |
| 对照：三态 reason 两两可分 | `{ok, business_error, empty_result}` | 同左 | ✅ |

**结论**：草稿 §二/§五"空 chunks 是**业务结果非错误**"坐实——查无命中（`isError:false`、chunks 空）是正常业务结果，不是协议 error、不上报，由内层"空回答重试"接管。它是 E4 三态二分的地基：唯有"空 / 业务失败 / 成功"三 reason 两两可分，loop 才能各走各的恢复路径。巩固型。

## E6：包一层无回归（决策 A/G）

**验证（决策 A、G）**：researcher 的 `tools` 站从"v6.0 硬编码 dict 派发 + UnknownTool 兜底"换成"MCP client 派发"后，内层 `turn_count`/`empty_retries`/synthesis-reserve/注入序列**逐位不变**。对照组：扰动的 station（吞内容→多走 INNER_RETRY）→ trace 与 old 不一致、被检出。

**实测输出**：

```
[E6 包一层无回归] PASS
      old/new station trace 全等 turn_count=4/empty_retries=0/synthesis_forced=True、injected 全等；扰动 station 被检出=True
```

| 判据 | 期望 | 实测 | 结果 |
|---|---|---|---|
| old/new 闸门计数全等 | `turn=4, empty=0, synth=True` | 同左 | ✅ |
| `injected` 序列全等 | 4 项逐位相等 | 同左 | ✅ |
| synthesis-reserve 触发 | `synthesis_forced=True` | True | ✅ |
| 对照：扰动 station 被检出 | trace ≠ old | `old2=(4,0,True)` vs `buggy=(5,2,False)` | ✅ |

**结论**：决策 A/G"包一层、blast radius 最小"坐实——`tools` 站换实现（硬编码 dict → MCP client），内层 `turn_count`/`empty_retries`/synthesis-reserve/inject 序列**逐位不变**。闸门全在 loop 里、与 station 实现无关，这正是"包一层"的设计主张：换的是工具怎么被调到，没换闸门怎么收口。

**精确化（落点 → §0）——坐实"不新增图级字段"，非发现型修正**：草稿 §0 说"若桩测发现需把'两层错误类型'显式记进 state，再在 v0.2 补"。本次结果**反向坐实了不需要**：两层错误类型在 loop 里是**瞬态路由信号**（`route_tool_outcome` 算出 `ESCALATE/INNER_RETRY/SUCCESS` 后立即被消费——escalate 就 break、retry 就 `empty_retries+1`、success 就 inject），**用完即走、不跨 super-step 持久化**，故不需要新增 `AgentState` 字段。对照组也证回归断言**有分辨力、不是空转**：扰动（吞内容→每次走 INNER_RETRY）使 `empty_retries` 从 0 涨到 2、`turn_count` 从 4 到 5（`(4,0)`→`(5,2)`），被逐位检出。

## 周四（v7.0 实现）技术决策清单

桩测结果对周四实现的直接约束：

1. **工具发现用真实 `tools/list`，派发前守卫读发现来的清单**（非硬编码副本）——守卫比对 `discovered_names`，幻觉名派发前拦（E1/E2）。
2. **`inputSchema` 校验**：错参数 → `INVALID_PARAMS` 路径保留；**周四对真实 FastMCP 观测校验在哪侧 fire / 什么形态，把 `client_validates` 设成与真实 SDK 一致**（E3，本周问不出的那条）。
3. **两层错误分流（决策 F）是周四最该盯的一刀**：`route_tool_outcome` 的 `isinstance` 二分依据，**周四按真实 SDK 的协议 error / isError 实际形态对齐**——若两层不是"异常 vs 结果字段"，二分逻辑要改、折回 v0.2（E4）。
4. **escalate 后半段端到端复证**：协议 error → 集成失败信封 → supervisor skip-and-advance（`replan_count+1`），桩没覆盖 supervisor 那端，搬进 v6.0 图复证（E4 scope）。
5. **stdio 链路第一次真跑**：决策 D 未被桩验（桩抽掉传输），周四在本机把 server 跑通、client 经 stdio 调它（E/D）。
6. **不新增 `AgentState` 字段**（E6 坐实 §0）——两层错误类型是 loop 内瞬态信号；除非端到端发现需持久化，否则不动图 state。
7. **包一层无回归**：桩层 old/new 全等，**周四搬进真实 v6.0 图（`test_graph.py`）复跑** `turn_count`/synthesis-reserve（E6 注，与第八周"桩测 → 真实图复跑"两层结构一致）。
