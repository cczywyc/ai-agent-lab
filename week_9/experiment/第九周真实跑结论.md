## 核心问题（喂周六）

### Q1：决策 F 健康中段——host 重建的单层失败信号，真能让活模型跑顺主力路径吗？

**能，且没拖累 healthy 路径——但"模型从失败中恢复"这条没被活模型自然触发（scope）。**

- **healthy 路径干净**：T2 模型 8 次工具调用**全 success**、`escalated=False`、`grounding=1.0`，全程没有 spurious escalate/retry。周四把"两层错误塌成 isError、host 派发前重建二分"挪了判据，**没有在主力路径上引入误伤**——真实模型经 MCP 把工具调顺了。
- **routing 在真实信号上分得开**：T3′ 守卫探针实测——幻觉名 → `escalate_to_supervisor/pre_dispatch_reject`、错参数 → `escalate_to_supervisor/invalid_args`、业务失败 → `inner_retry/business_error`。三态在**真实 SDK 的 isError 信号**上仍被 host 干净二分到三条恢复路径。
- **诚实**：T2/T3 的**活模型 loop 里没有自然发生业务失败**（8+4 次调用全 success），所以"单层 isError 信号帮**模型**recover"这半边**没被自然压到**——失败分流是拿确定性探针（T3′）验的，不是活模型撞失败后自己 recover 验的。同第八周"best-so-far 双 reject 未触发"，记 coverage 缺口。

### Q2：主轴坐实——周四的发现型（框架失败词汇表更弱、控制流仍归我；守卫从可选→承重墙），活模型下是被更印证还是被推翻？

**被更印证。** 守卫承重墙在活管线里实测兜住（T3′ 幻觉名 pre_dispatch_reject、错参数 invalid_args——都没触达 server、没被误当业务失败 retry）；host 重建的二分在真实 isError 信号上分得开；healthy 路径没因"单层塌缩"退化。**没有一处实测把周四的发现型推翻。** 框架（FastMCP）只给一层 isError、两层控制流靠我在 host 重建——这条在真实模型 + 真实 stdio 上成立。

## 真实跑坐实的（巩固型，real 级）

1. **健康中段主力路径（T2）**：模型对"本项目版本演进"题**只选 local_search**（8 次、零 http_fetch 漂移）、8/8 success、综合出 4331 字带 `[doc#section]` 引用的答案、**10/10 引用字面接地（grounding=1.0）**。这是确定性脚本根本验不到的——真实模型自己判断该调哪个工具、调几次、何时停。
2. **守卫承重墙活体坐实（T3′）**：周四"决策 E 守卫升成承重墙"在活管线实测兜住——幻觉名 `edit` → `pre_dispatch_reject`（本地拒、不触达 server）、错参数 → `invalid_args`（client 侧 schema 校验拦）、业务失败（黑名单域）→ `business_error/inner_retry`。三态分流与周四 verify 完全一致，**在真实信号上不是桩里的假设、是实测**。
3. **synthesis-reserve 真实跑触发（T2/T3）**：两题 `turns=5` 都跑到 `SYNTH_RESERVE_AT` 被逼综合——尤其 T2 模型想连调 8 次 local_search，synthesis-reserve 在最后一轮**逼它停手综合**（防过度检索撑爆 turn 预算）。继承 v5.0 的这道闸门在真实模型 over-retrieve 时 load-bearing。
4. **答案级接地诚实（T2）**：语料是 weeks 2–7、**v6.0 不在库**。模型检索后**诚实写**"项目版本演进只到 v5.0，第八周多 Agent/supervisor 仍在思考/规划阶段、尚未落地"，grounding=1.0、**没有从参数知识里编造 v6.0 细节**。被问 v6.0 而库里没有时，模型接地在真实命中的 v5.0 内容上、标出缺口——这正是接地纪律最该有的行为。

## 真实跑挖到的（发现型）

### 引用格式漂移 → 接地软闸门**字面失配**（T3，grounding 0.167）

- **现象**：T3（generic 措辞题）模型同样只调 local_search（4/4 success），但最终答案 6 条引用**只有 1 条**字面命中检索白名单（grounding=0.167），对比 T2 的 1.0。
- **机制（细看 friday_run.json）**：T3 的 6 条 answer 引用**指向的多是真实本地文档**，但模型把 `[doc#section]` 字符串**重排了**——把整段标题塞进 doc 位（`第七周学习复盘：多步骤任务规划（planner-executor-critic 技术调研 Agent）#七、…`），而 local_search 返回的格式是 `第七周学习复盘#第七周学习复盘：…`。**于是接地检查（精确字符串匹配）大面积失配**——0.167 是"引用真实但格式漂移"+"generic 题下确有松散引用"两者叠加，不是纯编造。
- **为什么 T2 是 1.0 而 T3 是 0.167**：T2（强项目语境题）模型**逐字照抄** local_search 给的 citation；T3（generic 题）模型倾向**自己改写** citation 串。→ 接地软闸门的"精确字符串匹配"对**citation 格式保真度**敏感：模型照抄则满分、改写则骤降，即便底层来源都真。
- **这其实坐实了接地闸门是 load-bearing**：T3 的 0.167 < 0.5——在 v6.0 完整图里，researcher 内层 critic 的接地软闸门会**判 retry**（正如第八周真实跑 B/C 抓引用堆砌），逼模型基于真实命中重写。**但本周 minimal real loop 没接这道 critic 闸门**（scope，见下），所以 T3 的低接地**没被拦下**——这反过来说明：healthy 路径不只靠"选对工具"，还靠接地闸门兜"答案别乱引"。
- **处置（记 v0.4，不做未复跑验证的改动）**：① local_search 返回一个**稳定的 citation id**、prompt 要求**逐字回显**（降低格式漂移）；② 或接地检查改**归一化/模糊匹配**（容忍 doc/section 重排）。两条都待下次真实跑复证，本次不改（承第八周处置纪律）。

## scope 诚实（本周没覆盖的，及为什么）

1. **完整 supervisor 图整合（评审 Task 1）+ supervisor 端 escalate skip（Task 4 后半段）未做**。理由三条：① 设计草稿 §六/决策 G **既定 scope 就是"researcher 内层 + MCP server"**；② async-in-LangGraph（把同步 `tools()` 节点接异步 MCP client + 协调 session 生命周期 + 重整工具名/结果 shape）是更大一刀，且要跑完整 supervisor→writer→reviewer 栈 = 大量 LLM 调用，**免费额度（qwen3.7-plus 也是 free tier）有炸穿风险**（第八周 max 就是这么耗尽的）；③ **内层 ESCALATE 本周已 real 验**（T3′），它喂的 **supervisor skip-and-advance / `replan_count+1` 机器是第八周 topic D 已 real 坐实的件**（`replan_count=2`）——没必要为复证已验件烧额度。→ Task 1/4 留作独立增量（async-in-LangGraph + 真实主力路径合并做），记 v0.4。
2. **模型自然撞失败后 recover 未触发**：T2/T3 活 loop 里 12 次调用全 success，业务失败/escalate 只由确定性探针（T3′）压出。"单层 isError 帮**模型**recover"这半边没被活模型自然验（要一个会自然失败的题，如逼模型 fetch 一个真会 403 的源——难稳定控制）。同第八周 best-so-far 未触发，记 coverage 缺口。
3. **语料 coverage**：库 = weeks 2–7（week_8/v6.0 故意不在），T2 的 v6.0 题命中真实缺口、被模型诚实标出。T2 因此验的是"健康中段选对工具 + 接地 + 诚实标缺口"，**不是**"v6.0 细节召回"。不重 ingest（额度 + 库的既定 scope）。
4. **T3 grounding 0.167 别过度解读成"模型乱编"**：如上，主要是 citation 格式漂移导致的字面失配，底层来源多为真实本地文档。
