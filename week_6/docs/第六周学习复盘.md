## 三、给上周预判打分（本周独有的反思）

第四五周复盘那节"第六周（LangGraph）"做了三条预判。这周逐条兑现：

| 上周预判 | 本周结果 |
|---|---|
| `memory/` → checkpointer（thread 内短期）+ store（跨 thread 长期） | ✅ 完全兑现：v4.0 用 checkpointer 接管图状态；**v4.1 把长期记忆迁进了 SqliteStore**，store 这半也落地了 |
| 最想让框架接管"持久化 + 时序"那块脏活 | ✅ 兑现：checkpointer 接走图状态持久化；踩坑 #2 的时序变成"读在 assemble、写在 update_memory"的节点顺序；v4.1 后**长期记忆的手动 `save()` 六件套也由 store 接管，两个 `embed_*` call site 消失** |
| 装配的"挑选"逻辑想自己留着 | ✅ 兑现：`assemble` 节点原样保留为核心；迁移时刻意把 `top_k`/`min_score`/段预算的过滤留在自己这侧（离线测试 S6 钉住） |

三条核心预判全部兑现——说明上周"先体会痛点、再看框架怎么消除痛点"的判断方向是对的。手写过 memory/ 六模块的人，迁 Store 时对"哪些该交、哪些该留"几乎没有犹豫。

## 四、关键决策

### 六个设计决策（草稿 v0.2 锁定，v4.0 落地）

| # | 决策 | 非显然之处 |
|---|---|---|
| A | 本周只迁控制流 + 短期状态，Store 推迟 | 分两步走，blast radius 最小（v4.1 才补 Store） |
| B | `retrieved_chunks` 手动累加 + init 清空，**不上 reducer** | E3 实证：带 `operator.add` 的字段 `return []` 是 no-op 追加、清不掉 |
| C | per-query 标志由入口 `init` 节点重置 | E2 实证：TypedDict 字段**无隐式默认**，init 是双重职责（首轮建默认 + 后续重置） |
| D | 纠正拆两个节点、共用一函数体 | 把"检索优先、联网其次"的优先级画进图里，代码不重复（`functools.partial`） |
| E | 长期记忆迁 LangGraph Store | 本周从"推迟"变"落地"——见下 |
| F | `human_review` interrupt 放 `finalize` 前、默认关 | 只读 agent 上是演示性质；真正刚需在第七周写工具后 |

### Store 选型：从"判死"到"选定 C-SqliteStore"

v0.2 当时把"原生 Store"判死（以为只有内存版）。本周调研发现 **SqliteStore 提供本地文件持久化 + 语义搜索**，前提不成立了，于是 v0.4 改判为**选 C-SqliteStore**：

- 决策 E 的初衷就是体验真实 Store；"包一层 BaseStore"大半精力在实现抽象方法、反而绕过 Store 本身。
- SqliteStore 解决了唯一硬伤（InMemoryStore 重启即失）。
- facts 向量从"和文档共库"挪进 store 自管——E2c 的"共用向量库"让位，"共用 embedding 模型"保留（`ltm_store.embed_for_store` 适配器），概念上更干净：文档=RAG 语料，事实=长期记忆，本就该分家。

## 五、实验与量化

### 框架机制验证（E1–E5，桩节点隔离）

每条都配了**对照**（验"对的能跑"也验"错的怎么错"）：E1 状态传递 + add_messages 累加；E2 init 重置 × checkpointer 持久化；E3 chunks 累加/清空；E4 cycle × turn_count 闸门；E5 interrupt 开关。全过，桩验证后由 `test_graph.py::T8` 在**真实节点**上再锁一遍。

**意外发现（E2）**：TypedDict 字段无隐式默认值——无 init 时首轮标志是 `None`（裸取 KeyError），不是 `False`。这把 init 的职责从"跨轮重置"升级为"首轮建默认 + 跨轮重置"，直接改进设计（v0.2→v0.3）。这是"实验教会我没预料的事"的典型，比验证已知假设更值。

### Store 迁移验证（v4.1，真实 embed，2026-06-05 实跑）

| 验证项 | 数字 |
|---|---|

## 六、踩坑与本周发现

### 计划外发现并修复的两个真问题（都补了回归测试）

1. **v4.0 潜伏 bug：装配窗口切掉段 1–3。** `nodes._window_start` 取"最后一条 system 消息"做窗口起点；但记忆开启时 assemble 产出**多条** system（段 1/2/3/4），起点落到段 4——**发给模型的窗口丢了 SYSTEM_PROMPT、偏好、摘要**。v4.0 的 T1–T8 全 `use_memory=False`，从未暴露。修复：锚点改为"内容等于 SYSTEM_PROMPT 的那条 system"。回归测试 `S10`。
   > **教训**：feature 关着时测试全绿 ≠ feature 开着时正确。这正是周四 review 标的头号风险"memory-ON 端到端没测过"——一开记忆，潜伏 bug 立刻现形。测试覆盖的盲区会精确地藏在"没开的那条路"上。
2. **空回答污染记忆。** qwen 偶发返回空 content（`finalize` 已有 `[模型返回空回答]` 占位）；但 `update_memory` 会把 `assistant=""` 的轮次写进短期记忆，下一问段 5 带着空轮诱发答非所问（复现一次）。修复：空回答跳过本轮记忆更新。回归测试 `S11`。

### 新增框架坑

- **踩坑 #4：SqliteStore 必须 `isolation_level=None`（autocommit）**，否则原生 sqlite3 的隐式事务与 store 内部 `BEGIN` 冲突。`from_conn_string` 已如此配置，但它是 contextmanager；进程级长连接要自建连接并显式设置。

### 定稿复跑揪出的新发现（2026-06-05，未修，先记录）

1. **空回答不是"偶发"，在 qwen3.7-plus 上接近两成。** 复跑当天 18 个真实问题里 4 个最终回答为空（`--test` Case 5、demo 第 4 轮、review 试探第 1 问、同题三连发 1/3——同一首五言绝句连跑 3 次，第 2 次空）。特征一致：空回答那次模型调用都**极快**（0.8–3.5s，正常 5s+），更像 API 侧 fast-fail 而非生成中断。S11 当场证明了价值——demo 第 4 轮空回答后 `short_term_turns=3`、`summary_chars=0`，空轮确实没写进记忆。但这个频率说明只"防污染"不够：agent 节点应该考虑对空 content 重试一次（挂给第七周）。
   > 顺带更正：复盘初稿和验证记录里写的模型是 qwen-plus，实际 `341c39c` 起 `config.MODEL` 已改为 **qwen3.7-plus**——v4.1 的全部验证（含 06-04 的 8 用例）都跑在 qwen3.7-plus 上。空回答频率变高很可能与换模型相关，v3.0 时代（qwen-plus）的"偶发"印象不能平移过来。
2. **Case 5 是假阳性 PASS——`--test` 判据自己的盲区。** 该用例回答是 `[模型返回空回答]` 占位符，但 `has_answer` 只排除 `[达到最大轮次]` / `[错误]` 两个前缀，占位符被当成了有效回答：工具路径全对、用户实际没拿到答案、报告记 PASS。这正是窗口切片 bug 那条教训的判据版——**盲区精确地藏在"判据没编码的那个值"上**。修法很便宜（前缀名单加一项），但动判据就动了与 v3.0/06-04 报告的可比性，先记录，留给第七周连同"Case 8 判据编码"一起重审。
3. **两个小 quirk（无害，记录在案）**：
   - demo 第 2 轮"请记住：以后回答涉及本地文档时…"触发了检索纠正——关键词启发式把含"本地文档"的**元指令**当成了文档问题。纠正一次性、被模型平滑接住，偏好照常抽取成功，但说明 `should_have_retrieved` 不区分"问文档"和"谈论文档"。
   - `human_review` 把**空草稿**也送审了（审批框里是空字符串——finalize 的占位符在审批**之后**才补）。歪打正着：人工有机会当场改写补救空回答；但也再次佐证周四 review 那条"`update_memory`/`human_review` 时序"确实值得清。

### 仍开口的旧坑

- **踩坑 #3（事实抽取依赖 `[doc#section]` 引用格式）未解。** 现有 11 条事实里 #6/#7 就是该格式不稳带来的抽取噪音，本次按计划原样迁移不处理。Store 不碰抽取侧，所以这条既没被解决也没被恶化——但事实越攒越多、第七周开始跨任务积累，这个契约不严会被放大，**第七周前应收紧**（上周就标了，本周顺延）。

## 七、对后续 Roadmap 的思考

第七周是"多步骤任务规划"（planner-executor、decomposition、reflection/critic、终止与失败回滚），实战做技术调研 Agent。本周给它打好了三块地基：

- **节点 vs 边的拆分能力直接复用**：planner / executor / critic 天然是节点，终止条件和失败回滚天然是条件边。本周把嵌套 while 拍平成图的训练，正是 planner-executor 的预演。
- **长期事实 + 语义召回 = 跨任务状态保持**（上周已预埋）：v4.1 把事实放进了**正经的跨 thread store**，planner 把子任务结论写进 store、executor 在新子任务里 `search` 召回——比 v4.0 压在手写库上更顺。**所以 Store 迁移排在第七周之前是对的排序**。
- **interrupt 彩排兑现**：本周放 finalize 前练的是输出审批；第七周调研 Agent 仍只读，但等后续接写工具，同一机制挪到"不可逆动作前"即可——结构性教训已就位。

一个排序提醒：周四那几个未完重构项（`tools` 节点瘦身、`update_memory`/`human_review` 时序）建议在第七周加复杂度之前先清掉，否则会在更大的图上放大。定稿复跑（§六）后这份清单再加两项：**空回答的 agent 节点重试**（qwen3.7-plus 上频率约两成，planner-executor 的多步链上空一步会废掉整条链）和 **`--test` 判据重审**（`has_answer` 占位符假阳性 + Case 8 边界编码）。

## 八、跨模型 / 通用设计原则（本周新增，每条挂证据）

- **节点干活、边做决策**（认知）：把"检查"归边、"注入/执行/收尾"归节点，第三周 150+ 行嵌套循环被拍平成 10 节点。这条规则是状态图思维的核心。
- **state 是活数据、checkpoint 是时间轴快照；checkpointer ≠ store**（概念）：thread 内短期 vs 跨 thread 长期，混用是新手最常见的架构错配——偏好塞 state 换 thread 就没了，短期塞 store 跨 thread 串味。
- **顺序图别发明 reducer**（state.py 实证）：无并行分支写同一字段时，唯一需要的 reducer 是 `messages` 的 `add_messages`，其余替换；要累加就在节点内手动做（决策 B）。
- **先在桩节点隔离验证框架机制，再上真实现**（E1–E5 → T8）：把"框架行为"和"我的业务 bug"分开验，零 API、确定、可复现。
- **feature 关着的测试全绿 ≠ feature 开着时正确**（窗口切片 bug）：测试覆盖的盲区会精确藏在"没开的那条路"上；新增可选能力时，端到端开着跑一次是必须的。
- **迁移验证标准定在行为等价，不是位级一致**（召回对照）：重新 embed 必有数值噪声，hit@k 不退化才是正确的验收口径。
- **判据没编码的值，就是下一个盲区**（复跑 Case 5 实证）：`has_answer` 漏了一个占位符前缀，空回答就成了 PASS。测试判据和被测代码一样会有 bug，且它的 bug 永远表现为假阳性——"全绿"里要先怀疑判据。
- **复盘数字要在定稿当天重跑出来**（2026-06-05 全量复跑实证）：通过率"持平 88%"的表面下，藏着假阳性、纠正路径首次真实触发、平均轮次/耗时翻倍的方差、以及空回答从"偶发"到"两成"的频率修正。不复跑，这四条全都看不见。

## 九、本周最大收获

**把一个手写的循环式 Agent 重写成状态图，本质不是"换个写法"，而是把"控制流"和"数据流"解耦——控制流交给图的拓扑（节点+边），数据流交给 state + 持久化（checkpointer/store），自己只保留真正的业务判断（装配挑选、纠正规则）。** 第三周那个会"分支爆炸"的循环之所以痛，就是因为三者全缠在一个循环体里；状态图把它们拆开，痛点随之消失。

而本周比"重写跑通"更深的一课，来自那个潜伏的窗口切片 bug：**我在周三周四把"按最后一条 system 切窗口"当成优雅设计，它在记忆关闭时一路全绿；直到 v4.1 把记忆打开，它立刻暴露成"模型看不到 system prompt"。** 测试的绿，只覆盖了我走过的那条路。这让我对"验证"有了更具体的敬畏：不是"测试过了"，而是"我开着每一条会真实走的路测过了"。
