## E1：状态在节点间传递 + add_messages 累加

**验证**：`messages` 用 `add_messages` 单 reducer、其余字段默认替换，状态能否在 `assemble → agent → finalize` 三个桩节点间正常流动。

**实测输出**：

```
最终 marker      : 'set_by_assemble'
最终 messages 条数: 3
   - msg from assemble
   - msg from agent
   - finalize saw marker=set_by_assemble
```

| 判据 | 期望 | 实测 | 结果 |
|---|---|---|---|
| 替换字段跨节点可见（C 读到 A 写的 `marker`） | `'set_by_assemble'` | `'set_by_assemble'` | ✅ |
| `messages` 累加而非覆盖 | 3 条（每节点各追加 1 条） | 3 条 | ✅ |

**结论**：草稿 §1 的 schema 设计成立——节点返回**部分 dict** 即可更新状态；`add_messages` 字段逐节点追加，替换字段写一次、下游全可见。另观察到 dict 消息会被框架转成 LangChain 消息对象（取 `content` 时需兼容两种形态），周四实现时注意。

## E2：init 重置 × checkpointer 持久化（本次核心）

**验证（决策 C）**：同一 `thread_id` 连续两次 invoke：(a) `messages` 历史跨轮累加；(b) per-query 标志被入口 `init` 节点打回初值。对照组：同一张图去掉 `init`，证明"不重置就泄漏"。

**实测输出**：

```
[有 init 的图]
  两轮后 messages 条数         : 4  (期望 4: Q1,A1,Q2,A2)
  agent 每轮进入时看到 has_retrieved: [False, False]  (期望 [False, False])

[无 init 的图 — 对照]
  两轮后 messages 条数         : 4  (期望 4: 持久化与 init 无关)
  agent 每轮进入时看到 has_retrieved: [None, True]  (期望 [None, True])
```

| 判据 | 期望 | 实测 | 结果 |
|---|---|---|---|
| `messages` 跨轮持久化 | 4 条 | 4 条 | ✅ |
| `init` 重置 per-query 标志 | `[False, False]` | `[False, False]` | ✅ |
| 对照：无 `init` 时标志泄漏 | `[None, True]` | `[None, True]` | ✅ |

**结论**：决策 C 坐实。关键不对称：`init` 对 `messages` **不返回该键 = 保留**（持久化继续累加）；对替换字段返回 `[]`、对标志返回 `False` = 清空/重置。"跨轮记历史、per-query 清零"两者兼得。

### ⚠️ 意外发现（唯一，建议进 v0.3）

对照组第一轮 `has_retrieved` 是 **`None` 而不是 `False`**——TypedDict 字段**没有隐式默认值**（裸取 `state["has_retrieved"]` 会 KeyError，探针里用的 `state.get()`）。

→ `init` 节点的职责不止草稿假设的"跨轮重置"，还包括"**首轮建立默认值**"——比草稿假设的更必要、不可省。
→ **落点**：v0.3 在 §1/§2 给 `init` 的职责补一句"为所有 per-query 字段建立首轮默认值"；周四实现时所有条件边读标志一律走 `state.get(k, 默认)` 或保证 `init` 先行。

## E3：retrieved_chunks 轮内累加 + 跨问题清空

**验证（决策 B）**：(a) `tools` 节点手动"当前+新"累加（替换语义），单 invoke 内多次检索能累起来；(b) `init` 返回 `[]` 能干净清空。对照组：同图放一个 `bad_chunks: Annotated[list, operator.add]`，`init` 同样对它返回 `[]`，验证草稿"为何不给 chunks 上 reducer"的推理。

**实测输出**：

```
[Invoke 1 — 一个问题内检索 2 次]
  retrieved_chunks: ['chunk_0a', 'chunk_0b', 'chunk_1a', 'chunk_1b']
  bad_chunks      : ['chunk_0a', 'chunk_0b', 'chunk_1a', 'chunk_1b']

[Invoke 2 — init 之后、再次检索之前的快照]
  retrieved_chunks: []
  bad_chunks      : ['chunk_0a', 'chunk_0b', 'chunk_1a', 'chunk_1b']   ← 清不掉
```

| 判据 | 期望 | 实测 | 结果 |
|---|---|---|---|
| 轮内累加（单 invoke 检索 2 次累成 4 条） | 4 条 | 4 条 | ✅ |
| 跨问题清空（替换字段 `init` 后为空） | `[]` | `[]` | ✅ |
| 对照：`operator.add` 字段 `init` 清不掉 | 4 条残留 | 4 条残留 | ✅ |

**结论**：决策 B 坐实，且草稿里的推理拿到实证——带累加 reducer 的字段，`return []` 是"追加空列表"的 no-op，**无法重置**。所以 `retrieved_chunks` 选"替换语义 + `tools` 节点手动累加"，`init` 才能用 `[]` 干净清空；"只有 `messages` 用 reducer"的简洁得以保住。

## E4：纠正 cycle × turn_count 闸门

**验证（设计②）**：回连边（`inject_* → agent` 的 cycle）配 `turn_count` 闸门能正常终止，在框架递归上限之前先收口。对照组：拆掉闸门、`agent → agent` 死循环，单靠 `recursion_limit=8` 兜底。

**实测输出**：

```
[有 turn_count 闸门]
  跑完轮次: 6  (MAX_TURNS=6)
  log     : ['turn 1', 'turn 2', 'turn 3', 'turn 4', 'turn 5', 'turn 6']

[无闸门 — 对照]
  抛出 GraphRecursionError(如期):Recursion limit of 8 reached without hitting a stop conditio...
```

| 判据 | 期望 | 实测 | 结果 |
|---|---|---|---|
| 闸门在框架上限前收口 | 停在 turn 6 | 停在 turn 6 | ✅ |
| 对照：无闸门撞 `recursion_limit` | 抛 `GraphRecursionError` | 抛 `GraphRecursionError` | ✅ |

**结论**：cycle 不死循环，v3.0 的 `MAX_TURNS=6` 显式语义在图上原样保留；框架 `recursion_limit` 只当最后兜底，**不该靠它来正常终止**（它的终止方式是抛异常，不是优雅收尾到 `finalize`）。

## E5：human_review interrupt 开关透明性

**验证（决策 F）**：`human_review` 节点内 `if INTERRUPT_ENABLED: interrupt(...)`——关闭时完全透明；打开时在节点处暂停（stream 吐 `__interrupt__`），`Command(resume=...)` 后接着跑完。注：`interrupt` 必须配 checkpointer。

**实测输出**：

```
[A 开关 OFF]
  出现 __interrupt__ : False
  review 字段        : 'auto(disabled)'
  跑到 finalize      : True

[B 开关 ON]
  出现 __interrupt__ : True
  暂停在节点         : ('human_review',)
  resume 后 review   : 'approved'
  resume 后跑完      : True
```

| 判据 | 期望 | 实测 | 结果 |
|---|---|---|---|
| OFF 透明放行（无 interrupt、直达 finalize） | 无 `__interrupt__` | 无 | ✅ |
| ON 在 `human_review` 暂停 | `next == ('human_review',)` | `('human_review',)` | ✅ |
| ON `Command(resume=)` 后跑完 | `review='approved'`、`next` 为空 | 一致 | ✅ |

**结论**：决策 F 坐实。`INTERRUPT_ENABLED` 默认关时图完全透明，批量评测可复现；打开时暂停/恢复行为符合预期。开关放节点内（而非建图时裁剪节点）即可，图结构不用变。

## 周四（v4.0 实现）技术决策清单

实验结果对实现的直接约束：

1. **`init` 节点不可省，且职责升级**：跨轮重置 + 首轮建默认值（E2 意外发现）。条件边读 per-query 标志一律 `state.get(k, 默认)` 防 KeyError。
2. **`retrieved_chunks` 严禁上 `operator.add`**：保持替换语义，累加在 `tools` 节点手动做 `state["retrieved_chunks"] + new`（E3 实证 reducer 字段无法重置）。
3. **`init` 不返回 `messages` 键**：返回了就会动持久化历史；只返回要重置的字段（E2 的关键不对称）。
4. **每条回 `agent` 的边都查 `turn_count < MAX_TURNS`**：靠自己的闸门优雅收口到 `finalize`，`recursion_limit` 只兜底（E4——撞框架上限是抛异常，不是正常返回）。
5. **`human_review` 用节点内 `if` 开关**，配合 `compile(checkpointer=...)`；测试/评测默认关（E5）。
6. **messages 兼容性**：dict 消息入图后变 LangChain 消息对象，读 `content` 需兼容两种形态（E1 观察）。

**对 v0.3 草稿的修订建议**：§1/§2 给 `init` 补"为所有 per-query 字段建立首轮默认值"职责（E2 意外发现，本次唯一一处草稿假设与实测的偏差——且偏差方向是"init 比假设的更必要"）。
