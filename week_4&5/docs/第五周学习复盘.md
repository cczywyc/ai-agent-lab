### 周三 — 验证实验

> 本节为 `experiments.py` 实验 E1–E3 的真实结果（设计阶段的假设逐条被验证/修正）。

在写正式代码前，用最小脚本验证关键假设：

**实验一：千问 embedding 链路（E1）。** 全部通过：`text-embedding-v3` 正常返回向量，**维度 1024**（与 `EMBEDDING_DIM` 一致），向量已 **L2 归一化**（实测范数 = 1.000，所以检索时余弦相似度 = 点积，省一次除法）；单批 3 条延迟 **592ms**；批量传 23 条（> 单批上限 10）**自动切 3 批并正确返回 23 个向量**；`embed_query` 返回 `(1024,)`。结论：链路稳定，自动分批可靠，可以放心建库。

**实验二：召回有效性 + chunking 策略对比（E2a/E2b/E2c）。** 用 15 条标注查询（答案确实落在 6 篇笔记里）评测：

- **召回有效性（E2a，按标题切的现有库 100 chunks）**：**hit@1 = 87%（13/15）、hit@3 = 93%、hit@5 = 100%**，top-1 平均相似度 0.764。2 条 hit@1 未命中都是 **README ↔ 第二周复盘**这对同主题文档 top-1 归属互换（内容仍稳定落在 top-3）——不是召回失败，是语义高度重叠时的文档归属抖动。
- **chunking A/B（E2b，按标题切 vs 按字数切）**：同一批源文件，**按标题切 hit@1 = 87%，按字数切（400 字滑窗/80 重叠）只有 53%（8/15）**，top-1 相似度 0.764 vs 0.683。关键点：两种切法 **chunk 数与平均尺寸几乎一样**（100 个@395 字 vs 94 个@389 字），所以这 34 个百分点的差距**纯粹来自"是否保住语义结构"**，而非 chunk 大小。**最终选按标题切**，这条把"检索质量是 RAG 天花板"从口号变成了实测数字。
- **共用向量存储会不会互相污染（E2c）**：不会。`docs` 与 `memory_facts` 是两套独立的 `.npy/.json`（按 namespace 前缀分文件），文件层隔离；内容层用"毒事实"探针验证——塞进 docs 的标记文本不会出现在 memory 库的检索里，反之亦然。结论：**文档库与记忆库共目录、分 namespace 的设计成立**。

**实验三：混合上下文优先级（E3）。** 构造段 3"旧摘要说 chunk 硬上限 = 500"、段 6"本轮检索说 = 1500"的冲突场景，跑两种条件各 3 次：A 带显式"冲突时以最新为准"指令、B 中性指令。**结果两条件都是 3/3 跟随本轮新检索（1500）**。结论：**第三周"json_schema 吃掉 tools"那个优先级坑没有复现**——把当前问题 + 本轮检索放在装配最末段（段 6），叠加 recency 和"带显式来源"的可信度，模型即便没有显式指令也偏向新信息。这反过来印证了六段装配"越靠后越强约束"的排序是对的。

产出：实验结论（见 `experiment_report.json` 的 E1–E3）。**有一个假设被修正**：原以为需要靠 system prompt 显式压制旧摘要，实测中性条件下模型已自然偏向新检索，"以最新为准"指令是冗余加固而非必需。

### 周五 — 记忆实现

实现比计划更进一步：`memory.py` 长成了 `memory/` 包（manager / short_term / long_term / assembler / summarizer / extractor 六个模块）。四个设计**全部落地**：

- **承重墙（优先）**：短期记忆双闸门裁剪（设计一）+ 上下文六段装配（设计四）。E4 验证：注入 5 轮自动 evict 到 K=3；E5 验证装配段随轮次动态变化（见下）。
- **长期记忆（设计二）**：三类都做了——偏好（显式信号触发，E4/E5 各捕获 1 条）、已确认事实（带引用才晋升，E4 抽出 4 条、E5 demo 累计 10 条，可语义召回）、主题计数（E5 末 top5：agent_loop×4 / tool_calling×3 / fallback×3 / rag×2 / chunking×2）。
- **没推迟，周五也做了摘要模块（设计三）**：`summarizer.summarize_low_fidelity` 已实装，只在 evict 时调一次模型，E4/E5 都真实触发并产出一句话摘要（E4："第三周连续失败需重置计数器，成功后自动清零……"）。**唯一真正推迟的是"主题兴趣加权召回"**——主题计数已在记，但加权进 prompt 还留着接口（`long_term.py` 注释明确标了"暂未实装，留接口"），推到第六周。

产出：**v3.0 雏形完整跑通**。E5 四轮真实对话里，六段动态装配清晰可见：

## 二、关键认知变化

> 以下认知点已用周三–周五的真实实现与实验（E1–E7）核对修正，括号内标注对应实验。

### 认知 1：RAG 没有死，而是演化成了基础设施

本周开头我的判断是"业界主流不用 RAG 了"。调研修正了这个偏差：退场的是 2023 那种"切块塞库 top-k 拼接"的朴素 RAG，RAG 本身演化成了 agentic RAG，并从"产品决策"下沉为"agent 按需调用的一个工具"。长上下文窗口让小语料不再需要复杂检索，但检索在大语料、权限控制、成本、可追溯引用场景仍是基线。

教训：**对"X 已经过时了"这类判断要警惕——往往不是消失,而是换了形态。** 跳过一个看似过时的主题前，先确认它的核心机制是否在后续主题里复活（RAG 的检索机制就在记忆系统里原地复活了）。

### 认知 2：我一直在做 agentic RAG，只是没意识到

第二、三周做的搜索 Agent，其结构（迭代循环 + 规则判断是否检索 + 纠错降级 + trace）正是 agentic RAG 的完整骨架。区别只在数据源（网页 vs 文档库）和检索方式（关键词 vs 向量）。

这让本周的 RAG 实现负担大幅降低——**不是从零学 RAG,而是给已会的 agentic 循环接一个新的检索工具。** 印证了一个模式：Agent 的能力是"循环 + 工具"的组合，换工具不换骨架。

**预期基本成立，但有一处必要的额外改动。** `agent.py` 主循环结构确实照搬，retrieve_documents 只是 `TOOL_REGISTRY` 里多注册的一项。需要额外加的是**纠正注入的优先级分支**：v2.0 只有 `should_have_searched`（直答时补联网），v3.0 要先判 `should_have_retrieved`（涉及本项目内容先查本地）、再判联网，且两个纠正各只注入一次、互不重叠（`agent.py:176-208`）。这一改动是"本地优先"语义的必然代价——换工具不换骨架，但**多了一个工具就多了一条"该不该用这个工具"的判定支路**。

### 认知 3：记忆系统的难点不在"存"，在"每轮怎么挑"

**实现后核对：成立，而且比预想更具体。** 短期裁剪、长期召回、摘要,本质都是"挑选"问题——每轮调用前决定什么进 prompt、什么留在 store 待召回、什么直接丢。上下文装配模块就是一个"挑选 + 降维"器。E5 的四轮装配表把这件事变成了肉眼可见的事实：**同一个 Agent，每轮喂进去的段构成都不一样**（从冷启动的 2 段长到满配 5 段），这正是"每轮重新挑选"的直接证据。

**最难权衡的是段 4（长期事实召回）与段 5（最近 K 轮）的预算分配。** 两者都想要更多空间，但角色不同：段 5 是"刚刚发生、不可重建"的对话，段 4 是"可以再召回一次"的事实。设计上给段 5 更大预算（4000 字符 ≈ 总预算一半）、段 4 只给 800——遵循 compaction 原则"优先留不可重建的"。实测 4 轮 demo 还没压到预算上限，这个取舍的真正压力测试要留到更长对话。

### 认知 4：上下文塞太满和工具集臃肿是同一类失败

第二周得出"工具集臃肿是最常见失败模式"，本周发现记忆这里有个对应版本：上下文塞太满同样稀释模型注意力、增加延迟和成本。两者都指向同一个原则——**给模型的东西不是越多越好,而是越对越好。**

### 认知 5：检索质量是可量化的天花板——切法错一步，召回直接腰斩

这是本周唯一一条被**数字钉死**的新认知。E2b 的 A/B 说得很直白：源文件相同、chunk 数和平均尺寸都几乎一样，只把"按标题切"换成"按字数切"，hit@1 就从 **87% 掉到 53%**。检索器、embedding、min_score、top_k 全没动，唯一变量是"切的时候有没有保住语义结构"。

含义有两层：① **检索和生成是两个独立的优化对象**——召回不准时，prompt 写得再好、引用约束再严，也救不回压根没被召回的内容；② **chunking 不是预处理小事，而是 RAG 的第一道天花板**。第二周"工具集臃肿"、第三周"上下文优先级"、本周"切法定召回"，三条其实是同一句话的不同侧面：**给模型的输入，质量比数量更决定上限。**

## 三、踩坑记录

> 详见独立文档《RAG 失败模式记录》。本表只记本周实测真实踩到的坑（数据来自 `experiment_report.json`）。

| # | 现象 | 原因 | 解法 | 归类 |
|---|------|------|------|------|
| 1 | E5 第 2 轮"请记住：以后回答涉及**本地文档**时先列结论"这句纯偏好设置，触发了检索纠正、白白多跑一轮 retrieve | `should_have_retrieved` 的 `local_pointers` 含"本地"，对只是提到"本地文档"的偏好语句也判 True（第三周规则引擎假阳性的记忆版） | 偏好语句先过 `extract_preference` 命中就短路，不再进检索纠正；或收紧本地指向词 | 设计（规则假阳性） |

| 4 | （担心的坑没出现）docs 与 memory_facts 共用 `data/` 目录，怕串库 | — | E2c 实测：按 namespace 前缀分文件，文件层 + 内容层均隔离，毒事实探针不串。**设计成立，无需处理** | 架构（已验证无问题） |

> 另有一条 RAG 侧的路由坑（不属记忆系统但本周实测到）：E6 Case8"MCP 协议是什么？"本应只联网，模型却**先查了本地再联网**（retrieve=True，多 1 轮、26.5s）。根因是 system prompt"本地优先 + 不认识的名词宁可多查"在外部概念上的副作用——把路由 8 用例的严格通过率从 5/22 的 8/8 拉到了今天的 7/8。属"本地优先"策略的过度检索成本，归类架构。

## 四、v2.0 → v3.0 量化对比

> 以下指标全部来自 `experiments.py` 真实跑测（`experiment_report.json`）。

### 功能指标

| 指标 | v2.0（第三周） | v3.0（第五周） |
|------|:---:|:---:|
| 数据源 | 网页（实时） | 网页 + 本地文档库 |
| 检索方式 | 关键词搜索 | 关键词 + 向量相似度 |

## 六、对后续 Roadmap 的思考

### 第六周（LangGraph）

本周手写记忆和装配逻辑后，能更具体地预见 LangGraph 为什么这么设计：

- 本周的 `memory/` 包手动管理短期/长期记忆,对应 LangGraph 的 **checkpointer**（thread 内短期状态）和 **store**（跨 thread 长期记忆）——框架把我手写的东西做成了基础设施。
- 本周的上下文装配是每轮手动拼 messages,LangGraph 用 **状态图的 state** 自动在节点间传递,省掉手动组装。
- **最希望框架接管的是"持久化 + 时序"这一块。** 本周手写里最琐碎、最容易出错的不是装配逻辑，而是：每轮结束后要手动 `save()` 六个 json + 两个向量库、跨会话要手动 `load_snapshot`、还踩了"偏好轮后才写、当轮装配读不到"的时序坑（踩坑 #2）。这种"状态在哪存、什么时候读写"的脏活，正是 checkpointer/store 想消除的。装配的"挑选"逻辑反而是我想自己留着的核心。

延续第三周的判断:手写正好帮我理解框架的设计动机——先体会痛点,再看框架怎么消除痛点。

### 第七周及以后

本周记忆系统给后续 planner-executor / 多 agent 打了两块地基：

- **长期事实 + 语义召回 = 跨任务的状态保持。** E4 验证了"带引用的结论会晋升为可召回事实"，这套机制不依赖对话连续性——换一个任务、换一个 agent，只要查询语义相关就能召回。planner 把子任务结论写进长期事实，executor 在新子任务里召回，天然支撑跨任务/跨 agent 的状态传递。
- **六段装配 = 多 agent 的"上下文路由"原型。** 现在是给一个 agent 每轮挑该看什么；多 agent 场景下，同一套"按预算 + 优先级挑选"逻辑可以扩展成"给哪个 agent 喂哪段上下文"。**挑选问题从单 agent 的时间维度，延伸到多 agent 的空间维度。**

待验证的风险：当前事实抽取依赖 `[doc#section]` 引用格式，而 E6 已暴露格式不稳定（踩坑 #3）——跨任务积累的事实越多，这个契约不严的问题会被放大，得在第七周前先收紧。

## 七、跨模型 / 通用设计原则（本周新增）

> 以下原则均由本周实测支撑（括号内为对应实验）。

- **检索质量是 RAG 的天花板**（E2b 量化）：召回不准,prompt 再好也救不回来——切法从"按标题"换成"按字数"，hit@1 直接 87%→53%。检索和生成是两个独立的优化对象。
- **结构 > 大小**（E2b）：chunk 数和平均尺寸几乎一样时，"保住语义结构"仍带来 34 个百分点的召回差。chunking 的关键变量是结构完整性，不是长度。
- **记忆 = 每轮的挑选问题**（E5）：不是存储问题,而是上下文工程问题——同一 agent 每轮装配的段构成都不同（2 段→5 段），"挑选"是可观测的。
- **共用基础设施、分 namespace**（E2c）：文档检索和记忆召回共用同一套 embedding + 向量存储，按前缀分 namespace 即可隔离，文件层 + 内容层都不串。
- **越靠后越强约束**（E3）：把当前问题 + 本轮检索放装配最末段，叠加 recency，模型即便无显式指令也偏向新信息——装配顺序本身就是一种优先级声明。
- **多一个工具，多一条判定支路**（认知 2）：能力是"循环 + 工具"的组合，但每加一个工具就要配一条"该不该用它"的路由规则，而规则引擎天生会假阳性（踩坑 #1）——这是工具数量的隐藏成本。

## 八、本周最大收获

**记忆系统让 Agent 从"每轮重新开始"进化到"跨轮、跨会话地积累和挑选信息"——而这套"挑选"逻辑本质就是上下文工程，是 Agent 工程化的下一层基础设施。** 而本周最让我踏实的，是第一次把这件事**用数字钉死**：RAG 没死、它是记忆系统的地基（认知 1）只是判断，但 E2b 的"按标题切 87% vs 按字数切 53%"、E5 的"每轮装配段 2→5"、E3 的"3/3 跟随新检索"——是把"检索质量是天花板""记忆是挑选问题""装配顺序即优先级"这些口号变成了可复现的实测。**从"我觉得这样设计对"到"我有数据证明这样设计对"，这是本周比写完代码更大的收获。**
