一、项目定位

这是 roadmap 第 4 周（RAG）和第 5 周（记忆系统）的合并实现，代号 v3.0。它不是从零写的，而是在第 2/3 周的「搜索 Agent」（web_search + fetch_webpage + 受控 Agent Loop）基础上，叠加了两层能力：

  • 第 4 周：给 Agent 接本地知识 → 新增 retrieve_documents 工具 + 整套 RAG 管线（切块 / 嵌入 / 向量库 / 检索）
  • 第 5 周：给 Agent 接记忆 → 新增六段上下文装配 + 短期 / 长期记忆 + 摘要压缩
一句话概括业务逻辑：用户提问 → 装配上下文（带记忆）→ Agent Loop 决策（本地优先、联网兜底、直答）→ 强制引用 → 回答后更新记忆。

三、主流程（最重要）

3.1 入口分发 main.py

main() 用 argparse 分发到 5 种模式：

命令函数作用
--ingest [--dry-run]run_ingest扫描项目内 *.md 重建向量库（dry-run 只切块不嵌入，验证切块对不对）
--query "..."run_single单次查询（默认不带记忆，--with-memory 开启）
--test [--rag-only]run_test跑 8 条测试用例，验证 search/retrieve 决策，输出 test_report.json
--memory-demorun_memory_demo跑 4 轮模拟对话，演示偏好记忆、事实召回、装配
（无参数）run_interactive交互模式，支持 /memory、/reset 调试命令

测试用例设计得很有意思：每条都带 expect_search / expect_retrieve 两个预期标志，断言的是 Agent 做了什么决策（是否调对工具），而不只是回答对不对。Case 4-7 是「本地 RAG 类」，Case 8「MCP 是什么」预期走联网（本地没覆盖），正好测试「本地优先、联网兜底」的边界。

3.2 核心循环 agent.py::run_agent

整个系统的心脏。签名 run_agent(user_message, max_turns, *, memory=None)，返回 (answer, trace)。

第一步：构造初始 messages

  • memory is not None → 调 memory.assemble_context() 走六段装配
  • memory is None → 退化成 v3.0 RAG-only 行为（只有 system + user 两条消息）
关键设计：记忆是可插拔的。测试模式刻意不带记忆，以保证用例可复现。

然后进入 for turn in range(max_turns) 循环（默认 MAX_TURNS=6）

每轮先做上下文长度监控（超 MAX_CONTEXT_CHARS=8000 仅 warning），调模型后按 finish_reason 分三个分支：

分支 1：finish == "stop"（模型想直接回答）

v3.0 最精彩的「纠正注入」逻辑。只有当模型从未碰过任何工具就想直答时才考虑纠正：

any_tool_called = has_searched or has_retrieved
  1. 检索纠正优先：should_have_retrieved(问题) 为真 → 注入 RETRIEVAL_CORRECTION_MESSAGE，逼它先查本地库
  2. 搜索纠正其次：should_have_searched(问题) 为真 → 注入 CORRECTION_MESSAGE，逼它联网

每种纠正最多注入一次，注入后 continue 重进循环。一旦模型已调过工具，就尊重它 stop 的判断——避免「模型已查本地 → 又强行让它联网」的多余轮次。都不触发就正常结束，最后 memory.update_from_turn() 更新记忆。

分支 2：finish == "tool_calls"（模型要调工具）

  • execute_tool() 执行并计时
  • retrieve_documents 成功时额外抽取召回 chunk 元信息存进 trace
  • 记录 has_searched / has_retrieved（即使失败也算「尝试过」，防止反复纠正）
  • 工具结果以 role:"tool" 塞回 messages

连续失败处理有个细节：只有 fetch_webpage 失败才累加 consecutive_errors；retrieve_documents 的 VectorStoreNotReady 是配置问题，不该触发降级。当连续 fetch 失败 ≥ MAX_CONSECUTIVE_ERRORS=2，注入 FALLBACK_MESSAGE 让模型基于已有摘要回答。

分支 3：其它 finish_reason

异常兜底，有内容就返回，否则继续。循环耗尽返回「[达到最大轮次]」。

四、RAG 模块详解（第 4 周）

数据流：ingest（切块）→ embed（向量化）→ store（存） 建库；查询时 retriever（embed query → store 检索 → 阈值过滤）。

rag/ingest.py — 切块策略

核心思路是结构优先、超长再滑窗：

  • _parse_markdown_sections：正则解析标题并维护标题栈，把正文挂到完整标题路径上（如 RAG-1 文档摄入 > Chunking 策略）。特意处理代码块——``` 内部的 # 不会被误判成标题。
  • chunk_markdown_file：① 太短的 section（< CHUNK_MIN_CHARS=50）合并到相邻段；② 超长段用 _split_long_text 按 \n\n 累加，超 CHUNK_TARGET_CHARS=800 切开，相邻块留 target/4 字符重叠，硬上限 CHUNK_MAX_CHARS=1500。
  • 每个 chunk 带 {doc, section, chunk_id, text, path}，text 前拼了 # 标题路径，给嵌入提供上下文，也方便引用。

rag/embed.py — 嵌入

  • 复用 config.client（DashScope OpenAI 兼容接口），模型 text-embedding-v3，1024 维。
  • 自动分批（EMBEDDING_BATCH_SIZE=10），可选批间 sleep 防限流。
  • 关键：返回前做 L2 归一化——检索时余弦相似度直接退化为点积，省一次除法。

rag/store.py — 最小向量库

  • 不上 FAISS/Chroma：学习阶段语料几十~几百条，numpy 足够、零运维、启动即载入。
  • 持久化 = {namespace}_vectors.npy（N×D 矩阵）+ {namespace}_metadata.json，靠下标对齐关联向量和元数据。
  • search：scores = vectors @ query_vector（点积=余弦），np.argpartition 取 top-k 再排序。
  • 多 namespace 设计：docs 存文档，memory_facts 存长期事实——同一个类，不同文件前缀，复用给记忆系统。

rag/retriever.py — 检索门面

  • retrieve() 做阈值过滤：score < RETRIEVE_MIN_SCORE=0.30 丢掉——这就是「只检索不够，还要引用约束/过滤」。返回 dict 列表方便 JSON 序列化进 tool result。
  • get_retriever(namespace)：进程内单例缓存，避免每次查询重新 load 向量库。

RAG 怎么接进 Agent

tools.py::retrieve_documents 是工具实现：延迟 import → 取单例 retriever → is_ready() 检查 → retrieve() → 统一的 error/空结果结构。

「本地优先」不是写死的，而是三处协同：

  1. System Prompt 的 Core Rule: Local First, Web Second + 工具描述里的「Use this FIRST」
  2. 工具定义顺序：retrieve_documents 排第一
  3. checks.should_have_retrieved 作为兜底纠正——模型漏查时强制拉回
强制引用双保险：System Prompt 要求每条事实带 [doc#section]；记忆模块只把带引用的句子当「已确认事实」存进长期记忆。

五、记忆系统详解（第 5 周）

整个记忆系统对 agent.py 只暴露 MemoryManager 两个方法：调模型前 assemble_context、调模型后 update_from_turn。

memory/manager.py — 顶层协调

组合 short / long / summarizer / assembler 四个组件。update_from_turn（每轮回答后）做 6 件事：

  1. extract_topics 抽主题 → long.bump_topics 计数
  2. extract_preference 抽偏好 → long.set_preference
  3. _extract_and_store_facts：从带引用回答抽事实 → 批量 embed → 写长期记忆（json + 向量库）
  4. 本轮包成 ConversationTurn 加进短期记忆
  5. _maybe_evict_and_summarize：需要 evict 时对超窗旧轮做一句话摘要再丢弃
  6. 全部持久化落盘

memory/extractor.py — 零成本写入通道

重要取舍（注释点明是「第三周假阳性教训的记忆版」）：不每轮调模型抽记忆（成本/延迟高、易假阳性），改用纯 regex 规则：

  • extract_topics：12 个项目主题正则，命中计数（实际数据 agent_loop:5, tool_calling:4, fallback:4 就是这么累积的）。
  • extract_preference：只在「记住/请记得/我喜欢/I prefer」等显式信号时捕获。
  • extract_fact_candidates：只抽带 [doc#section] 引用的句子——引用即「已确认」。

memory/short_term.py — 短期记忆

  • ConversationTurn = 一轮完整 user+assistant（不存中间工具结果，那是「可重新获取的原材料」，丢了最划算）。
  • 双闸门 needs_eviction：超 SHORT_TERM_K=3 轮 或 最近 K 轮字符数超 SHORT_TERM_CHAR_BUDGET=4000 → 触发 evict。
  • 支持 snapshot/load，跨会话恢复对话。

memory/long_term.py — 长期记忆（三类）

类型结构写入时机进 prompt 方式
偏好key-value显式信号每轮全量
已确认事实list[Fact] + 向量带引用回答抽取语义召回 top-k
主题兴趣counter每轮规则抽取留接口（暂未加权进 prompt）

事实存两份：json 存元数据，namespace="memory_facts" 向量库存向量，靠下标对齐。recall_facts 用 MEMORY_FACTS_TOP_K=3 + MEMORY_FACTS_MIN_SCORE=0.30 做语义召回。

这就是「长期记忆 ≠ 聊天历史」的关键：不是把历史塞回去，而是按当前问题语义检索相关事实。

memory/summarizer.py — 分层压缩

被 evict 的旧轮做两级处理：

  1. 高保真：extract_promoted_facts 把带引用的事实晋升长期记忆（不丢失）
  2. 低保真：summarize_low_fidelity 对剩下的过程性对话调一次模型，压成 ≤80 字摘要
成本控制：事实抽取走规则（零成本），摘要只对 evict 旧轮调一次模型，不每轮调。累积摘要上限 1200 字符防膨胀。

memory/assembler.py — ★ 上下文装配

固定 6 段顺序，越稳定/强约束越靠前：

段内容进出规则预算(字符)
1System prompt永远在固定
2用户偏好永远在（很小）200
3历史摘要有则在，超预算保留最近400
4召回的长期事实语义 top-k，按问题召回800
5最近 K 轮对话双闸门裁剪4000
6当前问题永远在6000

裁剪优先级：超预算时段 5 先裁 → 段 3/4 次之 → 段 1/2/6 不可裁。段 5 二次裁剪保留 user 全文（短且不可重生），只截断旧轮 assistant 到 200 字。装配完产出 AssemblyReport（在场/被裁/总字符/召回事实数）写进日志做可观测。

六、可观测性 trace.py

三层结构 AgentTrace → TurnTrace → ToolCallTrace，记录每轮决策、每次工具调用（含耗时、成功与否、retrieve 召回的 chunk）、纠正/降级是否触发。summary() 输出像 [2轮 | 1340ms] retrieve_documents(✓) [检索纠正] 的人类可读串，to_dict/to_json 进测试报告。这呼应了第 10 周「评测与可观测性」的铺垫。

七、值得注意的设计亮点 / 取舍

  • 记忆可插拔：测试无记忆保证可复现，交互/demo 带记忆，Agent Loop 统一处理两种情况。
  • 本地优先三重保险（Prompt + 工具顺序 + 规则纠正），且检索纠正优先于联网纠正。
  • 写入双通道：规则抽取（主题/事实，零成本）+ 模型显式（偏好/摘要）——刻意避免「每轮让模型抽记忆」的延迟与假阳性。
  • 引用即事实：只有带 [doc#section] 的回答才晋升长期记忆，把「引用约束」和「记忆质量」绑在一起。
  • 降级计数只算 fetch_webpage：RAG 配置类错误不污染 fallback 逻辑，边界处理到位。

│  MemoryManager.update_from_turn(msg, answer, trace)                           │
│    1. extract_topics  → LongTerm.bump_topics            (规则, 零成本)         │
│    2. extract_preference → LongTerm.set_preference      (仅"记住/我喜欢"信号)  │
│    3. extract_fact_candidates(带[doc#section]) → embed_texts → add_fact        │
│                                       (json + memory_facts 向量库)             │
│    4. ShortTerm.add_turn(本轮)                                                │
│    5. needs_eviction? → Summarizer.summarize_low_fidelity (LLM一次) → evict   │
│    6. save: prefs/facts/topics/向量库 + summary.json + short_term.json         │
└──────────────────────────────────────────────────────────────────────────────┘
                                       │
用户/CLI ◀──────── (answer, trace) ────┘

看图要抓的三个关键点

  1. Embedding API 被调用三次、目的各不同：装配段4（召回事实，查 memory_facts）、Loop 内 retrieve_documents（查 docs 文档库）、更新阶段事实入库——同一接口服务三条数据流。
  2. LLM 在一次查询里可能被调多次：Agent Loop 每轮一次（含纠正注入的额外轮次），结尾若触发 evict 还会为摘要单独调一次——但摘要只对被裁旧轮调，不每轮调，这是成本控制的核心。
  3. 记忆的「读」和「写」分居模型调用前后：装配（读）在前，update_from_turn（写+落盘）在后，中间夹 Agent Loop——这就是「上下文工程」相对「模型自己记住」的本质：全靠这两端的工程动作。
