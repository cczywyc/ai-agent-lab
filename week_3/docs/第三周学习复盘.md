### 周二 — 设计方案制定（概念日第二天）

不写代码，但产出可以直接指导编码的三份设计草稿：

**任务一：System Prompt 重写。** 核心设计决策是"翻转默认方向"——旧 prompt 说"可以使用搜索工具"（许可语气），新 prompt 说"必须先搜索，以下情况例外"（默认搜索）。同时写了 3 个 few-shot examples，分别覆盖事实查询（搜索）、技术概念（搜索）、创意写作（不搜索）。完成后对第二周 7 个测试用例做了逐案预期分析。

**任务二：Structured Output Schema 设计。** 分析了三种架构方案——纯 tool calling、纯 response_format、混合方案。结论是第三周采用混合方案（方案 C）：主流程用 tool calling，可选的格式化步骤用 response_format。同时设计了 AgentTrace 数据结构，让 Agent 的每一步决策可追踪。

**任务三：Agent Loop 升级方案。** 画了新旧 Loop 对比流程图，设计了三个检查机制的具体参数：`should_have_searched()` 规则引擎（5 条排除规则 + 6 条搜索规则）、`MAX_CONSECUTIVE_ERRORS = 2`、上下文监控（简化版只警告）。

产出：三份设计文档，共约 8000 字。

### 周三 — 跑通验证实验

在写正式升级代码之前，先用最小脚本验证关键技术假设。这是本周最重要的一天之一——因为实验结果直接推翻了一个设计假设。

**实验一：json_object 模式。** 4 个子测试。核心发现：千问强制要求 messages 中包含 "json" 关键词，否则 400 报错。这是文档里没有明确说的限制。

**实验二：json_schema 模式。** 4 个子测试。好消息：`strict: True` 非常可靠，enum 约束严格生效，即使 prompt 试图诱导也不会违反。

**实验三：tools + response_format 互斥性。** 这是最关键的实验。7 个子测试，得出结论：

| 组合 | 结果 |
|------|------|
| tools + json_object | 400 报错（"json" 关键词限制） |
| tools + json_schema | **不报错，但 json_schema 吃掉了 tools！模型不调用工具，直接返回 JSON** |

**这个发现直接推翻了周二的方案 C**（同时传 tools + response_format）。修正方案：主流程只传 tools；格式化步骤用**单独请求** + json_schema。绝不在同一个请求中同时使用。

这个经验印证了一个重要原则：**API 的实际行为必须用实验验证，不能只看文档。** 文档说两者都支持，但没有说同时使用时的优先级规则。

产出：实验结论文档 + 修正后的技术决策。

### 周五 — 分析测试报告 + 修复问题

仔细分析 test_report.json，发现了一个重要问题：

**纠正机制假阳性率 57%。** 10 个 case 中 7 个触发了纠正，但其中 4 个是假阳性——模型在 Turn 1 已经搜索过了，Turn 2 准备回答时被误判为"跳过工具"。

根因分析：`should_have_searched()` 只看用户原始问题，不看对话历史中是否已经搜索过。修复方案是在纠正检查中增加前置条件：

```python
already_searched = any(
    tc.tool_name == "web_search" and tc.result_success
    for t in trace.turns
    for tc in t.tool_calls
)

if (not correction_injected
    and not already_searched  # ← 新增：已搜索过就不纠正
    and should_have_searched(user_message)):
```

修复后预期：纠正触发率从 70% 降到 20%，平均轮次从 3.5 降到 2.5，平均耗时从 25.7s 降到 15-18s。

还发现降级机制（fallback）0/10 触发——因为 Case 3 在 v2 中模型直接用 snippet 回答了，根本没调 fetch_webpage。这是好事（prompt 层面解决了问题），但也意味着降级机制需要额外的测试用例来验证。

产出：测试报告分析文档 + 假阳性修复方案。

## 二、关键认知变化

### 认知 1：Prompt 优化和代码检查是互补关系，不是替代关系

这是本周最核心的认知。第二周我把 Agent 的问题归因为"prompt 不好"，以为只要优化 prompt 就够了。第三周的实践证明：

- **Prompt 解决了 75% 的问题**：8 个需要搜索的 case 中，6 个在 Turn 1 就主动调用了工具。新版 prompt 的"默认搜索"策略确实有效。
- **但 25% 的 case 仍然需要代码兜底**：Case 5（RAG）和 Case 7（xyzabc123）模型就是跳过了工具，prompt 再怎么写都拦不住——因为模型"非常确信自己知道答案"。

这两层是"预防 + 检测"的关系。Prompt 是预防层，降低问题发生的概率；代码检查是检测层，捕获漏网之鱼。两层必须同时存在。

类比：安全领域的"纵深防御"——防火墙（prompt）减少攻击面，入侵检测（代码检查）发现漏过的威胁。

### 认知 2：API 的实际行为必须用实验验证

周三的 tools + response_format 互斥性实验是本周最有价值的发现。文档说千问同时支持两者，但没有说同时使用时的行为。如果我跳过实验直接按周二的方案 C 写代码，会在周四浪费大量时间 debug 一个"工具怎么都不被调用"的问题。

这个教训适用于所有 LLM API 开发：**文档告诉你"支持什么"，但不会告诉你"组合使用时的优先级"。** 唯一可靠的方法是写最小脚本验证。

### 认知 3：假阳性比漏检更容易被忽视

第二周我只关注"模型跳过工具"（漏检问题），于是设计了纠正机制。但第三周测试报告显示，57% 的纠正是假阳性——模型已经搜索过了，纠正反而导致重复搜索、浪费轮次、增加延迟。

假阳性的隐蔽性在于：**它不导致功能错误，只导致效率下降**。如果我只看"通过率 100%"就满意了，不分析 trace 细节，就不会发现这个问题。这说明**评测不能只看通过率，还要看代价**（轮次、延迟、额外 API 调用）。

### 认知 4：Agent 控制循环的复杂度呈指数增长

第二周的 Agent Loop 只有两个分支（stop / tool_calls），逻辑清晰。第三周加了三个检查机制后，分支变成了：

```
stop → 已搜索过？ → 是 → 正常返回
                    → 否 → should_have_searched? → 是 → 已纠正过？ → ...
                                                  → 否 → 正常返回
tool_calls → 执行 → 失败？ → 连续N次？ → 已降级过？ → ...
```

每新增一个检查机制，分支数接近翻倍。这让我理解了为什么第六周要学 LangGraph——手写的 while 循环在 3-4 个检查机制后就变得难以维护。状态图是管理这种复杂度的更好工具。

### 认知 5：结构化 Trace 的价值远超预期

第二周靠 print 调试，效率很低——看完日志还要人脑重建"模型在第几轮做了什么决策"。第三周有了 AgentTrace 后：

- **调试时间缩短 80%**：一个 JSON 就能看到完整决策链，不用翻日志
- **假阳性问题是 trace 暴露的**：如果没有 `correction_triggered` 字段，我根本不知道 70% 的 case 被纠正了
- **评测自动化成为可能**：可以程序化地检查 `searched == expect_search`，不用人工判断

Trace 不是"锦上添花"的功能，它是 Agent 工程化的基础设施。

### 模块依赖关系

```
main.py
  ├── agent.py
  │     ├── config.py    （读取配置）
  │     ├── tools.py     （执行工具）
```

## 六、对后续 Roadmap 的思考

### 第四周（RAG）

搜索 Agent 的 `web_search → fetch_webpage → 基于内容回答` 模式和 RAG 的 `检索 → 获取文档 → 基于文档回答` 在逻辑上几乎同构。区别在于：

| 维度 | 搜索 Agent | RAG |
|------|-----------|-----|
| 检索方式 | 关键词搜索（DuckDuckGo） | 向量相似度搜索（embedding） |
| 数据源 | 互联网（实时） | 预处理的文档库（离线） |
| 内容获取 | HTTP 抓取（不稳定） | 本地读取（稳定） |
| 引用约束 | snippet + URL | chunk 位置 + 文档名 |

可以复用的：Agent Loop 核心框架、trace 记录机制、should_have_searched 规则引擎（改为 should_have_retrieved）、错误处理模式。

需要新增的：embedding 生成和存储、向量检索工具、chunk 管理、rerank 逻辑。

### 第五周（记忆系统）

本周的上下文监控（检查机制 3）是简化版——只做警告不实际裁剪。第五周需要实现真正的上下文管理：

- 短期记忆：对话历史裁剪策略（保留 system + 最近 K 轮）
- 长期记忆：用户偏好（如"总是用中文回答"）、已确认的事实
- 上下文装配：每轮调用前动态组装 messages，而不是简单追加

### 第六周（LangGraph）

本周最大的感受是手写 while 循环在 3 个检查机制后就变得复杂。agent.py 的主循环有 150+ 行，嵌套层级达到 4-5 层。LangGraph 的状态图可以把"纠正""降级""正常回答"拆成独立节点，每个节点的逻辑更简单，整体流程更可视化。

具体来说，本周的 Agent Loop 可以拆成这些节点：
- `route`: 判断 finish_reason
- `execute_tools`: 执行工具调用
- `check_should_search`: 规则检查
- `inject_correction`: 注入纠正
- `check_consecutive_errors`: 失败计数
- `inject_fallback`: 注入降级
- `return_answer`: 返回回答

每个节点不超过 20 行代码，但组合起来实现了完整的控制逻辑。

## 七、Anthropic vs OpenAI/千问的新发现

### Structured Output 差异

| 维度 | OpenAI | 千问（OpenAI 兼容） | 影响 |
|------|--------|------|------|
| json_object 模式 | 只要设 response_format 即可 | **强制要求 messages 含 "json" 关键词** | 需在 prompt 中显式提及 JSON |
| json_schema strict | 100% schema 遵循 | 同样严格 ✅ | 可靠 |
| tools + response_format | 行为未知（未测试） | **json_schema 覆盖 tools，模型不调工具** | 绝不能同时使用 |
| 思考模式 + structured output | N/A | 不支持同时使用 | 需二选一 |

### 通用的、与 LLM 无关的设计原则

本周新增的跨模型通用认知（加上第二周的）：

- **控制循环的"纠正-降级"双机制**：适用于任何 Agent 框架
- **规则引擎做第一道检测**：成本为零，准确率 75-85%，足以覆盖大多数场景
- **Trace 是 Agent 工程的基础设施**：不是可选的 debug 工具
- **API 组合行为必须实验验证**：文档不会告诉你互斥性和优先级
- **评测不能只看通过率，还要看代价**：轮次、延迟、API 调用数都是成本
