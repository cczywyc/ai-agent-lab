## 一、Agent Loop 的本质：一个带安全网的决策循环

### 1.1 最简模型

把所有修饰去掉后，Agent Loop 的本质就是一句话：

**反复问模型"接下来做什么"，直到它说"我做完了"。**

```
loop:
    decision = ask_model("接下来做什么？")
    if decision == "done":
        return answer
    else:
        execute(decision)
        goto loop
```

这跟人类的工作方式完全一样：接到一个任务，想一下需要什么信息，去查，拿到信息后判断够不够，不够就再查，够了就输出结果。Agent Loop 就是把这个过程代码化。

### 1.2 但模型不是可靠的决策者

上面的最简模型有一个致命假设：**模型的 decision 总是正确的。** 第二周的实测彻底否定了这个假设：

| 失败类型 | 表现 | 频率 |
|---------|------|------|
| 应该做但不做 | 模型跳过搜索直接回答（Case 1/5/7） | 43% |
| 做了但做不成 | 工具连续失败，模型不知道放弃（Case 3） | 14% |
| 做完但不停 | 模型可能陷入搜索循环 | 潜在风险 |

所以真实的 Agent Loop 不是"信任模型的循环"，而是**"带安全网的决策循环"**——在模型每次决策后、执行前和执行后，插入一系列检查和保护措施。

### 1.3 我的理解：Agent Loop = 决策循环 + 三张安全网

```
           ┌─────────────────────────────┐
           │       模型决策循环            │
           │  (model makes decisions)     │
           └──────────┬──────────────────┘
                      │
        ┌─────────────┼─────────────────┐
        │             │                 │
   ┌────┴────┐  ┌────┴────┐  ┌────────┴────────┐
   │ 安全网 1 │  │ 安全网 2 │  │    安全网 3      │
   │ 决策审查 │  │ 执行保护 │  │    资源控制      │
   │         │  │         │  │                  │
   │ 该做没做？│  │ 做了失败？│  │ 轮次/时间/上下文 │
   │ → 纠正   │  │ → 降级   │  │ → 强制终止       │
   └─────────┘  └─────────┘  └──────────────────┘
```

- **安全网 1（决策审查）**：模型说"我做完了"，但你不信——检查它是否跳过了该做的事
- **安全网 2（执行保护）**：模型做了正确的决策，但执行失败——检测连续失败并降级
- **安全网 3（资源控制）**：不管决策对不对，到达上限就强制停止——max_turns、context 长度

三张安全网独立运作，解决不同层面的问题。任何一张缺失都会留下盲区。

## 二、三个控制点的设计取舍

### 2.1 控制点 1：决策审查（检查机制 1）

#### 核心设计决策：检查放在 stop 分支，而不是每轮都检查

我选择只在 `finish_reason == "stop"` 时检查（模型打算直接回答时），而不是在每轮开头都检查。原因：

| 方案 | 优点 | 缺点 |
|------|------|------|
| **每轮开头检查** | 能捕获"模型在工具调用之间突然决定不搜索了" | 每轮多一次规则匹配；可能在模型正在使用工具的过程中误干预 |
| **只在 stop 时检查** ✅ | 只在"关键时刻"（模型准备结束时）才介入；不干预正常的工具使用流程 | 不能捕获中途放弃的情况 |

选择后者的理由：搜索 Agent 的典型流程是"搜索 → 回答"，模型在搜索和回答之间不会有"突然放弃"的情况。唯一需要检查的就是 stop 分支——模型是不是在该搜索的时候直接回答了。

#### 核心设计决策：纠正最多 1 次

为什么不纠正 2 次或无限次？

```
纠正 0 次：模型跳过工具 → 用户拿到未验证的答案    ← 太危险
纠正 1 次：模型跳过工具 → 纠正 → 大概率搜索了    ← 平衡点 ✅
纠正 2 次：纠正后还不搜索 → 再纠正 → 有死循环风险 ← 太激进
纠正 N 次：几乎一定会死循环                        ← 不可接受
```

实测数据验证了这个选择：第二周 Case 5 和 7 在纠正 1 次后都成功搜索了。没有出现"纠正后仍然不搜索"的情况。如果 1 次纠正不够，说明 prompt 有更深层的问题，继续纠正也没用——应该去改 prompt 而不是加纠正次数。

#### 周五发现的假阳性问题及修复

假阳性根因：检查逻辑不看对话历史。模型在 Turn 1 搜索后 Turn 2 回答，此时 `should_have_searched()` 仍然返回 True（因为用户问题没变），触发不必要的纠正。

修复：在纠正检查前加一个"是否已经搜索过"的前置条件。

这个教训的设计启示：**任何检查机制都需要考虑上下文状态，不能只基于静态输入做判断。** `should_have_searched(user_message)` 是无状态函数——它不知道对话进行到了哪里。要做有状态的判断，必须引入 trace 信息。

### 2.2 控制点 2：执行保护（检查机制 2）

#### 核心设计决策：降级阈值为什么是 2

| 阈值 | 在 max_turns=6 时的效果 | 适用场景 |
|------|------------------------|---------|
| 1 | 1 次失败就降级，剩 5 轮给回答 | 对延迟极敏感的实时场景 |
| **2** | 2 次失败后降级，剩 3-4 轮给回答 | **通用搜索场景（平衡点）** ✅ |
| 3 | 3 次失败后降级，只剩 2-3 轮 | 需要尽可能获取完整信息的场景 |

选 2 的推理过程：

- 第二周 Case 3 连续 5 次 fetch_webpage 失败，全部轮次浪费。阈值 2 能在第 3 轮就触发降级，保留 3 轮给模型基于 snippet 回答
- 阈值 1 太敏感——一次 timeout 或偶发 403 就放弃深度获取，会降低回答质量
- 阈值 3 在 max_turns=6 时只给降级回答留 2-3 轮，如果降级回答本身需要再搜一次（补充关键词），空间就不够了

#### 核心设计决策：降级指令放在 user 角色而不是 system 角色

千问 API 只支持一条 system message 在开头，不能中途插入新的 system message。所以降级指令以 `role: "user"` 注入。

但这引出一个微妙问题：模型可能把降级指令理解为"用户在催我"而不是"系统在纠正我"。解决方式是在指令前面加 `[System Notice]` 前缀，让模型区分来源。

实测中这个方式是有效的——但这是一个脆弱的约定，模型不一定总是尊重。更健壮的方案是用专门的"系统消息注入机制"（如果 API 支持中途插入 system message 的话）。

#### 连续失败的计数策略：成功即重置

```python
if is_error:
    consecutive_errors += 1
else:
    consecutive_errors = 0  # 任何一次成功就重置
```

替代方案是"滑动窗口"（最近 N 次中失败超过 M 次就降级），更精细但也更复杂。对搜索 Agent 来说，"连续失败"已经足够——因为搜索 Agent 的工具调用是线性的（搜索 → 抓取 → 搜索 → 抓取），不存在"成功和失败交替出现"的常见场景。

### 2.3 控制点 3：资源控制（检查机制 3）

#### 核心设计决策：第三周只做监控不做裁剪

```python
if context_chars > MAX_CONTEXT_CHARS:
    logger.warning(f"Context size ({context_chars}) exceeds threshold")
    # 不实际裁剪
```

为什么不直接裁剪？

1. **搜索 Agent 的单次对话不太会溢出**：max_turns=6，每轮 200-1500 tokens，最多 9000 tokens，远低于千问的 128K 窗口
2. **裁剪逻辑需要仔细设计**：裁掉哪些消息？工具结果能裁吗？纠正指令能裁吗？这些决策在没有充分测试数据之前不应该仓促做
3. **第五周会专门学上下文管理**：到时候有记忆系统的完整框架，裁剪策略会更合理

这是一个"知道什么时候不该做"的设计决策。过早优化不如先监控、积累数据、再决策。

#### max_turns 从 5 调到 6 的考量

v1 的 max_turns=5。v2 改为 6，多出的 1 轮是为降级机制预留的：

```
worst case: 搜索(1) → 抓取失败(2) → 抓取失败(3) → 降级注入(3) → 降级回答(4)
留 2 轮 buffer: 如果降级回答不够好，还有 2 轮供模型补充搜索
```

但不能无限加大 max_turns——更多的轮次意味着更高的延迟和成本。6 是一个"够用但不浪费"的值。

## 三、从测试数据中学到的

### 3.1 假阳性纠正的教训

测试报告中最大的意外不是功能失败，而是**效率问题**——7/10 的 case 被不必要地纠正了。这教会我三件事：

**第一，无状态检查在有状态系统中是危险的。** `should_have_searched()` 是纯函数，只看输入不看上下文。但 Agent Loop 是有状态的——"模型是否已经搜索过"是一个随对话进展变化的状态。用无状态检查做有状态判断，假阳性是必然的。

**第二，"通过率 100%"可能掩盖效率问题。** 如果我只看通过率就收工了，假阳性问题就永远不会被发现。Trace 数据让我看到了"正确但低效"的行为模式。

**第三，修复假阳性比修复漏检更需要克制。** 修复漏检很直觉——加更多规则、更强的约束。修复假阳性则需要**放松**约束——让检查机制在某些条件下"放行"。这违反了"安全第一"的直觉，需要基于数据而不是感觉来决策。

### 3.2 降级机制未触发说明了什么

0/10 的降级触发率看起来像是"降级机制没用"。但实际情况是：

- v1 的 Case 3 因为连续 403 无输出，是**真正需要降级的场景**
- v2 的 Case 3 在 prompt 层面就解决了（模型不再调 fetch_webpage，直接用 snippet 回答）
- 这意味着**预防层（prompt）解决了问题，治疗层（降级）没有被需要**

这是好事。最好的安全网是永远不需要接住人的安全网——因为走钢丝的人技术足够好。但安全网仍然必须存在，因为你不能保证每次都不失足。

类比软件工程：数据库的备份恢复机制可能一年用不上一次，但你不会因此取消备份。

### 3.3 Prompt 和代码两层防御的协作关系

从 10 个 case 的 trace 数据中，可以清楚地看到两层防御各自的贡献：

| 防御层 | 作用 | 覆盖的 case | 覆盖率 |
|--------|------|------------|--------|
| **Prompt 层** | 引导模型在 Turn 1 就做出正确决策 | Case 1/2/3/4/6/8/9/10 | 80% |
| **代码层（纠正）** | 在 Prompt 不够时兜底 | Case 5/7 | 20% |
| **代码层（降级）** | 在工具失败时兜底 | 本次未触发 | 0%（但必须存在） |

80/20 法则在这里非常明显。如果只有 Prompt 层，通过率约 80%；加上代码层才到 100%。但代码层不能替代 Prompt 层——如果 Prompt 层只有 50%（v1 的水平），代码层需要兜底 50% 的 case，纠正和降级的负担会大得多，延迟也会高得多。

**最佳策略是：让 Prompt 层尽可能强，让代码层尽可能不需要触发。** 代码层的价值不在于"经常使用"，而在于"关键时刻能兜住"。

## 四、Agent Loop 的设计模式总结

经过两周实践，我总结出 Agent Loop 设计的几个通用模式：

### 模式 1：分支-检查-行动

```
每轮循环 {
    模型决策 → 判断 finish_reason
    
    分支 "stop" {
        检查：该做没做？（决策审查）
        行动：纠正 or 返回
    }
    
    分支 "tool_calls" {
        行动：执行工具
        检查：连续失败？（执行保护）
        行动：降级 or 继续
    }
    
    全局检查：资源超限？（资源控制）
}
```

关键点：每个分支都有"检查 → 行动"的结构，检查不通过时有明确的恢复行动。

### 模式 2：检查机制的三要素

每个检查机制都需要定义三个要素：

| 要素 | 检查机制 1（纠正） | 检查机制 2（降级） | 检查机制 3（资源） |
|------|-------------------|-------------------|-------------------|
| **触发条件** | stop + 未搜索 + 规则匹配 | 连续失败 ≥ 2 | 上下文 > 阈值 |
| **恢复行动** | 注入纠正消息 + continue | 注入降级消息 | 裁剪 messages |
| **防护限制** | 最多 1 次 | 最多 1 次 | 每轮检查 |

"防护限制"是防止检查机制本身陷入循环的关键。没有防护限制的检查机制是危险的——它可能把 Agent 拖入"检查→恢复→再检查→再恢复"的无限循环。

### 模式 3：有状态检查 vs 无状态检查

| 类型 | 例子 | 优点 | 缺点 |
|------|------|------|------|
| **无状态检查** | should_have_searched(user_msg) | 简单、可单独测试、无副作用 | 不考虑对话进展，可能假阳性 |
| **有状态检查** | already_searched（看 trace） | 准确、考虑上下文 | 依赖外部状态、更难测试 |

最佳实践：**先用无状态检查做第一道过滤，再用有状态检查做精确判断。**

```python
# 第一道：无状态过滤（快，但可能假阳性）
if should_have_searched(user_message):
    # 第二道：有状态确认（准确）
    if not already_searched_in_trace(trace):
        inject_correction()
```

这就是修复假阳性后的逻辑结构。两道检查的组合比任何单独一道都好。

### 模式 4：Trace 驱动的迭代优化

```
v1: 写代码 → 跑 → 看 print 日志 → 凭感觉改
v2: 写代码 → 跑 → 分析 trace → 用数据改
```

Trace 改变了优化方式。没有 trace 时，发现问题靠人工观察，改进靠直觉。有了 trace，发现问题靠自动分析（如统计纠正触发率），改进靠数据（如假阳性率从 57% 降到 0%）。

**Agent 系统的可改进性取决于它的可观测性。** 不能观测的系统就不能被可靠地改进。

## 五、未解决的问题（留给后续周次）

### Q1：规则引擎的上限在哪里？

`should_have_searched()` 是基于关键词和正则的启发式规则。它在 28 条测试上达到了 100% 准确率，但规则引擎的根本局限是：**它无法理解语义。**

例如"帮我解释一下这段代码是什么意思"——包含"什么意思"（规则匹配→搜索），但实际上是代码解释（不该搜索）。这种语义冲突只能靠 LLM 来判断，不能靠关键词匹配。

后续方向：第十周（评测）时用更大的测试集（50-100 条）压测规则引擎的边界，找到必须升级到 LLM 判断的阈值。

### Q2：降级后的回答质量如何保证？

当前降级机制告诉模型"用已有的 snippet 回答"，但 snippet 质量取决于搜索引擎的返回——有些 snippet 信息量很低（只有一两句话）。如何判断"snippet 够不够回答这个问题"？不够时怎么办？

后续方向：可能需要在降级指令中加入"如果摘要信息不足以回答，请诚实告知用户而不是编造内容"的约束。

### Q3：多 Agent 场景下 Loop 怎么嵌套？

第八周会学多 Agent。如果一个 orchestrator agent 调用子 agent，子 agent 自己也有 loop，那纠正和降级机制在哪一层生效？子 agent 的降级是否应该通知上层 agent？

这些跨 agent 的控制流问题，单 agent 场景下无法回答。

### Q4：如何做到真正的"在线学习"？

当前的纠正和降级都是预设策略。理想情况下，Agent 应该从每次交互中学习：如果某类问题总是被纠正，就自动调整"何时该搜索"的判断标准；如果某个域名总是 403，就自动加入黑名单。

后续方向：第五周的记忆系统 + 第十周的评测，可能是实现这种在线学习的基础。
