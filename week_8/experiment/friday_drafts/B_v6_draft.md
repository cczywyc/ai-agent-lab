在技术写作多 Agent 系统的演进过程中，检索模块的架构设计经历了从“单 Agent 工具调用”到“多 Agent 角色拆分”的重要转变。本文基于第四周至第七周的落地实现与设计规划，梳理基础 RAG 检索模块的核心设计，并探讨后续多 Agent 架构中 Researcher 检索职责的演进逻辑与架构思考。

### 一、 第四五周落地实现：Agentic RAG 与“换工具不换骨架”

在第四、五周的开发中，RAG 并非作为独立的静态检索管道存在，而是以 Agentic RAG 的形态实现 `[第四周第五周学习复盘#第五周学习复盘：Agentic RAG 与记忆系统 > 一、本周时间线 > 周二 — 设计方案 + RAG 概念厘清#3]`。

1. **检索作为 Agent 的工具**：检索工具 `retrieve_documents` 被注册进 Agent 的 `TOOL_REGISTRY` 中，由大模型在运行时自主决定何时调用 `[第四周第五周学习复盘#第五周学习复盘：Agentic RAG 与记忆系统 > 一、本周时间线 > 周二 — 设计方案 + RAG 概念厘清#3]`。
2. **“换工具不换骨架”的设计思路**：系统直接复用了第二、三周搜索 Agent 的核心骨架（包括迭代循环、规则判断、纠错降级与 trace 机制），仅将底层的检索方式从基于关键词的 `web_search` 替换为基于向量相似度的 `retrieve_documents` `[第四周第五周学习复盘#第五周学习复盘：Agentic RAG 与记忆系统 > 二、关键认知变化 > 认知 2#9]`。
3. **低侵入性的代码改动**：在这一设计下，`agent.py` 的主循环逻辑基本保持不变，新增检索能力仅需在 `tools.py` 中完成工具注册 `[第四周第五周学习复盘#第五周学习复盘：Agentic RAG 与记忆系统 > 一、本周时间线 > 周四 — RAG 实现#5]`，并配套构建了 `rag/` 包的相关底层模块以支撑向量化与检索逻辑 `[第四周第五周学习复盘#第五周学习复盘：Agentic RAG 与记忆系统 > 五、本周代码架构]`。

### 二、 第七周架构规划：从“角色节点”向“真·多 Agent”的演进思考

随着系统复杂度的提升，第七周在复盘与后续规划中，对多 Agent 架构下的检索职责提出了新的思考。需要明确界定的是，以下内容为第七周对后续架构的规划与展望，而非已落地的实现事实。

1. **角色拆分原型的跑通**：第七周已实现了 planner、executor、critic 的“一张图里的角色拆分原型”，其中 researcher 的职责近似于 executor，reviewer 近似于 critic，该雏形已在 v5.0 版本中跑通 `[第七周学习复盘#七、对第八周的思考（多 Agent）#12]`。
2. **核心架构疑问与权衡**：在规划真正的多 Agent 架构（即独立 Agent 结合 handoff 机制，各自拥有独立的 messages 与上下文）时，核心要解决的问题是：相比第七周的“角色节点”模式，真·多 Agent 架构额外付出的协调成本、上下文隔离成本以及 handoff 协议成本，究竟能换来哪些实打实的能力提升 `[第七周学习复盘#七、对第八周的思考（多 Agent）#12]`。这一思考直接决定了后续 Researcher 检索职责的改造方向。

### 三、 两种检索设计的核心差异与演进逻辑

对比第四五周的落地实现与第七周的设计规划，检索模块的定位发生了本质变化：

1. **设计 A（v3.0 / 第四五周已落地）：检索作为单 Agent 的工具**。在此模式下，`retrieve_documents` 作为 `TOOL_REGISTRY` 中的一个工具，由单一模型在同一个 Agent Loop 中自主决策调用 `[第四周第五周学习复盘#五、本周代码架构#18]`。其优势在于架构简单、复用性高，通过“换工具不换骨架”快速实现了从 Web 搜索到本地 RAG 的切换 `[第四周第五周学习复盘#一、本周时间线 > 周二 — 设计方案 + RAG 概念厘清#3]`。
2. **设计 B（v5.0 / 第七周规划）：检索下沉为 Executor 的核心职责**。在 planner-executor-critic 的外循环规划中，检索不再是单一 Agent 的可选工具，而是下沉为 executor 角色的核心职责 `[规划器与执行器职责边界#1. 三个角色定义]`。Executor 负责接收单个子任务，执行检索、工具调用与总结，最终产出带引用的单步结果 `[第七周设计草稿#决策定稿（v0.6）]`。这种设计将检索与具体的子任务执行深度绑定，提升了多 Agent 协作下的任务专注度与结果可追溯性。

### 结论

从第四五周的 Agentic RAG 到第七周规划的多 Agent 角色拆分，检索模块经历了从“通用工具调用”到“专职角色执行”的演进。前期的“换工具不换骨架”策略以极低的成本快速验证了 RAG 能力，而后续向 Executor 职责下沉的规划，则旨在通过上下文隔离与职责细分，解决复杂任务下的检索精准度与多 Agent 协作效率问题。这一演进逻辑清晰地反映了系统从单一能力验证向复杂工程架构迈进的过程。
