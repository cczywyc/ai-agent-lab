### 一、Tool Calling 的实现原理

**核心机制：模型不执行工具，只决定调用什么。** 这是整个 Tool Calling 架构的基石。LLM 返回 `tool_calls` 后，所有执行权在开发者代码手中，参数校验、权限检查、危险操作确认都可以在执行前拦截。[第二周学习复盘#二、关键认知变化 > 认知 2：模型不执行工具，只决定调用什么]

完整的 Tool Calling 流程包含以下关键概念：

1. **请求-响应结构**：模型返回 `finish_reason`、`tool_calls`、`tool_call_id` 等关键字段，构成完整的调用信号链。[第二周学习复盘#周三 — 跑通官方示例（正式写代码）]
2. **Agent Loop（while 循环 + 工具注册表模式）**：这是所有 Agent 的核心骨架——循环检查模型是否需要调用工具，调用后将结果回传模型，直到模型给出最终回答。[第二周学习复盘#周三 — 跑通官方示例（正式写代码）]
3. **工具三分类**：Data（数据获取）、Action（执行操作）、Orchestration（编排调度）。[第二周学习复盘#四、通用的、与 LLM 无关的概念]
4. **Description 六要素模型**：工具描述的质量直接决定模型能否正确选择和调用工具。[第二周学习复盘#四、通用的、与 LLM 无关的概念]
5. **返回值"高信号低噪音"原则**：工具返回结果应精炼有效，避免无关信息干扰模型。[第二周学习复盘#四、通用的、与 LLM 无关的概念]
6. **错误返回的标准格式**：`error_type + message + recoverable + suggestion`，为错误恢复提供结构化依据。[第二周学习复盘#四、通用的、与 LLM 无关的概念]

# Agent Loop 核心机制与第三周演进

## 一、Agent Loop 的核心机制

Agent Loop 的本质可以概括为一句话：**反复问模型"接下来做什么"，直到它说"我做完了"** `[Agent_Loop_设计笔记#一、Agent Loop 的本质：一个带安全网的决策循环#1.1]`。

### 1.1 最简模型（第二周基础版）

第二周实现的 Agent Loop 只有**两个分支** `[第三周学习复盘#二、关键认知变化 > 认知 4#chunk_id:9]`：

```
初始化 messages = [system_prompt, user_message]

for turn in range(max_turns):
    response = LLM.call(messages, tools)

    if response.wants_to_answer:     # finish_reason == "stop"
        return response.text         # 出口 1：正常结束

    if response.wants_tools:         # finish_reason == "tool_calls"
        messages.append(response)    # 保留决策记录
        result = execute(tool_call)  # 开发者执行
        messages.append(result)      # 追加工具结果
        continue                     # 回到循环顶部

return "达到最大轮次"                  # 出口 2：安全终止
```
`[第二周学习复盘#五、Agent Loop 核心模式#chunk_id:17]`

但这有一个致命假设：**模型的决策总是正确的** `[Agent_Loop_设计笔记#1.2 但模型不是可靠的决策者#chunk_id:2]`。

### 1.2 三张安全网体系

第三周实测揭示了三类失败模式，由此抽象出 **"Agent Loop = 决策循环 + 三张安全网"** 的核心架构 `[Agent_Loop_设计笔记#1.3 我的理解#chunk_id:3]`：

| 安全网 | 解决的问题 | 机制 |
|--------|-----------|------|
| **安全网 1：决策审查** | 该做没做（43% 概率） | 模型说"做完了"，但检查它是否跳过了该做的事 → **纠正注入** |
| **安全网 2：执行保护** | 做了但做不成（14% 概率） | 检测连续失败并降级 → **Fallback 策略** |
| **安全网 3：资源控制** | 做完但不停 | 不管决策对不对，到上限就强制终止 → **max_turns/上下文长度** |

三张安全网**独立运作，解决不同层面的问题，任何一张缺失都会留下盲区** `[Agent_Loop_设计笔记#1.3#chunk_id:3]`。

## 二、第三周的演进：从两个分支到多层检查

### 2.1 分支爆炸

第三周加了三个检查机制后，循环的分支从 2 个变成了**指数级增长** `[第三周学习复盘#认知 4#chunk_id:9]`：

```
stop → 已搜索过？ → 是 → 正常返回
                    → 否 → should_have_searched? → 是 → 已纠正过？ → ...
                                                  → 否 → 正常返回
tool_calls → 执行 → 失败？ → 连续N次？ → 已降级过？ → ...
```

`agent.py` 主循环膨胀到 **150+ 行，嵌套层级达 4-5 层** `[第三周学习复盘#六、对后续 Roadmap 的思考#chunk_id:19]`。

### 2.2 通用设计模式："分支-检查-行动"

从这些复杂度中，总结出了可复用的设计模式 `[Agent_Loop_设计笔记#四、模式 1：分支-检查-行动#chunk_id:15]`：

```
每轮循环 {
    模型决策 → 判断 finish_reason

    分支 "stop" {
        检查：该做没做？（决策审查）
        行动：纠正 or 返回
    }

    分支 "tool_calls" {
        行动：执行工具
        检查：连续失败？（执行保护）
        行动：降级 or 继续
    }

    全局检查：资源超限？（资源控制）
}
```

**核心原则：每个分支都有"检查 → 行动"的结构，检查不通过时有明确的恢复行动** `[Agent_Loop_设计笔记#模式 1#chunk_id:15]`。

### 2.3 演进的本质

| 维度 | 第二周（能跑通） | 第三周（受控运行） |
|------|-----------------|-------------------|
| 决策信任度 | 信任模型决策 | **不信任**，每次决策后插入检查 |
| 分支数量 | 2 个（stop / tool_calls） | 多层嵌套，接近指数级 |
| 错误处理 | 依赖模型自行恢复 | 主动注入纠正/降级消息 |
| 目标 | 跑通流程 | **受控运行** |

第三周的目标就是将搜索 Agent 从 **"能跑通"升级为"受控运行"** `[第三周学习复盘#header#chunk_id:0]`，产出物是搜索 Agent v2.0 + Agent Loop 设计笔记。

# RAG 与 Agent 的结合：第四周如何解决外部知识获取与上下文扩展

## 一、核心认知突破：搜索 Agent 本身就是 Agentic RAG

第四周（实际在第五周落地）的关键认知转变是：**第二、三周做的搜索 Agent，其结构（迭代循环 + 规则判断是否检索 + 纠错降级 + trace）正是 agentic RAG 的完整骨架**，区别只在数据源（网页 vs 文档库）和检索方式（关键词 vs 向量）`[第四周第五周学习复盘#认知 2：我一直在做 agentic RAG，只是没意识到]`。

这带来了一个重要结论：**不是从零学 RAG，而是给已会的 agentic 循环接一个新的检索工具**。Agent 的能力本质是"循环 + 工具"的组合，换工具不换骨架`[第四周第五周学习复盘#认知 2：我一直在做 agentic RAG，只是没意识到]`。

搜索 Agent 与 RAG 在逻辑上几乎同构`[第三周学习复盘#第四周（RAG）]`：

| 维度 | 搜索 Agent | RAG |
|------|-----------|-----|
| 检索方式 | 关键词搜索（DuckDuckGo） | 向量相似度搜索（embedding） |
| 数据源 | 互联网（实时） | 预处理的文档库（离线） |
| 内容获取 | HTTP 抓取（不稳定） | 本地读取（稳定） |
| 引用约束 | snippet + URL | chunk 位置 + 文档名 |

---

## 二、落地方案：RAG 作为 Agent 的新工具

### 2.1 代码架构

`agent.py` 主循环**基本不动**——检索只是循环里新增的一个工具`[第四周第五周学习复盘#周四 — RAG 实现]`。`retrieve_documents` 只是 `TOOL_REGISTRY` 里多注册的一项`[第四周第五周学习复盘#认知 2]`。

新增的 `rag/` 四模块从数据流上游往下游实现`[第四周第五周学习复盘#周四 — RAG 实现]`：

| 模块 | 功能 | 关键细节 |
|------|------|---------|
| `rag/ingest.py` | 笔记加载 + chunking | **按标题切**（非按字数），100 chunks / 6 篇笔记，平均 395 字 |
| `rag/embed.py` | embedding 生成 | 千问 `text-embedding-v3`，1024 维，自动分批 + L2 归一化 |
| `rag/store.py` | 向量存储 + top-k | numpy 零依赖，argpartition top-k，持久化 .npy+.json |
| `rag/retriever.py` | 检索 + 引用打包 | min_score 过滤，多 namespace 单例 |
| `tools.py` | 注册 retrieve_documents | 描述为"本地优先"语义 |

### 2.2 关键技术选型：按标题切 > 按字数切

实验 E2b 的 A/B 测试显示了一个关键发现`[第四周第五周学习复盘#周三 — 验证实验]`：
- **按标题切**：hit@1 = 87%（13/15），top-1 相似度 0.764
- **按字数切**（400 字滑窗/80 重叠）：hit@1 = **53%**（8/15），top-1 相似度 0.683

两种切法的 chunk 数与平均尺寸几乎一样（100 个@395 字 vs 94 个@389 字），**34 个百分点的差距纯粹来自"是否保住语义结构"**。结论："检索质量是 RAG 天花板"从口号变成了实测数字`[第四周第五周学习复盘#周三 — 验证实验]`。

## 三、纠正注入的"本地优先"升级

这是从 v2.0 升级到 v3.0 时**必要的额外改动**`[第四周第五周学习复盘#认知 2]`：

- **v2.0**：只有 `should_have_searched`（直答时补联网）
- **v3.0**：**先判 `should_have_retrieved`**（涉及本项目内容先查本地）、**再判联网**，且两个纠正各只注入一次、互不重叠（`agent.py:176-208`）`[第四周第五周学习复盘#认知 2]`

这一改动的本质是：**换工具不换骨架，但多了一个工具就多了一条"该不该用这个工具"的判定支路**`[第四周第五周学习复盘#认知 2]`。这解决了 Agent 对本地知识的获取问题——当用户问题涉及本项目内容时，Agent 会先被纠正去查本地文档库，再考虑联网搜索。

## 总结

第四周解决 Agent 外部知识获取与上下文扩展的核心策略是：

1. **换工具不换骨架**：将 RAG 视为 Agent Loop 中的一个新工具（`retrieve_documents`），复用已有的循环、规则、trace 机制
2. **本地优先双纠正**：`should_have_retrieved` 先判本地文档、再判联网，实现"本地知识优先"的分级检索
3. **语义保全优于机械切分**：按标题切（保住语义结构）比按字数切（400 字滑窗）召回率高 34 个百分点
4. **文档与记忆共用基础设施**：向量存储分 namespace 隔离，为后续记忆系统的上下文装配奠定基础

### 一、核心认知：记忆 ≠ 存储问题，而是"每轮怎么挑"

第五周的首要认知突破是：**记忆系统的难点不在"怎么存"，而在"每轮怎么挑"** `[第四周第五周学习复盘#二、关键认知变化 > 认知 3]`。短期裁剪、长期召回、摘要压缩，本质都是上下文工程问题——每轮调用前决定什么进 prompt、什么留在 store 待召回、什么直接丢 `[第四周第五周学习复盘#二、关键认知变化 > 认知 3]`。

同时厘清了一个核心区分：**RAG 不是记忆、聊天历史也不是长期记忆** `[第四周第五周学习复盘#一、本周时间线 > 周一 — 概念输入与认知建立]`。

### 二、记忆分层设计模式

基于 Microsoft AI Agents for Beginners 记忆课和 LangGraph memory 概念页，确立了记忆分层框架 `[第四周第五周学习复盘#一、本周时间线 > 周一 — 概念输入与认知建立]`：

| 层级 | 范围 | 内容 | 框架对标 |
|------|------|------|----------|
| **短期记忆** | 会话级（thread 内） | 对话历史、工作区状态 | LangGraph short-term |
| **长期记忆** | 用户级（跨 thread） | 偏好、画像、摘要、事实 | LangGraph long-term |

compaction 原则来自 Anthropic《Effective Context Engineering》：**优先丢可重获取的、留不可重建的** `[第四周第五周学习复盘#一、本周时间线 > 周一 — 概念输入与认知建立]`。

### 三、第五周四大设计（全部落地）

#### 设计一：短期记忆——双闸门裁剪

| **主题** | `dict[str,int]` 计数 | 自动统计 | **未进装配**（加权召回留接口，推到第六周）`[第四周第五周学习复盘#一、本周时间线 > 周五 — 记忆实现]` |

#### 设计四：上下文六段装配

六段按固定顺序装配，每段有独立字符预算 `[第四周第五周学习复盘#二、关键认知变化 > 认知 3]`：

| 段号 | 内容 | 预算策略 |
|------|------|---------|
| 段 1 | system prompt | 固定 |
| 段 2 | 偏好（全量） | 小 |
| 段 3 | 摘要（evict 产出） | 中 |
| 段 4 | 长期事实语义召回 | 800 字符 |
| 段 5 | 最近 K 轮对话 | **4000 字符（≈总预算一半）** |
| 段 6 | current（当前问题 + 本轮检索） | 放最末段，叠加 recency |

**关键预算权衡**：段 5（不可重建）> 段 4（可再召回），遵循 compaction 原则"优先留不可重建的" `[第四周第五周学习复盘#二、关键认知变化 > 认知 3]`。装配顺序本身也是一种优先级声明——**越靠后越强约束**，模型即便无显式指令也偏向新信息 `[第四周第五周学习复盘#七、跨模型 / 通用设计原则]`。

### 四、多轮对话上下文保持——E5 四轮实证

E5 用四轮真实对话验证了六段装配的**动态变化** `[第四周第五周学习复盘#一、本周时间线 > 周五 — 记忆实现]`：

| 轮 | 装配出现的段 | 上下文字符 | 事实召回 | 说明 |
|:--:|------|:--:|:--:|------|
| 1 | system, current | 1390 | 0 | 冷启动，记忆为空 |
| 2 | system, **facts**, **recent**, current | 2908 | 3 | 上一轮的事实开始召回 |
| 3 | system, **preferences**, facts, recent, current | 3750 | 3 | 偏好段出现，满配 |
| 4 | 同上（满配五段） | 4026 | 3 | — |

**同一 Agent，每轮喂进去的段构成都不一样**（从冷启动的 2 段长到满配 5 段），这是"每轮重新挑选"的直接证据 `[第四周第五周学习复盘#二、关键认知变化 > 认知 3]`。

### 5. LangGraph的核心概念与状态控制流机制是什么，在第六周是如何重构Agent以实现复杂工作流与多节点编排的？
## 调研子任务 5/6：LangGraph 核心概念与第六周重构

1. **LangGraph 的核心抽象是 StateGraph：一个由节点（node）和边（edge）组成的状态图，数据沿边流动、逻辑在节点执行，把控制流从循环体里抽出来交给图的拓扑** `[第六周学习复盘#九、本周最大收获]`。

2. **框架的第一条设计原则是"节点干活、边做决策"：节点负责执行和改 state，条件边负责读 state 决定下一步去哪，两者不混** `[第六周设计草稿#二、节点切分（设计问题 ②，本周最核心）]` `[第六周学习复盘#八、跨模型 / 通用设计原则]`。

3. **State 用 TypedDict 定义，字段可绑定 reducer（如 `messages` 用 `add_messages` 累加），其余字段默认替换；顺序图不需要为每个字段发明 reducer，要累加就在节点内手动做** `[第六周设计草稿#一、State schema（设计问题 ①）]` `[第六周学习复盘#八、跨模型 / 通用设计原则]`。

4. **Checkpointer 管 thread 内的短期状态（每节点后存快照、按 thread_id 归档，多轮对话靠它续上），Store 管跨 thread 的长期记忆（如偏好/事实/主题），两者混用是新手最常见的架构错配** `[第六周设计草稿#五、状态图（定稿）]` `[第六周学习复盘#八、跨模型 / 通用设计原则]`。

5. **Conditional edge 是实现分支逻辑的原语，配合 `interrupt()` 可在任意节点插入人工审批点，暂停执行等人 `Command(resume=...)` 后继续** `[第六周设计草稿#二、节点切分（设计问题 ②，本周最核心） > 2.3 最终条件边清单]` `[第六周设计草稿#六、周三验证清单]`。

6. **TypedDict 字段无隐式默认值，不写 `init` 节点首轮标志就是 `None` 直接 KeyError，这是 E2 实验的意外发现** `[第六周设计草稿#一、State schema（设计问题 ①）]` `[第六周学习复盘#五、实验与量化 > 框架机制验证]`。

7. **第六周重构的本质：把第三周 150+ 行、4–5 层嵌套的手写 while 循环，拍平成 10 个节点 + 一组条件边的状态图，控制流/数据流/业务判断三者解耦** `[第六周学习复盘#二、本周做的事，一句话]` `[第六周学习复盘#九、本周最大收获]`。

8. **10 个节点分别是：`init`（建默认值+跨轮重置）、`assemble`（六段装配）、`agent`（调模型）、`tools`（执行工具）、两个 `inject_correction_*`（检索/联网纠正）、`inject_fallback`（降级）、`update_memory`（写长期记忆）、`human_review`（interrupt 审批）、`finalize`（写 answer 收尾）** `[第六周设计草稿#二、节点切分（设计问题 ②，本周最核心） > 2.2 最终节点清单]`。

9. **条件边复现了 v3.0 的全部控制逻辑：`route_by_finish` 分流 tool_calls/stop，`need_correction` 复刻"只在从未用过工具时才纠正"的分支1，`check_failures` 复刻连续失败≥2 触发降级，所有回连边带 `turn_count < MAX_TURNS` 闸门** `[第六周设计草稿#二、节点切分（设计问题 ②，本周最核心） > 2.3 最终条件边清单]`。

10. **重构策略分两步：v4.0 先只迁控制流+短期状态（StateGraph + checkpointer），v4.1 再把长期记忆从自研 json+共用向量库迁到 LangGraph SqliteStore，分步做 blast radius 最小** `[第六周设计草稿#〇、本周目标与范围（决策 A 已定）]` `[第六周设计草稿#三、框架边界 + 存储（设计问题 ③） > 3.3 记忆事实存储]`。

11. **五个验证实验（E1–E5）用桩节点隔离验证框架机制：状态传递、init 重置×持久化、chunks 累加/清空、cycle×闸门防死循环、interrupt 开关，13/13 判据全过，把"框架行为"和"业务 bug"分开验** `[第六周设计草稿#六、周三验证清单]` `[第六周学习复盘#五、实验与量化 > 框架机制验证]`。

12. **本周最大收获不是"换了个写法"，而是验证了"先手写再迁框架"的路径：手写的痛让你理解框架每个设计决策的动机，框架不是替代理解而是消除你已理解的苦工** `[第六周学习复盘#九、本周最大收获]` `[第四周第五周学习复盘#六、对后续 Roadmap 的思考 > 第六周（LangGraph）]`。

#### 第 6 周：结构化控制流 — LangGraph 状态图重写
- 把手写循环重写成 **10 个节点 + 一组条件边**的 LangGraph StateGraph`[第六周学习复盘#二、本周做的事，一句话]`。
- 本质是**三层解耦**：控制流交给图的拓扑（节点+边），数据流交给 state + 持久化（checkpointer/store），自己只保留业务判断（装配挑选、纠正规则）`[第六周学习复盘#九、本周最大收获]`。第三周"分支爆炸"的痛点根源正是三者全缠在一个循环体里，状态图把它们拆开，痛点随之消失`[第六周学习复盘#九、本周最大收获]`。
- 长期记忆从自研 json + 共用向量库迁到 LangGraph SqliteStore（v4.1），"手动 save() 六件套"由框架接管`[第六周学习复盘#三、给上周预判打分（本周独有的反思）][第六周学习复盘#二、本周做的事，一句话]`。
- 验证了**"先手写、再框架"的学习路径正确**：三条核心预判全部兑现，手写过 memory/ 六模块的人迁 Store 时对"哪些该交、哪些该留"几乎没有犹豫`[第六周学习复盘#三、给上周预判打分（本周独有的反思）]`。

### 三、技术收益总结

| 维度 | 第 2 周起点 | 第 6 周终点 | 收益性质 |
|---|---|---|---|
| **控制流** | 2-分支 while 循环 | 10 节点状态图 + 条件边 | 从线性到拓扑，分支爆炸问题消除 |
| **可控性** | 无 | 多层防御（Prompt + 规则 + 降级 + trace） | 从"能跑"到"可控且可观测" |

| **记忆** | 无（每轮重新开始） | 短期裁剪 + 长期事实 + 语义召回 + 跨 thread Store | 从一次性到跨轮/跨会话积累 |
| **持久化** | 无 | Checkpointer（thread 内状态）+ SqliteStore（跨 thread 长期） | 从内存到可恢复、可恢复 |
| **可复用性** | 一次性循环体 | 节点独立可配、工具可插拔 | "换工具不换骨架"模式成立 |
| **验证方法** | 手工观察 | 量化实验（E1–E6）+ 自动化测试套件 | 从"我觉得对"到"数据证明对" |

### 四、贯穿始终的三条设计原则

1. **"循环 + 工具"是最小不变量**：从第 2 周到第 5 周，Agent 能力增长的本质就是给同一个循环换工具、加判定支路。第 6 周把循环拍平成图，但"节点干活、边做决策"的分离原则与"工具三分类 + 注册表"一脉相承`[第六周学习复盘#九、本周最大收获][第二周学习复盘#四、Anthropic vs OpenAI/千问 格式对比 > 通用的、与 LLM 无关的概念]`。

2. **"先手写痛点，再用框架消除痛点"**：手写循环 → 体会分支爆炸 → LangGraph 解耦；手写 memory/ 六模块 → 体会持久化脏活 → Store 接管。手写不是浪费，是理解框架"为什么这么设计"的必要前置`[第六周学习复盘#三、给上周预判打分（本周独有的反思）][第二周学习复盘#七、对后续 Roadmap 的思考 > 第六周（LangGraph）]`。

3. **"工程细节区分 demo 与生产"**：工具设计质量、prompt 精确度、循环控制策略、错误恢复机制、装配挑选逻辑、量化验证 — 这些"不性感"的工程细节，才是五周演进中真正积累的核心能力`[第二周学习复盘#八、本周最大收获][第四周第五周学习复盘#八、本周最大收获]`。
