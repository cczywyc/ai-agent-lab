本文梳理了基于 LangGraph 的 Agent 控制流架构从单步执行到多子任务规划的演进过程，重点剖析单步状态图的设计原则、Planner-Executor-Critic 外循环的核心机制，并客观对比两者的收益与工程成本。

### 一、 单步控制流架构的设计特点

在架构重构中，系统将早期版本中 150+ 行、4–5 层嵌套的手写 while 循环，彻底重写为包含 10 个节点与一组条件边的 `StateGraph` 核心拓扑 `[第六周学习复盘#二]`。通过 `draw_mermaid()` 导出的实际拓扑与设计图保持高度一致 `[第六周学习复盘#二]`。

在节点与边的切分上，确立了“节点负责执行与修改状态，边负责读取状态进行路由决策而不改状态”的核心原则 `[第六周设计草稿#二]`。基于此原则，早期设想的 7 个“节点”中，`route`、`check_should_search` 和 `check_consecutive_errors` 被重新归类为条件边 `[第六周设计草稿#2.1]`。

在 State schema 设计方面，全部状态被收敛至一个 `TypedDict` 中 `[第六周设计草稿#一]`。系统设定唯一的 reducer 为 `messages` 的 `add_messages`，其余字段均采用默认替换语义 `[第六周设计草稿#一]`。特别地，`retrieved_chunks` 采用替换语义结合节点内手动累加的方式，刻意避免使用 `operator.add` reducer `[第六周设计草稿#一]`。

### 二、 Planner-Executor-Critic 外循环核心机制

为突破单步控制的局限，系统在 v4.2 版本的“单问题单答案”内层 tool-use 引擎外部，嵌套了一层“多子任务”的 plan 外循环，而非进行底层重写 `[第七周设计草稿#〇][第七周学习复盘#二]`。该设计仅新增了 `planner`、`critic`、`step_init` 3 个节点，以及 `route_after_planner`、`route_after_critic` 2 个 dispatcher 条件边 `[第七周学习复盘#二]`。v4.2 版本的内层引擎（包含 `agent↔tools↔inject_*`）、收尾三连及 Store/RAG 模块均被原样复用 `[第七周设计草稿#〇]`。

双层循环职责分明且互不串扰 `[第七周设计草稿#〇]`。内层循环专注于“这一步怎么做对”，通过 `agent↔tools` 的 tool-use 机制与 `turn_count` 闸门进行控制 `[第七周学习复盘#九]`。外层循环则负责“做什么、做到什么程度算完”，通过 `planner→executor→critic→planner` 的流转以及 `step_index` 和 `replan_count` 双闸门来管理 `[第七周学习复盘#九]`。

### 三、 架构演进的收益对比

**单步控制流（内层引擎）的收益**：
首先，其 blast radius 最小，仅处理单问题单答案的 tool-use 循环，复杂度低且状态字段少，仅需 `turn_count` 一道闸门即可收口 `[第七周设计草稿#〇]`。其次，复用基线稳固，前期重构的五项工作（tools 瘦身、收尾时序反转、空回答重试、判据重审、引用契约收紧）均在单步引擎上落地，为外循环奠定了坚实基础 `[第七周设计草稿#〇]`。最后，调试更为简单，10 节点加条件边的单层链路使得 `draw_mermaid()` 导出图与设计图一致，问题排查路径极短 `[第六周学习复盘#二]`。

**外循环架构的收益**：
系统能力从“单问题单答案”成功扩展至“多子任务结构化报告” `[第七周设计草稿#〇]`。v4.2 内层仅能回答单一问题，而 v5.0 外循环使 Agent 能够完成“输入问题 → 输出研究计划 → 分步检索总结 → 生成结构化报告”的完整链路 `[第七周设计草稿#〇]`。实测表明，同一问题可实现 5/5 全步 accept，生成约 28.7K 字符的报告 `[第七周设计草稿#v0.5→v0.6 变更]`。此外，该架构采用增量叠加而非重写的方式，仅新增 3 个核心节点即可实现能力跃升 `[第七周学习复盘#二]`。

### 四、 外循环引入的工程成本分析

**Token 消耗成本显著增加**：
`planner` 与 `critic` 各自代表一次额外的 LLM 调用 `[第七周设计草稿#决策定稿（v0.6）#chunk_35]`。每个子任务至少需经历拆解、执行与裁决，`critic` 作为独立节点需消耗大量 Token 来审查单步质量 `[第七周设计草稿#决策定稿（v0.6）#chunk_35]`。在真实运行中，单步引用的 12–40 条内容均需经过 `critic` LLM 的裁决 `[第七周设计草稿#第七周设计草稿：planner-executor 技术调研 Agent#chunk_3]`。

**上下文窗口压力转移**：
真正的窗口压力并非来自摘要膨胀，而是子任务内检索 chunk 的累加。实测证明，各步结论摘要受 `MAX_STEPS` 与每步 400 字符上限的约束，峰值仅约 1307 字符，无需额外压缩 `[第七周设计草稿#四、分层装配（设计问题 ④）#chunk_25]`。真正的压力源于宽主题步骤中反复调用 `retrieve_documents`，导致窗口峰值达到约 12.7K–19K tokens `[第七周设计草稿#第七周设计草稿：planner-executor 技术调研 Agent#chunk_3]`。

### 结论

从单步控制流到 Planner-Executor-Critic 外循环的演进，通过“双层嵌套、增量叠加”的设计，在保持内层引擎高内聚、低复杂度的同时，大幅拓展了 Agent 处理复杂多子任务的能力。尽管外循环引入了额外的 Token 消耗与上下文窗口压力，但其在结构化报告生成与任务规划上的收益远超成本，为后续构建更复杂的 Agentic 系统提供了稳固的架构基座。
