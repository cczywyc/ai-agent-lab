# 第八周实验结论（E1–E7：supervisor 多 Agent 框架机制隔离验证）

> 目的：用最小桩节点（worker 不接 LLM、按 cfg 返回预设 `worker_result`）验证 [设计草稿 v0.1](../docs/第八周设计草稿.md) 锁定的 supervisor 多 Agent **框架接线**——路由 / handoff 契约 / 打回循环 / 上下文隔离 / 双层闸门——是否成立。零 API、确定、可复现，每条配对照组（验"对的能跑"也验"拆掉它怎么错"）。
> 对应《第八周实验计划》E1–E7 与草稿 v0.1 的"待验证清单"7 项。结果并入草稿 v0.2。
> 实测环境：LangGraph **1.2.4** / langgraph-checkpoint **4.1.1** / Python 3.12.3（项目 `.venv`）。本文所有数据均为 **2026-06-12** 实跑输出（`run_all.py` 总计 **7/7**），复现方式见文末。

1. **巩固型（升格后照旧成立，符合预期）**：E2/E3/E4/E6 —— researcher=executor、reviewer=critic、supervisor=planner 的"升格不重写"在桩上坐实，v5.0 的闸门/计数器纪律降一层到打回环原样有效。
2. **发现型 ①（计划被证伪 → 实验本身改对）**：E1 计划原定的"prebuilt `create_supervisor` 对拍基线"实测**不是零 API**（prebuilt 路由是 LLM 工具调用决策、需绑定 model），与桩测纪律冲突；且原脚本对照组是**恒真常量** `"writer" != "reviewer"`、根本没跑路由。已重做成零 API 的"写歪路由"探针（真跑一个恒派 writer 的坏 supervisor，让 `routing_accuracy` 评测抓它，命中 1/3）。
3. **发现型 ②（§四/§六 文档缺口，正是计划预判最可能出意外的那条）**：E5 证明 **LangGraph 的"上下文隔离"不是框架分区、是 view-builder 投影纪律**——每个节点物理上拿到的是全 state（对照组实测 writer 越界读到 17 个键、含 reviewer 私有 `draft_score/review_verdict/best_draft/review_notes`），越界读**不报错、只静默串台**。草稿 §四/§六 必须写明隔离要落成"每 worker 显式投影函数"且"框架无护栏"，否则实现一不留神就读全 state。

## E1：supervisor 路由正确性（routing accuracy）★最先跑

**验证（决策 A、B）**：`make_supervisor` 拿到 state 快照后，按当前阶段派给正确 worker——无 findings→researcher、有 findings 无 draft→writer、有 draft→reviewer。把这 3 条沉淀成 `routing_accuracy` 评测种子集（第十周评测周扩到 20+ 条）。桩里 supervisor 不接 LLM，纯按 state 字段做条件分发。

**实测输出**：

```
[E1] supervisor 路由正确性: PASS
      routing_accuracy=3/3 派对；对照(恒派writer)=1/3→评测可检出写歪=True
```

| 判据 | 期望 | 实测 | 结果 |
|---|---|---|---|
| 空 state → researcher | researcher | researcher | ✅ |
| 有 findings 无 draft → writer | writer | writer | ✅ |
| 有 draft → reviewer | reviewer | reviewer | ✅ |
| 对照：写歪路由被评测检出 | 命中 < 3/3 | 1/3（恒派 writer：仅"有 findings"那条蒙对） | ✅ |

**结论**：草稿 §二状态机的阶段判定（`findings`→`draft`→`review_verdict` 三级跌落）成立，supervisor 一个条件函数即可按阶段派活。`routing_accuracy=3/3`。

**精确化（落点 → §一 决策 A′ / 实验计划 E1）——一处计划被证伪 + 对照组重做**：

实验计划与草稿 §一 A′ 原定 E1 对照组用 `langgraph-supervisor` 的 prebuilt `create_supervisor` 做"routing 正确性对拍基线"。本次落地暴露两个问题，证伪了这条计划：

- **prebuilt `create_supervisor` 不是零 API**：它构造的是一个 tool-calling 的 supervisor agent，路由决策**由 LLM 调 `transfer_to_X` handoff 工具产生**、必须 `model=...` 绑定一个模型。拿它当桩测对拍 = 引入 API 调用，直接破"桩 worker 不接 LLM、零 API"的第一纪律。`langgraph-supervisor` 当前也未装、不在 `requirements.txt`。
- **原脚本对照组是恒真常量**：旧 `e1_routing.py` 的对照是 `bad_route, expected = "writer", "reviewer"; ctrl = bad_route != expected`——这是字面量 `"writer" != "reviewer"` 恒为 True，**根本没跑任何路由逻辑**，是没说服力的绿（违反"每条配对照组"纪律）。

→ **修正**：对照组重做成**零 API 的"写歪路由"探针**——`make_supervisor` 加一个 `route_bug` 旋钮（恒派 writer、无视阶段），用同一套 `routing_accuracy(SEED)` 评测真跑它，实测命中 **1/3**（只有"有 findings"那条蒙对），坐实"评测能抓出写歪"。这同时把评测从"3 条断言"升级成可复用的 `routing_accuracy(cfg)` 函数，对得上计划"先写 routing_accuracy 评测再写第二个 worker"。**prebuilt 对拍留到周末真实跑日做**（那天本来就有 API），不在桩测层。

## E2：task_description 四要素完整传递（objective/output_format/tools_hint/boundary）

**验证（决策 C、D）**：supervisor handoff 携带的 `task_description` 四要素原样齐全，尤其第 4 条 **boundary**（v5.0 唯一系统性缺的那条）不被吞。对照组：`drop_boundary` 故意只传前 3 要素，验"缺要素可被检出"。

**实测输出**：

```
[E2] task_description 四要素传递: PASS
      四要素齐全=True（boundary='只查多 Agent，别碰…'）；漏 boundary 可检出=True
```

| 判据 | 期望 | 实测 | 结果 |
|---|---|---|---|
| 四要素齐全 | `{objective, output_format, tools_hint, boundary}` ⊆ td | 四键齐全 | ✅ |
| boundary 非空且为派发值 | "只查多 Agent，别碰单 Agent loop（归 week_3）" | 同左、非空 | ✅ |
| 对照：drop_boundary → boundary 缺失可检出 | `boundary not in td` | True | ✅ |

**结论**：决策 C 的四要素契约在 handoff 层成立，boundary 作为 B 拓扑"各自独立上下文必须显式写归属"的第 4 条原样到达。巩固型，无意外。

**scope 诚实（落点 → §三）**：本条对照组验的是"漏 boundary 在 supervisor 输出层可观测"，**没有**进一步验"worker 缺了 boundary 真会漂"（计划原话"worker 桩能检出契约不完整并标记"）——那要 worker 接 LLM 才看得出漂移，属周末真实跑的事。桩测只能锁"契约这一端传全了"，锁不住"worker 那一端用对了"，这正是"离线验接线、真实验策略"的边界。

## E3：review_count 打回闸门收口

**验证（决策 E）**：reviewer 桩**恒 reject** 时，外层 `review_count < MAX_REVIEW=2` 闸门先于 `recursion_limit` 收口——停在 `review_count=2`、走 finalize（best-so-far 收口路径），不抛 `GraphRecursionError`。对照组：`gate_on=False` 拆掉闸门只靠 `recursion_limit=12`。

**实测输出**：

```
[E3] review_count 闸门收口: PASS
      闸门收口@review_count=2/2；拆闸门撞 recursion=True
```

| 判据 | 期望 | 实测 | 结果 |
|---|---|---|---|
| 恒 reject 停在 `review_count==MAX_REVIEW` | 2 | 2 | ✅ |
| 走 best-so-far 收口出口（finalize） | `worker_result.kind=="final"` | final | ✅ |
| 对照：拆闸门只靠 recursion → 抛异常 | `GraphRecursionError` | 抛出（limit=12） | ✅ |

**结论**：第七周 E4"外循环双闸门主动收口、recursion 只兜底"的教训在 writer↔reviewer 打回环原样成立——`MAX_REVIEW` 是断路器（优雅收口到 finalize），`recursion_limit` 只兜底（触发方式是抛异常、非优雅收尾）。巩固型。

## E4：review_count 与 replan_count 不串扰（两个独立计数器）

**验证（决策 E）**：制造"一次子任务 skip（`replan_count+1`）+ 恒 reject 跑满 review（`review_count` 到 2）"，两计数器各记各的、互不吃额度。对照组：`shared_counter` 合并成一个计数器，看 skip 先占一格后 review 提前收口。

**实测输出**：

```
[E4] 两计数器不串扰: PASS
      独立: review 跑满 2 次（replan=1/review=2）；合并: skip 吃额度→只 review 1 次
```

| 判据 | 期望 | 实测 | 结果 |
|---|---|---|---|
| 独立计数器：replan / review 各记 | `replan=1, review=2` | `replan=1, review=2`，reviewer 被调 2 次 | ✅ |
| 独立：skip 不消耗 review 额度 | review 跑满 2 次 | `_n_reviews=2` | ✅ |
| 对照：合并 → skip 吃掉一次返修额度 | review 不足 2 次 | `replan=2, review=0, _n_reviews=1`（只 review 1 次就收口） | ✅ |

**结论**：第七周"三计数器别串扰"在多 Agent 层重演成立——`replan_count`（supervisor 级、兜"子任务做不下去、跳过"）与 `review_count`（writer↔reviewer 级、兜"这稿不够好、返修"）兜正交维度，合一会出现"某子任务被 skip 顺手吃掉一次返修额度"的串扰（实测合并组只 review 了 1 次就撞闸门收口）。

**精确化（落点 → §五 / §二）——串扰的精确机制**：对照组合并计数器时，关键不在"少跑几次 review"，而在**计数器写字段必须 ≡ 闸门读字段**。`shared_counter` 桩里让 reviewer 的 reject 写 `replan_count`、而 `make_route_after_reviewer` 的 `gate_field` 也切到读 `replan_count`，二者一致才得到"skip(+1)占一格→1 次 review 后闸门收口"的干净串扰演示；若写字段与读字段不一致（写 `replan` 却仍读 `review`），不是"少跑几次"而是**计数器永不达阈、直接死锁撞 recursion**。→ 实现 §二 schema 时，计数器字段的"谁写"与闸门边的"读谁"必须钉成同一个 key（呼应第七周 E5"schema 外更新被静默丢弃"的纪律）。

## E5：上下文隔离（worker 只看该看的）★A→B 唯一的真·新机制

**验证（决策 D、F）**：researcher 只见 `task_description`、writer 只见 `findings+review_notes`、reviewer 只见 `draft`，不读全 state。对照组：关隔离 + 恒 reject（writer 返修第二遍），看 writer 越界读到 reviewer 私有字段 = 串台。

**实测输出**：

```
[E5] 上下文隔离: PASS
      researcher 见=['task_description']；writer 见=['findings', 'review_notes']；隔离关→writer 越界读到 reviewer 私有=True
```

| 判据 | 期望 | 实测 | 结果 |
|---|---|---|---|
| researcher 可见集 = 设计视图 | `{task_description}` | `['task_description']` | ✅ |
| writer 可见集 = 设计视图 | `{findings, review_notes}` | `['findings', 'review_notes']` | ✅ |
| reviewer 可见集 = 设计视图 | `{draft}` | `['draft']` | ✅ |
| 对照：关隔离 → writer 越界读 reviewer 私有 | 含 `draft_score/review_verdict` | writer 读到全 17 键，含 `best_draft/draft_score/review_notes/review_verdict` | ✅ |

**结论**：草稿 §六的坑被坐实——**LangGraph 默认共享 message/state ≠ 物理隔离**。隔离开时三 worker 可见集精确等于各自设计视图；隔离关时 writer 在返修第二遍越界读到整张 state（17 个键），其中 4 个是 reviewer 的私有产出（`draft_score/review_verdict/best_draft/review_notes`）——这就是"writer 错引 reviewer 上一轮措辞"的串台机制。

**精确化（落点 → §四 / §六）——一处文档缺口（本周唯一真·新机制，正是计划预判要盯的那条）**：

E5 暴露的不只是"隔离要拧旋钮"，而是**隔离的实现形态本身**，草稿 §四/§六 现在只说了"state 分区 + worker 视图裁剪"这个意图，漏写了三条机制事实：

1. **框架不分区**：LangGraph 每个节点函数物理上收到的是**全 state dict**（对照组实测 writer 能读到全 17 键）。所谓"分区"不是框架给的护栏，是**约定**——靠每个 worker 的 view-builder 主动只取该取的 key。
2. **越界读静默、不报错**：writer 读 `state["draft_score"]` 不会抛异常、只会拿到 reviewer 私有值并串进下一稿。没有任何框架级信号提示"你越界了"——这比"忘了声明字段被静默丢弃"（第七周 E5）更隐蔽，因为那是少读、这是**多读**。
3. **所以隔离必须落成"每 worker 一个显式投影函数"**：`researcher_view(state)→{subtask_query,boundary}`、`writer_view(state)→{findings,outline,review_notes}`、`reviewer_view(state)→{draft,rubric}`，喂给 LLM 的只能是投影结果、不能是 `state` 本身。这和第七周 E7"executor 看局部靠每子任务重产 SYSTEM_PROMPT 锚"同构——隔离不是天生的，是每次刻意裁出来的。

→ §四 决策 F 必须补"隔离 = 每 worker 显式投影函数 + 框架无越界护栏（多读静默串台）"，否则实现时若图省事直接把 `state` 丢给 worker 的 prompt，对照组那种串台就会上真模型复现。

## E6：best-so-far 收口（防 behavioral collapse）

**验证（决策 E）**：达 `MAX_REVIEW` 仍未 accept 时，收口取历史评分最高的稿、而非最新稿。评分序列 `[0.8, 0.5]`（预算内最新稿 0.5 比历史最好 0.8 更差，模拟 collapse）。对照组：取最新 → 交付更差的 0.5。

**实测输出**：

```
[E6] best-so-far 收口: PASS
      best-so-far 取=0.8（历史最好）；取最新=0.5（更差）
```

| 判据 | 期望 | 实测 | 结果 |
|---|---|---|---|
| best-so-far 收口取历史最好 | score=0.8 | 0.8 | ✅ |
| 对照：取最新 → 更差稿 | score=0.5 | 0.5 | ✅ |

**结论**：决策 E 的"达上限取 best_draft 收口"坐实——后一稿可能比前一稿更差（behavioral collapse），取最新会交付 0.5、取 best-so-far 交付 0.8。

**精确化（落点 → §五）——load-bearing 边界（很少触发的安全网，但极低成本）**：best-so-far 仅在"`MAX_REVIEW` 预算内真发生 collapse（后稿更差）"时 load-bearing；若 writer 单调变好则 `best≡latest`、机制空转。它是**很少触发但极低成本**的兜底（一个 `if score>best.score` 比较），保留它无害；但别把它当主力——主力是早退（reviewer 一 accept 立即出环，正常根本到不了 `MAX_REVIEW`）。这条防止把"取 best-so-far"在文档里夸大成核心机制。

## E7：内外两层闸门嵌套 + recursion_limit 只兜底

**验证（决策 G、H）**：researcher 内层 `turn_count`（v5.0 引擎原样、桩里产出固定 3）与外层 `review_count` 互不误伤；恒 reject 时外层 review 闸门先收口（非 recursion）；总 super-step 远小于 `recursion_limit=10007`。对照组：拆外层闸门只靠 recursion。

**实测输出**：

```
[E7] 双层闸门嵌套: PASS
      turn_count 内层=3；外层 review 闸门收口=True；收口 super-step=10（距 10007 约 1000 倍）；拆闸门撞 recursion=True
```

| 判据 | 期望 | 实测 | 结果 |
|---|---|---|---|
| researcher 内层 `turn_count` 落在自己 scope | 3 | 3（终态不被外层累加污染） | ✅ |
| 外层 review 闸门先收口（非 recursion） | `review_count==2` | 2 | ✅ |
| 总 super-step ≪ 默认 recursion_limit | < 50，且 ≪ 10007 | 10（约 1/1000） | ✅ |
| 对照：拆外层闸门 → 撞 recursion | `GraphRecursionError` | 抛出（limit=12） | ✅ |

**结论**：决策 G/H 的"内层不动、外层新增"复用边界成立——内层 `turn_count`（researcher 内）与外层 `review_count`（supervisor 级）嵌套时各管各的、互不误伤；恒 reject 下外层 review 闸门先于 `recursion_limit` 收口。`DEFAULT_RECURSION_LIMIT=10007`（本周环境复核，与第七周 E6 一致），本任务收口仅 **10** 个 super-step、距它约 **1000 倍**——默认 limit 永远轮不到收口，无需显式调高。

**scope 诚实（落点 → §五 / 实验计划 E7）——一处 sub-claim 属继承非本周复证**：实验计划 E7 原话要验"`turn_count` **每次进 researcher** 从 0 起（不跨 worker 累加）"。但本周桩把 researcher 收缩成**单次调用**（findings 一次性返回、没有 v5.0 那种 plan 子任务内循环），所以 researcher 在一次完整流程里**只进一次**——"每次进归零"这条**没在本周被重新跑出来**，它是从第七周 E6（那条真跑了 4 子任务、实测 `[1,1,1,1]`）继承的。本周 E7 实证的是更弱但仍关键的一条：`turn_count` 被限在 researcher 内层 scope、终态不被外层循环累加成全局计数器。→ 若周四 v6.0 实现给 researcher 接回真正的 plan 内循环（多子任务多次进 researcher），这条"每次归零"需在那时重新复证；桩测层不替它背书。

## 周四（v6.0 实现）技术决策清单

实验结果对实现的直接约束：

1. **routing 用条件函数按阶段分发**（`findings`→`draft`→`review_verdict` 三级跌落），并把 E1 的 `SEED` 三条留作 `routing_accuracy` 评测种子（E1）。**prebuilt `create_supervisor` 对拍放到周末真实跑日**（它需 model、非零 API），不进桩测。
2. **task_description 四要素一个不能少**，boundary 作为第 4 条必须显式写归属（E2）；rubric 写成二元判据（草稿 §三既定）。
3. **`MAX_REVIEW=2` 闸门是断路器、`recursion_limit` 只兜底**：恒 reject 走 best-so-far 优雅收口到 finalize、不靠抛 `GraphRecursionError`（E3）。
4. **`review_count` 与 `replan_count` 两个独立字段**，且**每个计数器的"谁写"必须 ≡ 闸门边的"读谁"**——同一个 key，否则写读错位会死锁撞 recursion（E4）。
5. **隔离落成"每 worker 一个显式投影函数"**：`researcher_view/writer_view/reviewer_view`，喂 LLM 的只能是投影结果不能是 `state`——LangGraph 不分区、越界多读静默串台无护栏（E5，本周最该写进文档的一刀）。
6. **best-so-far 保留但别夸大**：一个 `if score>best.score` 的低成本兜底，仅 collapse 时 load-bearing；主力是 reviewer 一 accept 的早退（E6）。
7. **内外两层闸门各管各的、`turn_count` 限在 researcher scope**；总 super-step（本周桩 10）距默认 `recursion_limit=10007` 约三个数量级，**无需显式调高**（E7，复核第七周 E6 结论本周仍成立）。若给 researcher 接回真 plan 内循环，需复证"每次进归零"。

**对 v0.1 草稿的修订建议（→ 并入 v0.2）**：本周无"推翻核心决策"的意外，三处需并入——

- **E1（发现型）→ §一 决策 A′ 修正**：把"prebuilt `create_supervisor` 当周三 E1 对拍基线"改为"prebuilt 非零 API、对拍移到周末真实跑；桩测层 E1 对照用零 API 写歪路由探针"。
- **E5（发现型，文档缺口）→ §四/§六 必补**：隔离 = 每 worker 显式投影函数 + 框架无越界护栏（多读静默串台）；现 §四/§六 只写了"state 分区 + 视图裁剪"的意图，漏了"框架不分区、靠投影约定"这条机制事实与"越界多读不报错"这个坑。
- **E4/E7（精确化）→ §五/§二**：计数器"写字段 ≡ 闸门读字段"必须钉成同一 key（E4）；E7"每次进 researcher 归零"本周属继承（researcher 单次调用未复证），实现接回内循环后需复证（E7）。

并把草稿 §决策定稿表/待验证清单从"待周三实验验证"更新为 **"已实跑、7/7 过（E1–E7，每条配对照组）"**。
