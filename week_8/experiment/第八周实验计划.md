# 第八周实验计划：supervisor 多 Agent 框架机制隔离验证（E1–E7）

> 周三实验日。延续第七周 E1–E7 的打法：**桩节点隔离验框架机制**——worker 不接 LLM、返回预设结果，只验 supervisor 路由 / handoff / 打回循环 / 上下文隔离这些**框架接线**是否对。零 API、确定、可复现，每条配**对照组**（验"对的能跑"也验"错的怎么错"）。
> 对应：《第八周设计草稿 v0.1》的"待验证清单"7 项。结果跑完并入草稿 v0.2。
> 落地：新增 `test_multiagent.py`（W14– 续 W1–W13 编号），桩 worker 放 `experiment/stubs/`。

---

## 总原则（承第七周，每条都不许破）

- **桩 worker 不接 LLM**：researcher/writer/reviewer 桩按入参返回预设 `worker_result`，把"框架行为"和"模型质量"分开——本周验前者，后者留周末真实跑。
- **离线验接线、真实验策略**：第七周最硬的教训。桩测锁"supervisor 派对人、handoff 传全契约、闸门收口"，锁不住"reviewer 在真实模型下判得准不准"。**离线全绿是必要非充分。**
- **每条配对照组**：不光验机制成立，还要验"拆掉它会怎么错"——没有对照组的绿是没说服力的绿。
- **先写 routing_accuracy 评测再写第二个 worker**（业界通用纪律）：E1 就是这条评测，必须最先跑。
- **桩里也要能制造失败**：恒 reject 的 reviewer 桩、恒 fail 的 researcher 桩，是验闸门/收口/上报的前提。

---

## 实验清单

### E1 — supervisor 路由正确性（routing accuracy）★最先跑

**验什么**：supervisor 拿到 state 后，是否按当前阶段派给**正确**的 worker（无 findings→researcher；有 findings 无 draft→writer；有 draft→reviewer）。

- **桩设置**：构造三种 state 快照（空 / 有 findings / 有 draft），跑 supervisor 路由节点，断言 `active_worker` 与 `delegate_to_X` 工具调用匹配预期。
- **判据**：3/3 阶段路由正确；handoff 工具名与目标 worker 一致。
- **对照组**：用 `langgraph-supervisor` 的 prebuilt `create_supervisor` 跑同样三快照做**对拍基线**——手搓路由与 prebuilt 结论一致才算路由逻辑没写歪。
- **附带**：把这 3 条沉淀成 `routing_accuracy` 评测的种子集（第十周评测周会扩到 20+ 条）。

### E2 — task_description 四要素完整传递

**验什么**：supervisor handoff 携带的 `task_description` 四要素（objective / output_format / tools_hint / **boundary**）是否原样传到 worker 桩，**尤其第 4 条 boundary 不被吞**（v5.0 唯一系统性缺的那条）。

- **桩设置**：supervisor 派活时塞入四要素齐全的 task_desc，worker 桩把收到的 task_desc 原样回写进 `worker_result`，断言四键齐全、值未丢。
- **判据**：4/4 要素到达；boundary 字段非空且等于派发值。
- **对照组**：故意只传前 3 要素（漏 boundary），断言 worker 桩能检出"契约不完整"并标记——验"缺要素 worker 会漂"这个设计前提成立。

### E3 — review_count 打回闸门收口

**验什么**：reviewer 桩**恒 reject** 时，外层 `review_count < MAX_REVIEW` 闸门是否先于 `recursion_limit` 收口（停在 `review_count = MAX_REVIEW = 2`、出 best-so-far 收口路径）。

- **桩设置**：reviewer 桩永远返回 `verdict=reject`，writer 桩每次返回略改的稿；跑 writer↔reviewer 环。
- **判据**：环停在 `review_count=2`，走"取 best_draft 收口"出口，不抛 `GraphRecursionError`。
- **对照组**：拆掉 `review_count` 闸门、只靠 `recursion_limit`，断言恒 reject 会一直打回直到撞 limit 抛异常——坐实"闸门是断路器、recursion 只兜底"（第七周 E4/E6 在打回环的重演）。

### E4 — review_count 与 replan_count 不串扰（两个独立计数器）

**验什么**：制造"一次子任务 skip（`replan_count+1`）+ 一次 writer 返修（`review_count+1`）"，两个计数器**各记各的、互不吃额度**（第七周"三计数器别串扰"在多 Agent 层的重演）。

- **桩设置**：researcher 桩对某子任务恒 fail（触发 skip-and-advance，`replan_count+1`）；reviewer 桩对初稿 reject 一次再 accept（`review_count` 走 1 再早退）。
- **判据**：终态 `replan_count=1`、`review_count=1`，两者独立；skip 不消耗 review 额度、反之亦然。
- **对照组**：故意合成一个计数器，断言会出现"子任务被 skip 顺手吃掉一次返修额度"的串扰——坐实必须分开。

### E5 — 上下文隔离 / 路由真生效

**验什么**：worker 只看到该看的（researcher 看 `{subtask_query, boundary}`、writer 看 `{findings, outline, review_notes}`、reviewer 看 `{draft, rubric}`），**不读全 `messages` 历史**。

- **桩设置**：worker 桩把"自己实际能读到的 state 键"回写进 `worker_result`，断言每个 worker 的可见集等于设计的视图、不含其它 worker 的私有分区。
- **判据**：三 worker 可见集 = 各自设计视图；无越界读取。
- **对照组**：让 worker 桩读全 `messages` 共享历史，构造一个"writer 错引了 reviewer 上一轮 notes 里的措辞"的串台场景——坐实"supervisor 默认共享 messages ≠ 物理隔离、隔离要刻意拧"这个 §六 的坑。

### E6 — best-so-far 收口（防 behavioral collapse）

**验什么**：达 `MAX_REVIEW` 仍未 accept 时，收口取**评分最高的历史稿**而非最新稿。

- **桩设置**：reviewer 桩给三稿打分 `[0.6, 0.8, 0.5]`（最新稿最差，模拟 behavioral collapse），恒 reject 跑满 `MAX_REVIEW`。
- **判据**：收口交付的是第 2 稿（score 0.8 = best），不是第 3 稿（0.5）；`best_draft.score=0.8`。
- **对照组**：收口取最新稿，断言会交付更差的 0.5 稿——坐实"后一稿可能更差、必须取 best-so-far"。

### E7 — 内外两层闸门嵌套 + recursion_limit 只兜底

**验什么**：researcher 内层 `turn_count` 循环（v5.0 引擎原样）嵌进外层 supervisor 流程时，两层闸门互不误伤；`turn_count` 每次进 researcher 归零；外层闸门先于 `recursion_limit` 收口。

- **桩设置**：researcher 桩内层真跑几轮（mock `turn_count` 推进）、reviewer 桩制造若干次打回，全程统计 super-step 与各计数器。
- **判据**：`turn_count` 每次进 researcher 从 0 起（不跨 worker 累加）；内外闸门嵌套下都正常收口；总 super-step 远小于 `recursion_limit=10007`（沿用第七周 E6 结论，复核一次本周 `MAX_REVIEW×writer轮` 量级仍差几个数量级）。
- **对照组**：拆外层闸门只靠 recursion，断言会跑到撞 limit——同 E3 但聚焦"内层不动、外层新增"的复用边界（决策 G/H）。

## 判据汇总表

| E | 验的机制 | 主判据 | 对照组（错的怎么错） | 验哪个决策 |
|---|---|---|---|---|
| E1 | supervisor 路由正确性 | 3/3 阶段派对 worker | prebuilt 对拍不一致 = 路由写歪 | A、B |
| E2 | task_desc 四要素传递 | 4/4 到达、boundary 不丢 | 漏 boundary → worker 漂 | C、D |
| E3 | review_count 闸门收口 | 停在 count=2、走 best-so-far | 拆闸门 → 撞 recursion 抛异常 | E |
| E4 | 两计数器不串扰 | replan=1 / review=1 各记 | 合一 → skip 吃返修额度 | E |
| E5 | 上下文隔离 | 可见集=设计视图、无越界 | 读全历史 → 串台 | D、F |
| E6 | best-so-far 收口 | 交付 best 而非最新 | 取最新 → 交付更差稿 | E |
| E7 | 双层闸门嵌套 | turn_count 每 worker 归零、外层先收口 | 拆外层 → 撞 recursion | G、H |

## 与设计草稿的闭环

跑完 E1–E7 后回头审草稿 v0.1（第七周教训：**实验不止验假设，还是给设计挑错的数据源**——v0.2→v0.3 那 4 处缺口就是这么抓出来的）。重点回审三处最可能"该写而没写"的地方：

1. **E5 若暴露隔离漏洞** → 补 §六 worker 视图的精确边界（类比第七周 E7 逼出"跨子任务重锚"）。
2. **E2 若 boundary 易被吞** → 在 §三 task_description 加显式校验 / schema 约束（类比 E5-week7 的"critic_verdict 必须声明进 schema"）。
3. **E1 routing 若有歧义阶段**（如 findings 部分完成时派谁） → 回 §七 状态图补这条边的判据。

**无意外也要点明是"巩固型还是发现型"**（第七周教训）：本周机制大多是 v5.0 furniture 的升格复用，预期 E1–E7 多为"升格后照旧成立"的坐实型；真正可能出意外的是 E5（上下文隔离——这是 A→B 唯一的真·新机制，没有 v5.0 对应物）。带着这个预期跑，重点盯 E5。
