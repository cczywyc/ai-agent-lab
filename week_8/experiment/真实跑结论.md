## 三处"最可能摔"的实测落点（盯的就是这三处）

### ① reviewer 判太严 / 太松？→ 落在**健康中段**，但两条断路器路径真实跑没压到

- **没太松**：A/B/C 三题初稿**都被 reject**（0.85 / 0.40 / 0.65），打回循环真触发，不是"初稿即 accept 等于新机制没验到"。
- **没太严**：没有一题**连续 reject 两次**——返修稿全部 accept（统一 0.95）。**这不是橡皮图章**：round-1 NOTES 条条具体、可执行（见下），round-2 真把问题改掉了才 0.95。
- **reject 的理由是真的**（round-1 NOTES 实录）：
  - A：「存在未接地论断：'第三周 150+ 行、4–5 层嵌套'、'原设想的 7 个节点'、'v4.2' 等具体数字与版本」缺精确引用。
  - **B：「第一部分第2点出现 `[doc#section]` 及 `...#…` 等占位符和无效引用，且存在严重的引用堆砌（单句后挂十余个引用）」** ← reviewer 独立抓出了下面缺陷 #2 和 #4。
  - C：「严重的'引用堆砌'（第三、四部分单句后堆砌 10+ 个引用），未做到论断与引用的精准映射；部分引用格式不规范（含 `>`）」。
- **代价**：`MAX_REVIEW` / best-so-far 收口、skip 防御收口这两条**断路器路径**，在 A/B/C 的"健康中段 reviewer"下**根本没被压到**（早退主力，正如设计 §五"MAX_REVIEW 只兜死循环、正常到不了"）。→ 故意造了 **Topic D** 把 skip 路径压出来（见 ④ / 缺口表）；best-so-far（双 reject）仍只在桩 E6 验过、真实跑未触发（诚实记一笔）。

### ② findings 600 字截断饿 writer？→ **确认 100% 触顶，已修**

- A/B/C 三题 **12/12 条 ok finding 全部被砍到 600**（`finding_point_lens` 全 = 600）。`_make_finding` 的 `[:600]` 对每条 finding 都是 binding 约束——writer 只看 findings 投影、看不到研究全轨迹（隔离的代价），每条都被砍意味着信息在系统性流失。
- **症状**：C 题成稿偏教科书通用、丢项目特定细节（无 v2.0→v6.0 版本号、无具体决策编号）——隔离的"喂够 writer"被 600 拖了后腿。但三题成稿仍 accept@0.95，说明**没饿死、是饿瘦**。
- **修**：`FINDING_MAX_CHARS = 600 → 1500`（config 常量化）。**D 复跑实测**：week_6 那条 ok finding 现为 1500 字（修前必是 600），writer 拿到 2.5× 信息量。仍是"压缩摘要"非全轨迹（§四纪律不破）。

### ③ reviewer 不 follow VERDICT/SCORE/NOTES？→ 真实跑**不是问题**（潜在脆弱已记）

- 真实跑 **7/7 轮 parse_ok=True**——qwen 在真实 `REVIEWER_PROMPT`（明写三行格式 + 二元 rubric）下稳定吐 `VERDICT: accept|reject` / `SCORE: 0.x` / `NOTES:`。摔点③在真实跑里没发生。
- **但潜在脆弱有据**：建 harness 时用一个**裸 prompt** 探针，qwen 吐了 `VERDICT: PASS` / `SCORE: 10/10`——而 `_parse_review_verdict` 对 `FAIL` 会**默认 accept**（漏判反成放行）、`_parse_review_score` 对 `8/10` 会**默认 0.5**（best-so-far 选错稿）。真实 prompt 下没触发，**故不改解析器（不做未经真实跑验证的改动）**，记为 v0.3 潜在风险：若换模型/精简 prompt 致格式漂移，需补 PASS/FAIL、X/10、通过/不通过 的兜底解析。

## 真实跑坐实的（v6.0 三条新机制 + 继承机制，real 级）

1. **打回循环 + Reflexion verbal reinforcement**：A/B/C 三题 reject→返修→accept，score 全部**升**（0.85→0.95 / 0.40→0.95 / 0.65→0.95）。返修不是重写、是拿 review_notes 针对性改（B 把 `doc#section`/堆砌清理掉、A 给数字补引用）。**桩 E3 验"恒 reject 收口"，真实跑验"正常 reject→改好→早退"这条主力路径。**
2. **skip-and-advance + 正交计数器（D）**：杜撰 QSCC 子任务 escalate→skip、依赖它的对比子任务也 skip，`replan_count=2 / review_count=0`——**两计数器真实跑互不吃额度**（桩 E4 的 real 复证）。
3. **防御式部分收口（D）**：2/3 子任务 skip、只剩 1 条 ok finding，writer **仍写出 2161 字成稿**、reviewer accept——"ok_findings 不为空就别让 writer 从空气里写、但有一条就写"的收口判据成立。
4. **上下文隔离投影（E5 的 real 复证）**：4/4 题 `isolation_clean=True`，三 worker 真实可见集**精确等于设计视图**（researcher=`{boundary,subtask_query}`、writer=`{findings,outline,review_notes}`、reviewer=`{draft,rubric}`），真实跑里**没串台**。桩 E5 是本周唯一真·新机制、这里真模型复证。
5. **内层接地软闸门（继承 v5.0）真实跑有效**：B step0「49 cites grounding=0.02 → retry」、step2「35 cites 0.46 → retry」、C step1 retry——researcher 内层 critic 的"引用接地比例 < 0.5 判 retry"在真实模型下抓出了引用堆砌/编造，retry 后 accept。

## 真实跑挖到的策略缺陷与处置

| # | 缺陷 | 证据 | 处置 | 验证 |
|---|---|---|---|---|
| 1 | **findings 600 截断对 100% finding binding**、饿瘦 writer | A/B/C 共 12/12 条触顶 600；C 成稿丢项目细节 | **已修**：`FINDING_MAX_CHARS 600→1500` | D 复跑 finding=1500；48/48 桩测无回归 |
| 2 | **`doc#section` 字面模板占位漏成引用**（SYSTEM/WRITER_PROMPT 范例被原样抄进正文） | A/B/C 三题交付引用汇总都混进 `doc#section`；**reviewer 自己在 B round-1 抓出它** | **已修**：`_extract_citations` 剔除字面占位（doc=='doc'&section=='section'） | D 交付 `doc#section`=0（A 修前=1）；离线断言通过 |
| 3 | **skip 后 writer 静默吞掉跳过的子任务、不在正文标注缺口** | D：用户要"LangGraph vs QSCC 对比"，2 子任务 skip 后成稿只字未提 QSCC、reviewer 仍 accept（rubric"扣题"只查"覆盖 findings"、不查"覆盖原始主题"） | **记 v0.3**：writer/finalize 应在正文显式标注"未能覆盖：[skipped 子任务]"；rubric 加一条"是否声明研究缺口" | 待修（额度耗尽，不做未验证改动） |
| 4 | **引用堆砌**：writer 单句后挂 10+ 引用（findings 把整条 step 的引用集全挂到一点） | B/C round-1 NOTES 都点名"引用堆砌、论断-引用未精准映射" | **记 v0.3**：WRITER_PROMPT 约束"每论断只标最相关 1–2 引用"；**但打回循环已自愈**（round-2 清理掉了）——多 Agent reviewer 兜底 writer 过度，正是 §一价值点"角色专精"的实证 | 自愈（reject→改好），prompt 硬约束待验证 |
| 5 | **supervisor 把"本项目"题拆成通用子任务** → researcher web 漂移、烧 token | C：子任务拆成"Tool Calling 的基础实现原理是什么"（丢"我们项目"语境）→ researcher 上 wikipedia/brave/mojeek 查通用概念（且幻觉出不存在的 `edit` 工具名，被 UnknownTool 兜住）；最终靠 synthesis-reserve 收回本地引用 | **记 v0.3**：SUPERVISOR_PROMPT 拆解时保留"本项目/本地库"语境锚，减少 researcher 通用化漂移 | 待验证 |

> 处置纪律（承 verify/系统化调试）：**只修真实跑坐实的代码缺陷（#1/#2），且改完离线复验 + D 集成复跑确认**；#3/#4/#5 属策略/质量缺口或潜在风险，**不做额度耗尽下无法复跑验证的改动**，记进 v0.3 待下次真实跑压。

## 多 Agent 串接的成本账（token）

| 题 | researcher | critic | writer | reviewer | supervisor | 合计 | 耗时 |
|---|---|---|---|---|---|---|---|
| A | 171,941 (×12) | 17,410 (×4) | 17,806 (×2) | 8,466 (×2) | 1,128 (×1) | **216,751** | 167s |
| B | 246,419 (×18) | 19,269 (×3) | 15,549 (×2) | 8,536 (×2) | 1,015 (×1) | **290,788** | 202s |
| C | 234,802 (×24) | 30,318 (×6) | 17,881 (×2) | 8,473 (×2) | 1,367 (×1) | **292,841** | 319s |
| D | 202,828 (×13) | 15,273 (×3) | 5,418 (×1) | 3,151 (×1) | 1,800 (×1) | **228,470** | 372s |

- **researcher（继承 v5.0 的 RAG 引擎）独占 79–85% token**；v6.0 **新增的 writer+reviewer 打回层只占 ~8–12%**（每题 ~26k）。→ **多 Agent 串接没把成本炸开**：贵的是带检索的 researcher（老问题），writer↔reviewer 打回循环是小额加价。这条直接支撑 §一"本周价值在角色契约、不是更强"——成本主体仍是内层引擎。
- **硬天花板**：A+B+C+D 合计 **1,028,850 token**（+ 2 次 ingest + v5.0 对比跑），**一个 session 烧穿 qwen3.7-max 免费额度**。多 Agent 串接 × RAG 的 token 量级，在免费额度下跑不了几轮——评测/复跑要算额度账。

## v5.0 ↔ v6.0 纵向对比（同题 C，"降一层复用"最直接体感）

同一主题"第二周到第六周 Agent 演进"：

| | v5.0（week_7，planner-executor-critic） | v6.0（week_8，supervisor 多 Agent） |
|---|---|---|
| 交付正文 | **20,905 字**（`friday_drafts/C_v5_report.txt`） | **2,037 字**（`friday_drafts/C_v6_draft.md`） |
| 组织方式 | `finalize` 把 6 步 step_results **机械拼接**：`## 分步结论` 下每步一段，各步自带 `### 一/二/三` 各写各的结构 → 详尽但是结构**缝合怪**、重复引言 | writer **跨 findings 综合**：统一引言→5 节→结论，每节一段带引用，**reviewer 二元 rubric 把关到 0.95** |
| 体感 | 穷举式全量倒出（每步原文级），信息全但读着累、层级乱 | 成稿式收敛叙述（10× 更短更顺），代价是丢了部分项目特定细节（叠加缺陷 #1 的 600 截断） |

**"降一层复用"的实证**：v5.0 的 `finalize`（代码装配）→ v6.0 的 writer（LLM 写作角色），交付从"步骤拼接"升格成"角色成稿"。这正是周一那根刺的回应——顺序流水线本可用拓扑 A 更省，选 B 买到的是**角色专精的成稿质量 + reviewer 质量闸门**（v5.0 没有质量门、20,905 字直接交付）。

## 诚实记一笔（coverage 缺口）

- **best-so-far 双-reject 收口**：A/B/C 健康中段 reviewer 从不连续 reject 两次 → 该路径真实跑**未触发**，仍只桩 E6 验过。要真实压它需更挑剔 rubric 或更难的题逼出双 reject（额度耗尽，留下次）。
- **D 的 reviewer 跑在旧模型**（qwen3.7-plus）且 round-1 直接 accept——可能模型更松、也可能 D 的单主题成稿本就好过 A/B/C 的对比/评估题，**两个变量没解耦**，不据此结论"旧模型更松"。
- **skip 难触发本身是发现**：B 想压 skip 没压成（week_8 子任务被相邻 week_7 前瞻内容救活、researcher 诚实改写成"规划而非已落地"）——真实语料有相邻内容时，researcher 足够机敏、escalate 比桩里难得多。D 用**纯杜撰模块**（QSCC，本地+联网都查不到）才稳定逼出 skip。
