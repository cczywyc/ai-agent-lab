# 什么时候该用多 Agent，什么时候不要用

> **定位**：roadmap 第八周产出之一。不是转述 Anthropic / OpenAI 的观点，而是**拿自己第八周的 v6.0 demo（supervisor 多 Agent 技术写作 Agent）当样本，用真实跑数据把判据称出来**。
> **证据来源**：Anthropic《How we built our multi-agent research system》、OpenAI Agents SDK 文档（判据骨架）；自己的实测——`week_8/experiment/真实跑结论.md`（4 题真实 API 跑）、v5.0↔v6.0 同题纵向对比、桩测 E1–E7 + 离线 127/127。
> **配套**：判据的概念推导见 `多Agent概念.md`（§五/§十）；本文是它的**实证收口版**。

---

## 0. 一句话主轴

**从"一张图、共享 state"（拓扑 A）跨到真·多 Agent（独立上下文 + handoff），实打实多买到的只有"独立上下文 + 真并行"两样，代价是"约 15x token + 协调成本 + 隔离副作用"。它只在子任务能切成互相独立的并行分支时才划算；顺序依赖、要共享同一份上下文的活，留在一张图里反而更省更稳。**

但我这周的 demo 是一条**顺序流水线**（researcher→writer→reviewer，强依赖、买不到真并行），所以它本身就是这条主轴的**反例样本**——下面三块实证里有两块来自"在非甜区硬上多 Agent 会怎样"。

---

## 一、判据：一条主轴 + 三个体征

**主轴判据**（Anthropic 原则）：**需要所有 agent 共享同一上下文、或 agent 之间存在大量依赖的领域，目前不适合多 Agent。** 并行只在子任务真正互相独立时才有收益——**如果 subagent B 要等 A 的产出才能开工，"并行"就退化成带额外开销的串行。**

**三个"该用"的体征**：

| 体征 | 说明 | 反例（不该用） |
|---|---|---|
| **广度优先（breadth-first）** | 答案需横扫一大片来源/候选（"S&P 500 IT 板块所有董事会成员"），单上下文装不下 | 深度优先、单线推理 |
| **子任务可独立并行** | 各分支互不依赖、能同时跑 | 强顺序依赖的流水线 |
| **结果价值 > 15x 成本** | 高价值产出（深度调研报告）才值这个 token 账单 | 日常对话、快速问答 |

**配套：努力分级（effort-scaling）**——"派几个 agent"由任务复杂度驱动、不是常数：简单事实查证 1 个、直接对比 2–4 个、复杂研究 10+ 个。（呼应第七周 planner 拆 5/3/5/4/5 步的非确定性。）

---

## 二、明确不该用多 Agent 的场景

- **强顺序依赖的流水线**：下一步必须等上一步产出（researcher→writer→reviewer 正是此类）。多 Agent 在这里买不到并行，只剩协调与隔离的开销。
- **需共享同一份上下文的任务**：编码 / 调试这类要全程看同一份代码/状态的活，隔离反而割裂上下文、几乎必亏。
- **简单事实查询**：给简单问题猛拆 subagent 只会徒增冗余检索（早期系统的教训）。

## 三、三块真实跑实证压舱石（本文的核心，拿自己 demo 称出来的）

### 实证 1 — token 账：多 Agent 串接**没把成本炸开**，但有**硬天花板**

v6.0 真实跑 4 题的 token 分布（`真实跑结论.md`）：

| 角色 | token 占比（4 题） | 说明 |
|---|---|---|
| researcher（继承 v5.0 的 RAG 引擎） | **79–89%**（A/B/C 79–85%、D 88.8%） | 成本主体，带检索的内层引擎——这是**老问题**，不是多 Agent 带来的 |
| **writer + reviewer 打回层（v6.0 新增）** | **3.8–12.1%**（A/B/C 8–12%/约 24–26k 每题；D 单轮 3.8%） | 多 Agent 串接的**净加价**；D 因 reviewer round-1 即 accept、打回只 1 轮，占比最低 |
| supervisor / critic | 余下 | 拆解 + 内层自检 |

**两条结论**：
1. **多 Agent 串接（在中心编排 B 拓扑下、顺序调用）本身很便宜**——新增的 writer↔reviewer 打回循环只占 ~10%，没把成本炸开。"15x token"那个吓人的数字是**并行多 subagent 同时烧** 的代价（Anthropic 的广度优先研究场景），顺序串接的 B 拓扑根本没那个量级。
2. **但有硬天花板**：4 题合计 **~103 万 token，一个 session 烧穿了 qwen3.7-max 的免费额度**（A/B/C 跑完即 403、D 被迫换模型）。→ **多 Agent × RAG 的 token 量级，在免费额度下跑不了几轮**，评测/复跑要先算额度账。

> **判据含义**：成本不是"要不要上多 Agent"的主要否决项（串接很便宜）；真正的否决项是**"买不买得到并行/独立上下文"**。但如果你的内层是 RAG（像我的 researcher），token 天花板会先于一切把你拦住——先优化内层检索成本，再谈多 Agent。

### 实证 2 — v5.0 ↔ v6.0 同题对比：买到的是"成稿质量 + 一道质量门"，**不是"更强/更快"**

同一主题"第二周到第六周 Agent 演进"，v5.0（一张图、finalize 机械装配）vs v6.0（supervisor + writer + reviewer）：

| | v5.0（拓扑 A，无独立 writer/reviewer） | v6.0（拓扑 B，writer 成稿 + reviewer 质量门） |
|---|---|---|
| 交付正文 | **20,905 字** | **2,037 字** |
| 组织方式 | 6 步 step_results **机械拼接**，每步自带 `### 一/二/三` → 详尽但结构缝合怪、重复引言 | writer **跨 findings 综合**：统一引言→5 节→结论 |
| 质量门 | **无**（装配完直接交付） | **有**：reviewer 二元 rubric 审、reject→返修→accept 到 0.95 |

**这条顺序流水线买不到"真并行"**（没有可并行的独立分支），但拿出 v6.0 vs v5.0 一比，它确实买到了两样实打实的东西：
- **角色专精的成稿质量**：writer 是"专门写"的角色，把研究要点综合成 10x 更短、更顺的稿，而非 v5.0 那种各步原文缝合。
- **一道质量闸门**：reviewer 真在干活——A/B/C 三题初稿全被 reject（score 0.85/0.40/0.65）、返修后升到 0.95；它甚至自己抓出了 writer 抄进正文的 `[doc#section]` 占位符和"单句堆 10+ 引用"。**多 Agent reviewer 兜底 writer 过度，是"角色专精"在真实跑里自己证明了自己。**

> **判据含义**：当任务是顺序流水线、并行无从谈起时，**多 Agent 仍可能值得——如果你要的是"角色专精的产出质量"和"一道独立的质量门"**。但要诚实：这时你买的不是 Anthropic 那套"并行 + 独立上下文窗口"，而是**"composition 升格 + reviewer 闸门"**。别拿"多 Agent 更强"当理由，那是并行场景的话。

### 实证 3 — 本周 demo 自身就是"非甜区"反例：顺序流水线本可用拓扑 A 更省

按 §一 判据称 researcher→writer→reviewer：**强顺序依赖**（writer 等 researcher、reviewer 等 writer）→ 落在多 Agent 的**非甜区**。所以选拓扑 B（supervisor）**是为了学机制**（独立上下文投影 + 委派契约 + 打回循环），不是因为它更优。

**诚实记一笔**：纯粹论"把这份技术写作报告交付出来"，**拓扑 A（一张图加 writer/reviewer 节点、共享 state）更省更稳**——省掉了隔离投影的实现成本、省掉了"findings 600 截断饿瘦 writer"这种隔离副作用（v6.0 真实跑 12/12 条 finding 触顶、被迫把上限从 600 提到 1500）。隔离的代价是真金白银的：**worker 干净了，但 writer 看不到研究全轨迹、要专门拧"喂够下游"的旋钮。**

> **判据含义**：**选多 Agent 之前先问一句"我是为了学机制，还是任务真需要它"。** 如果是顺序流水线 + 单上下文够用，拓扑 A 几乎总是更省；多 Agent 的隔离会引入新的副作用（下游被饿瘦、上下文路由要显式兜）。我这周明知非甜区仍选 B，是因为**学习目标**是"体会 handoff / 隔离"——这是合理的，但要标清楚这是学习选择、不是工程最优。

## 四、决策清单（拿到一个任务，照着称）

1. **子任务能并行吗？**（互不依赖、能同时跑）→ 不能（顺序依赖）：基本别为"并行"上多 Agent，那是带开销的串行。
2. **需要共享同一份上下文吗？**（编码/调试类）→ 需要：留一张图（拓扑 A），隔离会割裂上下文。
3. **是广度优先、单上下文装不下吗？**（横扫一大片来源）→ 是 + 可并行 + 高价值：**这才是多 Agent 的甜区**（Anthropic 研究场景）。
4. **若是顺序流水线，但要"角色专精的产出质量 + 一道质量门"吗？**→ 要：可上**中心编排 B（orchestrator + agents as tools）**，成本只加 ~10%（实证 1），但买到 composition 升格 + reviewer 闸门（实证 2）。**别上去中心化 C**——顺序链上 handoff 收益有限、还要防打回不收敛。
5. **结果价值 > 15x 成本吗？**→ 否（日常对话/简单查询）：别拆，徒增冗余。
6. **内层是 RAG / 重检索吗？**→ 是：先算 token 天花板（实证 1：多 Agent×RAG 一个 session 能烧穿免费额度），先优化内层检索成本。
7. **派几个？**→ 努力分级：简单 1、对比 2–4、复杂研究 10+；由复杂度驱动、不是常数。

## 五、诚实的边界（这份判据**没**证明什么）

- **没证明"多 Agent 更强/更快"**：我的 demo 是顺序流水线，**真并行从未被压到**——这条主轴里"真并行"的收益，我没有自己的实测背书，只能引 Anthropic（Opus 4 lead + Sonnet 4 subagent 比单 agent 高 ~90.2%）。
- **没证明"独立上下文窗口解决长上下文压力"**：我的任务上下文没到 200K 量级，分摊压力这条也只引 Anthropic、没自测。
- **委派契约第 4 条 boundary 在本周弱化**：3 个异质角色顺序执行天然不撞车，boundary 不像 Anthropic"同质 subagent 并行"那样是刚需——但它在"同质 + 并行"时才是刚需，本周用不上不代表不重要（做并行 subagent 时会回来）。
- **隔离的副作用是真实代价**：findings 600 截断饿瘦 writer（12/12 触顶）、supervisor 拆解丢"本地库"语境锚致 researcher web 漂移——这些都是拓扑 A 没有、上了 B 才有的新坑。**判据要把"隔离会引入新副作用"算进成本侧。**
