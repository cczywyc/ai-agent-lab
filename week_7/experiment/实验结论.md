## E1：critic 三路条件边分发（accept / retry / escalate）

**验证**：单个条件边函数 `route_after_critic` 按 `critic_verdict` 把三种裁决正确路由到 `planner` / `executor` / `planner`；且"retry 达上限"塌缩到 escalate 通道（回 planner）。桩里 critic 按注入的 `_test_verdict` 直接写 verdict，不审质量。

**实测输出**：

```
=== E1: critic 三路分发 ===

  verdict=accept   retry_count=0 → planner  (期望 planner) ✓
  verdict=retry    retry_count=0 → executor (期望 executor) ✓
  verdict=retry    retry_count=2 → planner  (期望 planner) ✓
  verdict=escalate retry_count=0 → planner  (期望 planner) ✓

[通过判据]
  四种裁决全部路由到期望目标: ✓ PASS
```

| 判据 | 期望 | 实测 | 结果 |
|---|---|---|---|
| `accept` → planner | planner | planner | ✅ |
| `retry`（未达上限）→ executor | executor | executor | ✅ |
| `retry`（达上限 `retry_count=2`）→ planner | planner | planner | ✅ |
| `escalate` → planner | planner | planner | ✅ |

**结论**：草稿 §2.3 的 `route_after_critic` 设计成立，多路 dispatcher 一条边即可分发。

**精确化（落点 → §2.3 / 决策 A）**：三路 verdict 在拓扑上塌缩成**两条物理边**——`accept`、`escalate`、`retry-达上限` 三种结果里后两者同回 `planner`，所以 `add_conditional_edges` 的 mapping 只需 `{to_planner, to_executor}` 两个出口。§5 的 mermaid 已画出 C 节点三条箭头但两条落 planner，与此一致；实现时别把 mapping 写成三出口。"判定是三路、路由是两路"——决策落在 verdict 字段里，物理边只认落点。

## E2：两层重置并存（per-subtask 清零 × per-task 保留）

**验证（设计① 重置职责升级）**：v4.2 的"per-query 重置"在 v5.0 降一层——内层标志（`has_retrieved` 等）每个子任务清零，外层状态（`step_results`）跨子任务累加。对照组：去掉 step 转移重置，看 `has_retrieved` 从上一子任务泄漏到下一子任务。桩里 executor 进门先记下看到的 `has_retrieved`（验是否被上一步泄漏），再置 `True`。

**实测输出**：

```
=== E2: 两层重置并存 ===

[有 step 转移重置]
  每步进门 has_retrieved : [False, False]  (期望 [False, False])
  step_results 累加条数   : 2  (期望 2)

[对照 — 无 step 重置]
  每步进门 has_retrieved : [False, True]  (泄漏则为 [False, True])
```

| 判据 | 期望 | 实测 | 结果 |
|---|---|---|---|
| per-subtask 标志每步清零 | `[False, False]` | `[False, False]` | ✅ |
| per-task `step_results` 跨步保留 | 2 条 | 2 条 | ✅ |
| 对照：无 step 重置时标志泄漏 | `[False, True]` | `[False, True]` | ✅ |

**结论**：设计① 的"两层并存"坐实——`step_reset` 在 step 转移时只打回 `PER_SUBTASK_DEFAULTS`、不碰 `step_results`，于是"每子任务清零内层、跨子任务累加外层"两者兼得。这正是第六周 `init` "跨轮重置"思维的降一层版（per-query → per-subtask）。

**与第六周的对照（非意外，但值得记）**：本周对照组泄漏值是 `[False, True]`，而第六周 E2 对照组是 `[None, True]`。差别在于：本周 `init` 始终先建 `PER_SUBTASK_DEFAULTS`，第六周那条"TypedDict 无隐式默认值"的坑已被修复并带到 per-subtask 层，所以首步进门是 `False` 而非 `None`；泄漏纯粹来自"少了 step 转移重置"。→ 第六周的修复正确继承，本层不复发 `None` 问题。

## E3：step_results 节点内手动累加 vs reducer

**验证（决策 G）**：(a) `step_results` 在 executor 节点里"当前 + 新"手动累加，跨步攒结论；(b) `init` 返回 `[]` 能跨问题清空。对照组：给 `step_results` 上 `operator.add` reducer，`init` 的 `return []` 变成"追加空列表"的 no-op，清不掉——降一层重演第六周 `retrieved_chunks` 的实证。桩里预置一条脏数据 `step_id=99`，验 `init` 的 `[]` 能否清掉它。

**实测输出**：

```
=== E3: step_results 手动累加 vs reducer ===

[替换语义 + 节点内手动累加]
  累加结果 step_id : [0, 1, 2]  (期望 [0,1,2])
  init 清空脏数据   : 是

[对照 — 带 operator.add reducer]
  累加结果 step_id : [99, 0, 1, 2]
  脏数据 99 是否残留: 是（init 清不掉）
```

| 判据 | 期望 | 实测 | 结果 |
|---|---|---|---|
| 手动累加得 `[0,1,2]` 且 `init` 清空脏数据 | `[0,1,2]`、无 99 | `[0,1,2]`、无 99 | ✅ |
| 对照：`operator.add` 字段 `init` 清不掉脏数据 | 99 残留（`[99,0,1,2]`） | `[99,0,1,2]` | ✅ |

**结论**：决策 G 坐实，且完整重演第六周 chunks 实证——带累加 reducer 的字段，`return []` 是"追加空列表"的 no-op，**无法重置**。所以 `step_results` 选"替换语义 + executor 节点手动 `cur + new`"，`init` 才能用 `[]` 跨问题干净清空；"唯一 reducer 仍是 `messages` 的 `add_messages`"的简洁得以保住，仅降一层到 `step_results`。

## E4：外循环双闸门（step_index < MAX_STEPS 且 replan_count < MAX_REPLAN）

**验证（设计② 双层闸门）**：`planner → executor → critic → planner` 外循环靠两层闸门主动收口，在框架 `recursion_limit` 之前先终止（第六周 E4 教训在外循环重演）。对照组：拆掉闸门单靠 `recursion_limit=10` 兜底，看它抛 `GraphRecursionError`。桩里 critic 恒发 escalate，给外循环持续 re-plan 压力。

**实测输出**：

```
=== E4: 外循环双闸门 ===

[有双闸门]
  收口时 step_index=2  replan_count=2
  log: ['plan(replan=0)', 'exec(step=1)', 'critic→escalate', 'plan(replan=1)', 'exec(step=2)', 'critic→escalate', 'plan(replan=2)']

[对照 — 无闸门：单靠 recursion_limit]
  抛出 GraphRecursionError（如期）：Recursion limit of 10 reached without hitting a stop co...
```

| 判据 | 期望 | 实测 | 结果 |
|---|---|---|---|
| 双闸门在框架上限前先收口 | `replan_count` 或 `step_index` 触顶后停 | 停在 `step_index=2`、`replan_count=2` | ✅ |
| 对照：无闸门撞 `recursion_limit` | 抛 `GraphRecursionError` | 抛 `GraphRecursionError` | ✅ |

**结论**：外循环不死循环，`MAX_STEPS` / `MAX_REPLAN` 显式语义在图上原样保留；框架 `recursion_limit` 只当最后兜底（它的终止方式是抛异常，不是优雅收尾到 `finalize`）——第六周内层 `turn_count` 闸门的教训在外循环成立。

**精确化（落点 → §2.3 / 决策 D）**：双闸门**非冗余**，各兜一个正交维度——`step_index < MAX_STEPS` 兜"前进步数"（plan 走太长），`replan_count < MAX_REPLAN` 兜"原地绕圈"（反复 re-plan 不前进）。实测在 critic 恒 escalate 的压力下，是 **`replan_count` 闸门先收口**（停在 `step_index=2 < MAX_STEPS=4`、`replan_count=2 = MAX_REPLAN`），坐实 `MAX_REPLAN=2` 是 re-plan 环的有效断路器。若只设 `step_index` 一道闸门，escalate 死循环会一直原地 re-plan 直到撞 `recursion_limit` 抛异常——两道闸门缺一不可。

## E5：两档计数器互不串扰（empty_retries 传输层 × retry_count 业务层）

**验证（决策 B）**：`agent` 节点内的空回答重试（`empty_retries`，不进拓扑、不耗 turn）与 critic 驱动的业务重试（`retry_count`，走 retry 边回 executor）是两个独立计数器——一次 API 抖动不会吃掉一次业务重试额度。桩里 agent 在节点内 `while` 模拟"先空一次再成功"（`empty_retries+1`，不画进图）；critic 头一次发 retry、第二次发 accept。

**实测输出**：

```
=== E5: 两档计数器互不串扰 ===

  执行轨迹 log : ['agent(empty_retried=1)', 'critic→retry', 'agent(empty_retried=1)', 'critic→accept']
  empty_retries (传输层) : 2
  retry_count   (业务层) : 1
```

| 判据 | 期望 | 实测 | 结果 |
|---|---|---|---|
| 两计数器各自独立计数 | `empty=2, retry=1` | `empty=2, retry=1` | ✅ |
| 空回答重试未吃掉业务 retry 额度 | `retry_count=1 < MAX_RETRY=2` | `1 < 2` | ✅ |

**结论**：决策 B 坐实。agent 跑了两遍（首遍 + 业务 retry 回来一遍），每遍各自在节点内空重试一次 → `empty_retries=2`；业务 retry 恰好发生一次 → `retry_count=1`。传输层那 2 次抖动留在 `agent` 节点内，**没有挤占业务层额度**——若图省事合成一个计数器，"API 抖了一下就吃掉一次业务重试额度"的串扰就会出现，这正是职责边界 §4 末尾反复强调的那条。

**精确化（落点 → §三 / §一）**：实现时 `critic_verdict` 等 critic↔route 交互字段**必须**写进 State schema（桩里已踩：未声明则 critic 的更新被框架静默丢弃、条件边读不到）——延续第六周"schema 外字段被丢弃"的纪律，新增 7 字段一个都不能漏声明。

## E6（补充探针）：嵌套双循环 + recursion_limit 预算

**为什么补**：E1–E5 都是**单机制**隔离验。v5.0 的真实形态是 executor 引擎（`agent ↔ tools`，内层 `turn_count` 闸门）跑在外层 plan 循环（`step_index` / `replan` 闸门）里——**组合行为**没单独验过。E6 桩用 `agent/tools` 来回跑构成内层循环（每子任务跑 2 轮工具后 stop），嵌进 4 子任务的外层循环，重点验三件 E1–E5 照不到的事：(a) step 转移真能把内层 `turn_count` 归零（E2 用的是不跑内层循环的平凡桩）；(b) 内外两道闸门嵌套时各自照常收口、不互相误伤；(c) `recursion_limit` 在嵌套下还能不能只当"兜底"。

**实测输出**：

```
=== E6: 嵌套双循环 + recursion_limit 预算 ===

[跑通：4 子任务 × 每步 2 轮工具]
  每子任务首进 agent 的 turn_count : [1, 1, 1, 1]  (期望 [1, 1, 1, 1])
  总节点执行次数（≈super-step 数）  : 42

[关键探针：默认 recursion_limit 下跑同一正常任务]
  ✓ 默认 limit 下正常跑完（42 个 super-step 未触顶）

[通过判据 / 发现]
  (a) step 转移把内层 turn_count 归零      : ✓ PASS
  (b) 内外两层闸门嵌套下都正常收口          : ✓ PASS
  (c) ✓ 默认 recursion_limit 远大于本任务的 42 步……
      原假设'正常嵌套会撞限'被证伪：recursion_limit 确实只当兜底成立。
```

| 判据 | 期望 | 实测 | 结果 |
|---|---|---|---|
| (a) step 转移把内层 `turn_count` 归零 | 每子任务首进 agent 时 `turn_count==1` | `[1, 1, 1, 1]` | ✅ |
| (b) 内外两层闸门嵌套下都正常收口 | 4 子任务各跑满内层、外层正常推进 | 4 子任务全收口、共 42 super-step | ✅ |
| (c) 默认 `recursion_limit` 下正常任务不触顶 | 42 步 ≪ 默认上限、不抛异常 | 默认下跑完、未触顶 | ✅ |

**结论**：组合形态成立——内层 `turn_count`（per-subtask）由 `step_reset` 在每次 step 转移时归零，与外层 `step_index` / `replan_count` 闸门嵌套时**两道闸门各管各的、互不误伤**；E2 的"归零"在真正跑内层循环的桩上重新坐实（E2 桩不跑内层，E6 补上了这一刀）。

**精确化（落点 → 决策 D / §2.3）——一处假设被证伪 + 默认值精确实测**：

E6 脚本分支 (c) 原注释猜"LangGraph 1.2.4 默认值很高（单节点自环 250 步仍不触顶）"。本次复跑独立核了实值，结论更强也更准：

- **`DEFAULT_RECURSION_LIMIT = 10007`**（取自源码 `langgraph._internal._config`，并实测复现：单节点自环恰在 `Recursion limit of 10007 reached` 触顶）。这**既不是通说的 25，也不是脚本注释里的"250 仍不触顶"**——脚本作者只探到 250 就停了，低估了实值。
- 不传 / 传空 `config` / 传 `{"configurable": {}}` 都走这个默认 10007；只有**显式**传 `recursion_limit`（如 E4 对照的 `10`、或 `25`）才按所传值触顶。
- 故"正常嵌套任务会撞默认 limit"被**证伪**：v5.0 的有界任务（`MAX_STEPS × MAX_TURNS` 量级几十个 super-step）距 10007 差三个数量级，**默认 limit 永远轮不到收口**——真正收口的是 E4 坐实的显式 `MAX_STEPS` / `MAX_REPLAN` / `MAX_TURNS` 闸门。

→ **实现取舍**：v5.0 **无需显式调高 `recursion_limit`**（默认 10007 已远超任何有界任务）；它仍是真实存在、但几乎永不触发的最后兜底（且触发方式是抛 `GraphRecursionError`、不是优雅收尾到 `finalize`，同 E4）。仅当哪天把上界放得极大（如 `MAX_STEPS×MAX_TURNS` 逼近上万）时才需复核一次。

## E7（补充探针）：跨子任务 messages 重锚（executor 窗口隔离）

**为什么补**：`messages` 是全图唯一带 `add_messages` 的累加字段，且 `init` 故意**不重置**它（保留 thread 历史）。v5.0 里 executor 引擎每子任务都往 `messages` 追加（AI 回复 + tool 结果），**跨子任务只增不减**。那草稿 §四"executor 只看当前子任务"靠什么实现？答案是复用 v4.x 的窗口切片：agent 发给模型的窗口 = 最后一条 `content==SYSTEM_PROMPT` 的 system 消息起（`nodes._window_start`）。E7 桩原样复刻 `_window_start` + 真实 LangChain 消息对象（不调模型），验每子任务 `assemble` 重产 `SYSTEM_PROMPT` 锚后，子任务 2 的窗口是否只含子任务 2 的块；对照组 `assemble` 不重产锚（只追加 Human）。

**实测输出**：

```
=== E7: 跨子任务 messages 重锚 ===

[重锚：每子任务 assemble 重产 SYSTEM_PROMPT]
  全局 messages 累加条数 : 8（跨子任务只增不减，init 不清）
  每子任务 executor 窗口大小 : [2, 2]
  每子任务窗口是否串台早期 tool: [False, False]  (期望 [False, False])

[对照：assemble 不重产 SYSTEM_PROMPT（窗口锚死子任务 0）]
  每子任务 executor 窗口大小 : [1, 4]（应随累加变大）
  每子任务窗口是否串台早期 tool: [False, True]  (串台则 [.., True])

E7 全部通过 ✓
```

| 判据 | 期望 | 实测 | 结果 |
|---|---|---|---|
| 全局 `messages` 跨子任务累加（`init` 不清） | 2 子任务 × 4 条 = 8 | 8 | ✅ |
| 重锚后窗口隔离、不串台早期子任务 | `[False, False]` | `[False, False]` | ✅ |
| 重锚后窗口大小稳定（不随 `messages` 增长） | 两子任务窗口同尺寸 | `[2, 2]` | ✅ |
| 对照：不重锚则窗口锚死、发生串台 | 末子任务窗口混入更早 tool | `[1, 4]`、`bled=[False, True]` | ✅ |

**结论**：v5.0 的"executor 看局部"**不是新机制**——`assemble(role=executor)` 每子任务重产 `SYSTEM_PROMPT` 锚，复用 `_window_start` 切片即天然隔离：窗口大小稳定在 `[2, 2]`（每窗口 = 自己那段 system+human），不随全局 `messages`（8 条且只增）膨胀。对照组一旦不重产锚，窗口锚死在子任务 0、随累加涨到 `[1, 4]`，末子任务窗口里**串进了上一子任务的 tool 结果**（`bled=[False, True]`）。

**精确化（落点 → 草稿 §四）——一处文档缺口**：草稿 §四 现在讲的是 token 预算压缩（标注为"周四真实 API 实测项，桩节点不覆盖"），但**漏写了重锚这条隔离机制本身**。E7 证明：隔离机制（重产 `SYSTEM_PROMPT` 锚）用零 API 桩即可验、且是 §四"executor 看局部"成立的前提——§四必须显式写明"`assemble(role=executor)` 每子任务重产段 1 锚，executor 窗口靠 `_window_start` 自锚到本子任务"。漏了这条，实现时若让 `assemble` 只追加 Human 不重产锚，后一子任务就会看到前面全部 tool 历史（对照组实测）。（token 预算压缩仍是周四真实 API 的实测项，E7 不覆盖。）

## 周四（v5.0 实现）技术决策清单

实验结果对实现的直接约束：

1. **`route_after_critic` 的 conditional mapping 只写两个出口** `{to_planner, to_executor}`，别写成三出口——三路 verdict 物理上塌缩成两条边（E1）。
2. **外层两道闸门都要设、缺一不可**：`step_index < MAX_STEPS`（兜前进步数）+ `replan_count < MAX_REPLAN`（兜原地绕圈）；二者 OR 收口，escalate 压力下 `replan_count` 先触顶（E4）。`recursion_limit` 只兜底。
3. **`step_results` 严禁上 `operator.add`**：保持替换语义，累加在 executor 节点手动做 `state["step_results"] + new`，`init` 才能用 `[]` 跨问题清空（E3 降一层重演 chunks 实证）。
4. **`step_reset` 只打回 `PER_SUBTASK_DEFAULTS`、不碰 per-task 字段**（`plan` / `step_results` / `replan_count` / `done`）；这些只在 `init` 随新问题清零（E2）。
5. **新增 7 字段全部写进 State schema**，尤其 `critic_verdict`——schema 外字段的更新会被静默丢弃（E5）。条件边读字段一律 `state.get(k, 默认)`（沿用第六周纪律）。
6. **两档业务/传输计数器分开**：`empty_retries` 留 `agent` 节点内、不进拓扑；`retry_count` 走 critic→executor 回连边（E5）。
7. **不要为外循环显式调高 `recursion_limit`**：默认 10007 已远超任何有界任务（v5.0 量级几十个 super-step），收口永远由显式 `MAX_STEPS` / `MAX_REPLAN` / `MAX_TURNS` 闸门完成；`recursion_limit` 只是几乎不触发的最后兜底（E6）。`step_reset` 在 step 转移归零内层 `turn_count`，与外层闸门嵌套互不误伤（E6 在真跑内层循环的桩上重验 E2）。
8. **`assemble(role=executor)` 每子任务必须重产 `SYSTEM_PROMPT`（段 1）锚**：executor 窗口靠 `_window_start` 自锚到本子任务，复用 v4.x 切片即天然隔离；只追加 Human 不重产锚 → 后一子任务窗口锚死、串进前面全部 tool 历史（E7）。`messages` 跨子任务只增不减、`init` 不清——这是有意保留 thread 历史，隔离不靠清空而靠重锚。

**对 v0.2 草稿的修订建议**：本周无"推翻假设"的意外，五处可并入——

- **E1–E5（已并入 v0.2）**：E1 两出口 / E4 双闸门各兜一维度且 replan 先收口 / E5 schema 必须声明字段，已在 §2.3、§三、决策定稿表 A、D 行。
- **E6（新增）**：§2.3 / 决策 D 末尾补一句"默认 `recursion_limit=10007`（实测，非通说的 25），有界任务永远先被显式闸门收口，**无需显式调高**"；纠正 E6 脚本注释里"默认值很高（250 仍不触顶）"的旧猜测为实值 10007。
- **E7（新增、属文档缺口而非精确化）**：§四 **必须补写**"executor 窗口隔离靠 `assemble(role=executor)` 每子任务重产 `SYSTEM_PROMPT` 锚 + `_window_start` 自锚"这条机制——现 §四 只讲了 token 预算压缩（周四真实 API 项），漏了隔离机制本身，漏写则实现会串台。
- 并把 §六验证清单从"已实跑、13/13 过"更新为 **"已实跑、20/20 过（E1–E5 + E6–E7 补充探针）"**。
