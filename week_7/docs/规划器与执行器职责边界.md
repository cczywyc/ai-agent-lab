# 规划器 vs 执行器职责边界

> **定位**：关于第七周《规划器 vs 执行器职责边界》的一份概念说明，对着实战目标——技术调研 Agent（输入问题 → 出研究计划 → 分步检索总结 → 结构化报告）。</br>
> **说明**：planner-executor 的划线没有唯一正解，下面每个有分歧的点都**先表态再给理由**，并且我都标出了代价。</br>
> **基线**：search_agent v4.2（LangGraph 状态图，10 节点 + 条件边；checkpointer 管 thread 内短期，SqliteStore 管跨 thread 长期事实；收尾时序 `finalize → human_review → update_memory`）。v4.2 在 v4.1 之上落地了"第七周前重构五项"，其中四项直接动到本文档的设计前提，已逐条并入下文（见 §3 / §4 / §7 / §8）。

---

## 0. 这份文档要回答的核心问题

一条"先规划再执行"的链路里，**哪些决策归 planner、哪些归 executor、哪些归 critic，它们靠 state 的哪些字段交接**。

一句话边界：**planner 决定"做什么、做到什么程度算完"，executor 决定"这一步怎么做对"，critic 决定"这一步做没做好、接不接受"**。三者不越界——planner 不碰检索细节，executor 不决定下一步是什么，critic 不改计划（只触发 planner 去改）。

---

## 1. 三个角色定义

| 角色 | 一句话 | 不做什么 |
|---|---|---|
| **Planner** | 把开放问题拆成有序子任务、维护进度、判断何时终止；只产出和修订"做什么"的计划 | 不碰检索 / 工具 / 引用细节 |
| **Executor** | 拿单个子任务，执行检索 / 工具 / 总结，产出**带引用的单步结果**；只把当前步做完做对 | 不决定下一步是什么、不改计划 |
| **Critic** | 审单步结果质量，裁决 accept / retry / escalate（见 §4） | 不直接改计划、不重写结果，只发裁决信号 |

---

## 2. Planner 职责清单

| 候选职责 | 归 planner？ | 说明 |
|---|---|---|
| 任务拆解（产出有序子任务列表） | ✅ | 核心职责 |
| 拆解粒度 | ✅ | 一个子任务 = **一个可独立检索 + 总结的子问题**，不要细到"一次检索调用"——那是 executor 的内部事 |
| 子任务依赖（串行 / 并行） | ✅ | 调研任务多为串行或弱依赖；本周**先只做串行**，并行留到多 agent |
| 终止判断（"调研够了"） | ✅（持有信号） | planner 写 `done`，但**实际终止由条件边读这个字段**，调和"节点干活、边做决策" |
| 执行中 re-plan | ✅（受限） | 允许，但设上限 `max_replan`，否则会无限拆解绕圈 |

---

## 3. Executor 职责清单

| 候选职责 | 归 executor？ | 说明 |
|---|---|---|
| 单步执行（检索 / 工具 / 基于结果总结） | ✅ | 核心职责 |
| 上下文装配（保留你的 `assemble` 逻辑） | ✅（局部视角） | executor 只看：当前子任务 + 该任务召回 + 前序步结论**摘要**；不看其他步的检索原文。见 §5 第 2 条 |
| 单步结果格式约束（带引用） | ✅ | 沿用 `[doc#section]` 契约，但**先收紧**（踩坑 #3）；引用由 critic 校验 |
| 单步失败 / 空回答的即时重试 | ✅（先一层） | executor 自重试，仍不行才交给 critic 判——这是"空步废链"的第一道闸。**v4.2 已落地此层最薄的一档**：`agent` 节点对 `stop`+空 content 节点内重试一次（不画进图、不耗 `turn_count`，`empty_retries` 可观测），实证空回答在 qwen3.7-plus 上约两成、多为 API fast-fail。第七周把这档从"仅空回答"扩到"检索不准 / 引用错位"等失败模式时，再实测自重试次数（v4.2 取 N=1） |
| 把发现写入 store | ✅ | executor 把**单步检索发现**写 store；planner 写**子任务结论**（namespace 或标记区分，见 §6/§8） |

## 4. Critic 放在哪（核心决策）

**选 A：critic 是独立节点，审每一步的 executor 输出。**

为什么不选另外两种：

- **B（只审整体 plan / 最终报告）太晚**：调研 Agent 的失败几乎都在单步（空回答、检索不准、引用错位）。一步错却跑完全链才发现，多步链上是纯浪费。
- **C（critic 是 planner 的反思动作）耦合两种判断**：把"结果质量"和"计划好坏"塞进同一个角色，违反角色分离，而且 planner 本不该看检索细节。

**critic 的三种裁决（把"重试归谁"钉死）**：

```
executor 产出单步结果
        │
     critic 审
        ├── accept    → 进下一步（或 planner 判 done）
        ├── retry     → 回 executor 重做该步，retry_count += 1
        └── escalate  → 重试用尽仍不行，回 planner：
                         换子任务表述 / 标记该步失败跳过 / re-plan
```

即：**executor 自重试（无状态、最快）→ critic 判 → escalate 才动 planner**。三级而非一级，因为空回答约两成（v4.2 复跑实测），全压 planner re-plan 太贵。最底下那档"executor 自重试"在 v4.2 已有实现与实测基线（`agent` 节点空回答重试一次、`empty_retries` 量化"重试救回率"）——它兜的是传输层抖动（API fast-fail）；critic 主导的 `retry`（业务层：换措辞重做该步）是它之上独立的一档，第七周新增，二者不要混成一个计数器。

## 5. 边界灰色地带（逐个表态）

**① 单步失败 / 空回答的重试归谁？** —— 三级分摊（见 §4）。executor 先自重试（v4.2 已落地：空回答节点内重试一次），critic 判 accept/retry/escalate，只有 escalate 才到 planner。**这条直接回应你"空一步废全链"的担忧**：链路不会因为一步偶发空回答就崩，但也不会无脑无限重试。注意区分两档重试——v4.2 的节点内空回答重试是**传输层**兜底（API fast-fail，不进拓扑、不耗 turn）；critic 的 `retry` 是**业务层**重做（换子任务表述再检索一遍），第七周才加，二者不要混成一个计数器。

**② 上下文：planner 看全局、executor 看局部吗？** —— 是，**分层装配**。

- planner 看：原问题 + 当前 plan + 各步结论**摘要**（不看检索原文）。
- executor 看：当前子任务 + 该任务的召回 + 前序步结论摘要（不看其他子任务的检索细节）。

这正是你第四五周那条"六段装配 = 多 agent 上下文路由"从**时间维度扩到空间维度**——以前是给一个 agent 每轮挑段，现在是按角色挑段。`assemble` 逻辑复用，但要**参数化**（传入 role，按角色决定喂哪几段、各段预算）。

**③ plan 定死还是可 re-plan？** —— 允许 re-plan，但 `max_replan=2`。静态 plan 在调研里太脆（一个子问题查空，整条计划就废）；无限 re-plan 会绕圈。受限 re-plan 是两害相权。

**④ "调研够了"由谁、看什么信号判定？** —— planner 判，看两个信号取其一：所有子任务都 `accept` 或被标记完成，**或** 达到 `max_steps`。终止由条件边读 planner 写的 `done` 字段触发。

## 6. State 字段设计（交接契约）

**沿用 v4.2（概念照搬）**：`messages`（唯一带 `add_messages` reducer 的字段）、记忆 / 装配相关字段、trace。

**本周新增**：

| 字段 | 类型 | 谁写 | 谁读 |
|---|---|---|---|
| `plan` | `List[SubTask]` | planner | planner / executor / edge |
| `step_index` | `int` | planner（及推进边） | 全员 |
| `step_results` | `List[StepResult]`（text + citations + status） | executor | planner / critic / finalize |
| `critic_verdict` | `Literal["accept","retry","escalate"]` | critic | edge |
| `retry_count` | `int` | executor / critic | critic / edge |
| `replan_count` | `int` | planner | edge（防绕圈） |
| `done` / `termination_reason` | `bool` / `str` | planner | edge |

**reducer 注意**：顺序图无并行分支写同一字段，所以除 `messages` 外**全部替换语义**（沿用你"顺序图别发明 reducer"那条）。`step_results` 要累加就在 executor 节点内手动 append，不靠 reducer——v4.2 已用 `empty_retries` / `retrieved_chunks` 跑通这条"替换语义字段 + 节点内手动累加 + init 清零"的先例，`step_results` / `retry_count` 照搬即可。

## 7. 映射回 v4.2 现有图（复用 vs 新增）

**直接复用**：`assemble`（改为按 role 参数化挑段）、`tools`（v4.2 已瘦身为 `TOOL_EFFECTS` 声明化注册表，第七周加检索 / 调研工具只登记注册表、节点零改动）、`update_memory`、`human_review`、`finalize`。**注意 v4.2 已反转收尾时序**为 `finalize → human_review → update_memory`（审批框永不空、人工可当场补救占位符，记忆记录审批后版本）——planner / critic 接入时，这条收尾三连是固定的链尾，不要再按 v4.1 的"human_review 在 finalize 前"来排。

**需新增**：`planner`（入口附近）、`critic`（executor 之后）。

**图骨架**：

```
START → planner → assemble → executor(tools) → critic
                      ▲                            │
                      │  retry                     │ accept
                      └────────────────────────────┤
                              escalate              │
                      ┌───────► planner             ▼
                      │                     [planner: done?]
                      └──────── re-plan ◄──── no ───┤
                                                    │ yes
                                       finalize → human_review → update_memory → END
```

**条件边**（沿用"节点干活、边做决策"）：
- critic 后：`accept` → 进 planner 的 done 判断；`retry` → 回 executor；`escalate` → 回 planner。
- planner 后：`done` → finalize；否则 → 下一步 assemble。

## 8. 待验证 / 风险

- **踩坑 #3（`[doc#section]` 格式不稳）**：多步积累会放大。**v4.2 已在抽取侧收紧**——`extract_fact_candidates` 接受 `allowed_sources`（本轮真实检索到的 `(doc, section)`），引用对不上白名单按无引用处理、零检索则引用全拒收（`_trace_shim` 把 `retrieved_chunks` 带进抽取，恒为 list）。第七周 critic 要做的是把同一契约**前移到单步输出侧**：executor 产出即强制格式 + critic 校验引用合法性，不合法直接判 retry，别等攒进 store 才发现。
- **空回答重试参数**：三级机制已定。最底档"executor 自重试"v4.2 已落地并实测（空回答重试一次、`empty_retries` + summary 的 `empty_retry_count` 量化重试救回率）——但那只覆盖空回答这一种失败。`N`（扩到检索不准 / 引用错位等失败模式后的自重试次数）和 `max_replan` 的具体值仍要实测，不能拍脑袋。
- **token 预算**：plan 越长，planner 看的"各步结论摘要"越胀，会顶到上下文上限——给摘要设预算，旧步结论超预算就压缩。
- **critic 自身假阳性**：这是你 `--test` 里 `has_answer` 占位符假阳性教训（Case 5）的复现版——critic 判 `accept` 的标准若编码不全，空回答会混过去。v4.2 已修这版教训：占位符名单收口 `config.PLACEHOLDER_PREFIXES` / `is_placeholder_answer`（写入方与判读方共用一份），`judge_case` 抽成纯函数并有 `test_criteria` 套件兜底——critic 的 accept 判据可直接复用 `is_placeholder_answer` 作"非空答案"的最低闸，并循 `judge_case`"纯函数 + 单测"的先例编码判据。但风险未消：**critic 的判据要比 agent 输出更严**，且默认怀疑"全部 accept"。
- **re-plan 与 store 的交互**：re-plan 后，被废弃子任务已写进 store 的结论要不要撤回？不撤会污染后续跨任务召回。本周至少给写入打 `plan_version` 标记，方便事后甄别。
