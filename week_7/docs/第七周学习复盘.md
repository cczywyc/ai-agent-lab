## 二、本周做的事，一句话

给 v4.2 那个"单问题、单答案"的内层 tool-use 引擎，**套一层"多子任务"的外循环**：planner 拆解研究计划、executor 逐步检索总结、critic 审单步质量、条件边管推进/重试/跳过/终止；最后 finalize 组装结构化调研报告。**核心认识：本周是给 v4.2 套一层外循环，不是重写**——v4.2 的单步引擎（agent↔tools↔inject_\*）、收尾三连、Store/RAG 全部原样复用，新增的只有 planner / critic / step_init 三个节点和两个 dispatcher。

而这周和第六周最大的不同：**第六周是"发现型"（E2 撞出 TypedDict 无隐式默认这个真意外），第七周是"巩固/放大型"——主线是"降一层复用"。** 几乎每个机制都是把第六周已验证的东西降一层再验一遍，实验坐实的是"降一层后照旧成立"，而不是推翻什么。

| 第六周机制（单层） | 第七周降一层 |
|---|---|
| `init` per-query 重置 | `step_init` per-subtask 重置（+ `retry_reset` 第三层轻量重置） |
| 单层 `turn_count` 循环 | 内层 turn 循环嵌进外层 plan 循环 + **双闸门**（`step_index`/`replan_count`） |
| 单答案 | 多子任务规划 → 结构化报告 |
| `retrieved_chunks` 手动累加、不上 reducer | `step_results` 同款（决策 G 照搬） |
| 空回答传输层重试（`empty_retries`） | 三级重试：+ 业务层 `retry_count` + 计划层 `replan_count` |
| 六段装配（时间维度：每轮挑段） | 按 role 装配（空间维度：按角色挑段，§四） |
| `_window_start` 窗口切片 | 跨子任务重锚（每子任务重产 SYSTEM_PROMPT 锚，E7） |

真正"新"的只有三块：critic（reflection 自检）、planner 的 LLM 拆解、以及三处**真实跑才暴露**的策略修复（下面 §六）。

## 五、实验与量化

### 桩节点验"框架接线"（E1–E7，20/20）

零 API、确定、可复现，只验框架机制不验业务质量。E1 critic 三路分发 / E2 两层重置并存 / E3 step_results 手动累加 / E4 外循环双闸门 / E5 两档计数器不串扰 / **E6 嵌套双循环 + recursion_limit 预算** / **E7 跨子任务 messages 重锚**。每条配对照组（验"对的能跑"也验"错的怎么错"）。

- **E6 证伪一条记忆**：原以为"正常嵌套任务会撞默认 `recursion_limit`"，且脑子里默认值是通说的 25。实测 `DEFAULT_RECURSION_LIMIT = **10007**`（取自源码、单节点自环恰在此触顶）——v5.0 有界任务（几十个 super-step、E6 实测 42）距它三个数量级，**默认 limit 永远轮不到收口**，真正收口的是显式 `MAX_STEPS`/`MAX_REPLAN`/`MAX_TURNS` 闸门。**无需为外循环调高 recursion_limit**。
- **E7 在实验阶段就暴露设计缺口**：`messages` 跨子任务只增不减、`init` 不重置它，那"executor 只看当前子任务"靠什么？答案是每子任务 `assemble` 重产 SYSTEM_PROMPT 锚、复用 `_window_start` 切片。对照组不重锚则窗口锚死（`[2,2]→[1,4]`、第二子任务串台 `[False,True]`）。这条草稿 §四原本没写明——**实验把"该写而没写"的隔离机制逼出来了**。

### 真实 API 端到端（最终 5/5）

同一个 5 子任务问题（"从第二周到第六周 Agent 怎么演进"，覆盖 5 条线），最终一跑：

| step | turn_count | 合成提示 | verdict | 引用条数 |
|---|---|---|---|---|
| 0 | 5 | **synth=True** | accept | 18 |
| 1 | 4 | — | accept | 33 |
| 2 | 3 | — | accept | 40 |
| 3 | 3 | — | **retry→accept** | 29 |
| 4 | 5 | **synth=True** | accept | 22 |

**5/5 步 accept**、引用 18/33/40/29/22 条均过接地软闸门、`replan=0`、`all_steps_done`、报告 ~28.7K 字符、0 报错。critic LLM 真发生过一次业务 retry→accept（step 3），宽主题步（step 0/4，含曾栽掉的 LangGraph 迁移）被 synthesis-reserve 逼综合、turn 上限前产出。

**§四 token 预算（原"待验证"项就此结案）**：摘要段峰值仅 **~1307 字符**（受 `MAX_STEPS`×每步 400 上限约束、有界、**无需压缩**）——原本担心它越拼越胖顶窗口，实测证伪。真正的窗口压力是**子任务内检索 chunk 累加**（峰值 ~12–19K tokens），已由 synthesis-reserve 控住。

## 六、踩坑与本周发现

### 最该记的一条：离线验接线，真实验策略

离线 64/64 让我一度以为 v5.0 稳了。但真实跑连摔三跤，**三处缺陷全在桩测结构上够不着的那一层——模型行为与策略，不是接线**：

1. **引用闸门过严（"离线对、真实错"的典型）。** 旧 critic 要求"全部引用精确命中本步召回"。桩测里 executor 答案只有 0–1 条引用、必然命中，全绿。真实模型写一节报告会引 **20–40 条**（含子节、相关节、缩写 section），多于本步召回集，**一条对不上就整步判 retry**。后果是级联：retry 耗尽→escalate→re-plan→烧光预算→`critic LLM 根本没被调到（裁决序列 []）`、整条研究 0/N 完成。
   - 修：① section 匹配放宽（容忍缩写/后缀/叶子，真实库 section 是很长的层级路径）；② 更关键的——从"全命中"改为**接地比例软闸门**（能溯源到本步召回的引用占比 ≥ `CITATION_MIN_GROUNDING=0.5` 就放行进 LLM 裁决）。store 持久化侧仍由 `extract_fact_candidates(allowed_sources)` 严格白名单兜底，双保险。补 W12。
2. **escalate 烧预算。** 旧 re-plan 保留 step_index、**重做同一步**——同样的失败原因（比如引用问题）反复触发，烧光 `MAX_REPLAN` 让后续步根本没机会跑。一步失败拖垮整条研究。
   - 修：**skip-and-advance**——escalate 就标记该步 skipped、推进 step_index，`replan_count` 计跳过数、达上限才放弃剩余。补 W13 之外另调 W2/W5/W9 语义。
3. **宽主题步过度检索、撑爆 turn 预算。** executor 在内层把 `MAX_TURNS` 全耗在反复 `retrieve_documents` 上、来不及综合 → turn 跑满空产出 → escalate。真实跑栽在"LangGraph 迁移"这种最宽的步上。
   - 修：**synthesis-reserve**——`after_tools` 在临近 turn 上限（`turn_count ≥ MAX_TURNS - 2`）且还在要工具时注入 `inject_synthesis`（"停止检索、立即综合"，每子任务最多一次），逼它在 turn 上限前产出。补 W13。

**三处都补了离线 W 测试（W11/W12/W13），TDD 没断。** 这条是本周核心教训：**离线 64/64 是必要非充分**——它锁住了"接线对"，锁不住"策略在真实模型行为下成不成立"。真实跑才是验策略的唯一手段，而且每修一处都要把它沉淀成一个能复跑的离线判据，否则下次重构又会回弹。

### 真实跑的迭代轨迹（每跑一次摔一跤、修一处）

| 跑 | 配置 | 结果 | 暴露 |
|---|---|---|---|
| #1 | 原始 v5.0（全命中闸门 + re-plan） | 1/5 | 引用全命中误杀（step 0 全路径引用命中→accept，step 1 缩写→误杀→级联 max_replan） |
| #2 | section 放宽、仍"全命中" | 0/3 | 放宽 section 不够——"全命中"对 27–37 条引用必有 1 条不中，critic LLM 仍 [] 没被调到 |
| #3 | 接地软闸门 | 3/5 | 引用修复坐实（critic LLM 正常 3×accept）；但 step 3/4 撞 **DashScope 免费额度 403** |
| #4 | 换模型 `qwen3.7-plus-2026-05-26` | 3/4 | step 3（LangGraph 宽步）过度检索 turn 耗光→escalate |
| #5 | synthesis-reserve | **5/5** | —— 收口 |

### 其它发现

- **免费额度耗尽是真实工程摩擦，且 403 歪打正着。** 跑 #3 的 step 3/4 报 `403 AllocationQuota.FreeTierOnly`（聊天模型 `qwen3.7-plus` 免费额度被这周反复跑堆爆，`chatcmpl` 调用——不是 embedding）。换有额度的快照 `-2026-05-26` 解决。但那几个 403 **顺手验了 skip-and-advance 的优雅收口**：agent 把 403 当 LLM 失败→占位→critic→跳过→出 3 段部分报告而非崩溃——本来要专门构造的"步失败"场景，被 API 额度白送了一个。
- **planner 拆解非确定**：5 次跑分别拆出 5/3/5/4/5 个子任务，同一问题每次粒度略不同。"5/5"里的"5"不是常数——评估"全步完成"比纠结"恰好 5 步"更有意义。
- **缺口在设计评审阶段（实现前）就抓住了。** 草稿 v0.2→v0.3 是"拿 E1–E7 的实验数据回头审设计草稿"抓出的 4 处缺口（跨子任务重锚机制、retry_count per-subtask、plan_version 承接 §8、planner 判 done 依据）——全在写实现代码之前补上。**这一步省了多少返工不好量化，但方向是对的**：实验不只是验假设，还是"回头给设计挑错"的数据源。

## 七、对第八周的思考（多 Agent）

第八周是多 Agent（handoff / 角色拆分 / centralized orchestration vs agent-to-agent）。第七周已经把地基铺好了：

- **planner / executor / critic 就是"一张图里的角色拆分原型"**——researcher（executor）/ reviewer（critic）/ 协调者（planner）的雏形已经在跑。
- **§四"分层装配 = 按角色路由上下文"是多 agent 上下文路由 / handoff 的种子**：不同角色看不同的段（planner 看摘要、executor 看局部召回），这正是 handoff 时"交接什么、不交接什么"的同一个问题，从时间维度（每轮挑段）扩到了空间维度（按角色挑段）。
- **critic ≈ reviewer**，三级重试里的 escalate 通道 ≈ "打回上游"。

**带一个问题进第八周**：真·多 agent（独立 agent + handoff，各自独立的 messages / 上下文）比第七周"一张图里的角色节点"到底**多买到什么**？这周用"一张图里的角色节点"已经把 decomposition / reflection / 终止回滚都跑通了，独立 agent 的额外成本（协调、上下文隔离、handoff 协议）换来的是什么实打实的能力——正好呼应 roadmap 那句"**不是越多 agent 越好**"。先带着这个怀疑去学，别一上来就堆 agent。

## 八、跨模型 / 通用设计原则（本周新增，每条挂证据）

- **离线验接线、真实验策略**（三处真实跑缺陷）：桩测锁"框架行为对"，锁不住"策略在真实模型行为下成不成立"。引用闸门、escalate、turn 预算三处都是离线全绿、真实才崩。离线全绿是必要非充分。
- **"全有/全无"的硬闸门在真实输出上太脆，要软**（引用接地软闸门）：模型富报告会引远多于召回集的来源，"全部命中"必挂；改"多数接地"放行 + 下游严格白名单兜底，才既不误杀又不漏编造。
- **失败要"跳过推进"而非"原地重做"**（skip-and-advance）：同一失败原因的原地重试会烧光预算、拖垮整条链；让失败步可跳过、后续步照常跑，单点失败不致全局失败。
- **给"综合"留预算，别让"检索"吃光轮次**（synthesis-reserve）：多步 agent 的每一步都既要取数又要出活；临近上限要主动逼停取数、转出活，否则跑满轮次却空手。
- **拿实验数据回头审设计，能在写代码前抓缺口**（缺口 1–4 / E7）：实验不止验假设，还是给设计草稿挑错的数据源；"实现前再审一遍设计"这步省返工。
- **证伪比验证更值，且要敢推翻脑内通说**（recursion_limit=10007）：默认值不是记忆里的 25；"正常嵌套会撞限"是错的。靠实测而非记忆下判断。
- **巩固型的一周也要明确点出它是巩固型**（降一层复用）：不是每周都有真意外。第七周几乎全是"降一层后照旧成立"，价值在放大与坐实、不在发现——把这点写明，免得误以为"没意外=没收获"。
- **复盘/报告数字要在定稿当天能跑出来**（沿用上周）：本周量化全部来自真实跑当天输出（20/20、64/64、5/5、引用 18/33/40/29/22、摘要段 1307 字符）。

## 九、本周最大收获

**把"单问题单答案"扩成"多步规划"，工程上不是加一个循环，而是把"控制流"再分一层——外层 plan 循环管"做什么、做到什么程度算完"，内层 tool-use 循环管"这一步怎么做对"，两层各有独立闸门、互不串扰。** 第七周能这么顺地搭起来，是因为第六周已经把"节点干活、边做决策""顺序图别发明 reducer""桩节点隔离验框架"这套吃透了——本周做的就是把它们**整体降一层**再跑一遍，实验坐实"降一层后照旧成立"。这是一周巩固/放大型的工作，不是发现型，**而把这点认出来本身就是收获**：不是每周都该期待"真意外"，把已验证的机制稳稳地放大一层，同样是进步。

但比"降一层跑通"更深的一课，来自真实跑那三跤：**离线 64/64 全绿时我以为 v5.0 稳了，真实模型一上来就用"引 40 条来源、宽主题狂检索、section 随手缩写"这些桩测里根本没有的行为，把三处策略全顶崩了。** 桩测验的是"我把图接对了"，真实跑验的是"我对模型行为的假设对不对"——后者只能在真实模型上验，且每验出一处，就得把它沉淀成一个能复跑的离线判据（W11/W12/W13），否则下次重构照样回弹。**"测试过了"要拆成两句：接线我用桩测过了，策略我用真实模型过了。** 少了后半句，全绿就是错觉。
